# OpenCode Swarm v8 Feature Roadmap

- Status: proposal (evidence-gated enhancement audit; no implementation in this document)
- Date: 2026-07-02
- Baseline: v7.103.0 @ b654aa6b
- Provenance: produced by a full codebase-review-swarm enhancement audit (Track G, focused, non-diluted). 87 evidence-quoted candidates were generated by 12 parallel explorer lanes, validated by 12 independent reviewer contexts that re-opened every citation, and challenged by 5 independent critic contexts (4 batch + 1 strategic) plus a final whole-report critic (PASS after one revision round). Every claim below survived that pipeline; run artifacts live under `.swarm/review-v8/runs/20260702T035413Z/` (local, git-ignored).

## TL;DR

v7 has already built the machinery for its own next act. The v8 flagship is **parallel-first execution** — retiring the published "one task executes at a time" contract in favor of safe-concurrent-by-default for provably disjoint work. Two co-pillars ship beside it: the **default auto-review engine** (implementing this repo's own `docs/proposals/auto-review-by-review-model.md`) and a **governed defaults-flip** that promotes matured opt-ins with a migration guide. Three more majors (background delegation GA, autonomous PR babysitting, graph-memory completion) land across v8.x. A v7.x runway of quick wins clears debt that would otherwise ship stale into v8.

## 1. The Flagship: Parallel-First Execution (ENH-G012-1) — the feature that justifies v7 → v8

**What changes.** The core execution contract. Today: `docs/architecture.md` — "One task executes at a time"; `src/config/schema.ts:1802-1806` — parallelization is a "dark foundation, disabled by default" with `maxConcurrentTasks` defaulting to 1. In v8, the architect executes provably disjoint task groups concurrently in isolated git worktrees by default, with serial execution as the automatic fallback whenever scopes overlap or are unknown.

**Why it is v8-major material.** Flipping a published plugin's default execution model is exactly the change the maintainers deferred as "risky for published plugin" in `docs/dev/worktree-parallelization-first-class-plan.md:199-203` — a semver major is the only correct vehicle. It is also the most legible story this product can tell: *the swarm that actually swarms*.

**Why it is buildable now.** The worktree-parallelization plan is "IMPLEMENTED & APPROVED" (162 tests); the reviewer-gate parallel exemption (C4) and `[PARALLEL EXECUTION PROFILE]` delegation directives already ship; lean turbo already runs up to 4 parallel coders by default (`schema.ts:1840`), providing real burn-in; conflict primitives exist as pure, tested functions (`epicPairConflict` path + co-change NPMI verdicts in `src/turbo/epic/cochange-conflict.ts:94-99`, `pathsConflict`/`isGlobalFile` in `src/turbo/lean/conflicts.ts`, repo-graph `blast_radius`).

**What to build.**
1. `plan_conflict_check` — the roadmapped, still-missing automated file-conflict-detection tool (ENH-G008-1, upheld HIGH): read-only, takes N proposed task groups, resolves declared scopes, returns a pairwise conflict-verdict matrix plus a suggested serialization order. Also disposes of the orphaned `src/parallel/dependency-graph.ts` / `meta-indexer.ts` modules (zero production importers) — either as this tool's backbone or deleted.
2. Harness-derived parallel/serial decision: parallel only when the conflict tool proves disjointness; serial otherwise. No user quiz.
3. `maxConcurrentTasks` default > 1 with worktree `policy: 'auto'` as the isolation guarantee.
4. Merge-conflict recovery UX (ENH-G008-3, upheld HIGH): persist worktree path + branch name + replay commands durably, surface preserved worktrees in `/swarm status`, and stop `cleanupOrphanedBranches` from force-deleting preserved lane branches.

**Risks.** Merge-recovery UX must land first (loss-of-work path exists today); token cost multiplies with concurrent coders; git-hosting quirks (submodules, LFS) multiply under worktrees. Composition rule from the strategic critic: parallel-first and background-delegation GA both change delegation semantics — **do not flip both defaults in the same release**.

## 2. Co-pillar: Default Auto-Review Engine + `/swarm review` (ENH-G012-2)

Implement the repo's own adversarially-validated proposal (`docs/proposals/auto-review-by-review-model.md`, gaps G1–G6): structured JSON reviewer findings with per-finding confidence calibration, an independent `critic_finding_validator` pass over HIGH/CRITICAL findings (generate-then-verify — closing the "same context invents and approves" gap the repo's own swarm-mode contract names), and an automatic merge-base whole-diff final review on the default path by generalizing the lean-turbo engine (`dispatchPhaseReviewer`, `src/turbo/lean/reviewer.ts:532`). Add the `/swarm review` command over the same engine. In v8.0, flip `auto_review.enabled` (today `schema.ts:469-471`, default false, advisory-only slice) to default-true advisory with confidence filtering; gate mode stays opt-in. Effort: M. This is the readiness leader — ~80% of the final-review pattern already ships for lean-turbo.

## 3. Co-pillar: The Governed Defaults-Flip (ENH-G012-5)

21 of 63 `enabled` flags default false. v8 is the only correct moment to promote the matured ones. Run a promotion board per opt-in; wave 1: auto-review advisory and architectural-supervision advisory, with memory recall injection and conservative context_map contingent on burn-in evidence per the strategic critic. Ship with: a migration guide, `/swarm config doctor` autofix mappings for deprecated/renamed keys, a one-line kill switch per flip, a conservative `preset` for users who want v7 behavior, an explicit amendment to `docs/design-rationale.md`'s "all default false" principle, and a published cost-delta statement (default-on injection changes token spend for every user). Effort: S per flip, M overall.

## 4. v8.x Majors

- **Background delegation GA (ENH-G012-3 / ENH-G001-1).** The fail-closed background-Task block (`src/hooks/delegation-gate.ts:1021-1027`) was always staged, not permanent: the durable pending-delegation ledger, trusted completion observer, and workspace-freshness Stage B gates already ship behind `hooks.background_subagents`. Graduate coder completions into task workflow state, deliver mid-turn architect advisories via the existing `pendingAdvisoryMessages` path, and flip the default once the upstream `OPENCODE_EXPERIMENTAL_BACKGROUND_SUBAGENTS` flag stabilizes. Enables pipelined execution (review of task N overlapping coding of task N+1) — multiplicative with parallel-first. Fix now in v7.x: the stale `schema.ts:227-235` docblock that still claims "NO workflow gate is advanced" (contradicted by `stage-b-gates.ts`).
- **Autonomous PR babysitting (ENH-G012-6).** Compose the three existing legs — pr_monitor events, the PR_FEEDBACK protocol, full-auto critic oversight — into an opt-in loop: after the swarm opens a PR, auto-subscribe; on CI failure/review comments/merge conflicts, run a bounded, oversight-gated fix-push-recheck session until green or human-paused. Needs denial/attempt budgets against CI flakiness.
- **Graph-memory completion (ENH-G012-4, KG-13 → KG-18).** Execute the remaining `docs/repo-memory-graph-plan.md` arc: gate-controlled injection of token-budgeted, provenance-bearing `context_pack` symbol slices into coder/reviewer context (KG-13 — today `getContextPack` is reachable only via the `repo_map` tool), stale-graph rebuild UX (KG-14), perf budgets (KG-15), cross-platform CI (KG-17), then default-capable promotion (KG-18). Feeds the flagship (blast-radius → conflict tool) and auto-review (sharper reviewer context).

## 7. Explicitly Not Recommended for v8

- **MCP interop (G012-7)** — rejected by the strategic critic: the product's differentiator is its fail-closed enforcement model, which cannot travel across an MCP boundary; zero in-repo demand signal.
- **Deferred pending evidence:** dense-embedding knowledge retrieval (G005-1) and embedding-based consolidation clustering (G005-7) — both gated on G005-2 harness evidence; test-impact-from-graph (G006-6); per-file pre-check caching (G007-6); evidence attestation certificates (G007-7); auto-checkpointing (G008-5); plan analytics (G009-1); manager.ts decomposition (G009-4, invariant-5 risk).

## 8. Release Sequencing (strategic critic's plan)

1. **v7.x runway** — quick wins above; conflict-detection tool + merge-recovery UX shipped opt-in; auto-review Phases 1–3 opt-in + `/swarm review`.
2. **v8.0** — parallel-first default for proven-disjoint plans (serial fallback); `auto_review` default-true advisory; defaults-flip wave 1 with migration guide + conservative preset; background delegation stays opt-in.
3. **v8.x** — background delegation GA (upstream-gated); PR babysitting opt-in GA; graph-memory KG-13→KG-18; Epic Mode GA as its own decision.
