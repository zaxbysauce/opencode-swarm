# Release Notes — v6.58.1

## Overview

Hardening patch for `/swarm close` session teardown and the critic-approved plan snapshot recovery path wired into `loadPlan`. This release fixes three correctness bugs in the Step 4b fallback that could either silently resurrect the wrong plan, loop a fresh process back into recovery, or leave a stale plan alive across sessions, plus a dead-code validation bug that blocked plan-free retro writes.

## Fixes

### Cross-identity guard on critic-approved snapshot recovery

Previously, `loadLastApprovedPlan` returned the last critic-approved snapshot it found regardless of which workspace authored it. A reused directory whose ledger retained a stale `critic_approved` snapshot from a prior swarm could resurrect the wrong plan during recovery.

**Fix:** `loadLastApprovedPlan` now accepts an optional `expectedPlanId` and skips any snapshot whose event-level `plan_id` does not match, with a belt-and-suspenders check against the snapshot payload's own `plan_id`. `loadPlan`'s Step 4b derives the expected identity from the ledger's first event (the `plan_created` anchor written by `initLedger`), and the hash-mismatch rebuild path (manager.ts:315) reuses the workspace's `currentPlanId` from the already-validated `plan.json`.

### Empty-events guard

`ledgerExists()` returns true whenever `plan-ledger.jsonl` is present on disk, but `readLedgerEvents()` silently returns `[]` for an unreadable or empty ledger. Without a guard, Step 4b would pass `expectedPlanId=undefined` and bypass the cross-identity filter entirely.

**Fix:** Step 4b now explicitly refuses recovery and returns `null` when `readLedgerEvents()` comes back empty, logging a warning that plan identity cannot be verified.

### Cross-process recovery loop (healing snapshot)

Step 4b successfully recovered the critic-approved plan in the first process, but the ledger tail still ended with the event that made replay fail in the first place (e.g. `plan_reset`). A second `loadPlan` from a new process — where the startup-check cache is empty — would re-enter the same recovery path indefinitely.

**Fix:** After a successful Step 4b recovery (both in the hash-mismatch rebuild and primary paths), `loadPlan` now appends a fresh `takeSnapshotEvent` with `source='recovery_from_approved_snapshot'`. `replayFromLedger`'s walk-backward picks this up before reaching the failing tail event on the next process start.

### Phase rollback visibility

Step 4b recovery could silently roll the plan back to an earlier phase without surfacing which phase the approved snapshot came from. The warning log now includes `approval.phase` so operators can notice if recovery moved the cursor backwards.

### Ledger cleanup at `/swarm close`

`plan-ledger.jsonl` and its sibling `plan-ledger.archived-*.jsonl` / `plan-ledger.backup-*.jsonl` files were left in `.swarm/` after close. The next `/swarm` invocation would then walk the stale ledger and try to resurrect the closed plan via Step 4b.

**Fix:** `close.ts` now includes `plan-ledger.jsonl` in both `ARCHIVE_ARTIFACTS` (copied into the close bundle) and `ACTIVE_STATE_TO_CLEAN` (deleted from `.swarm/`), and sweeps any leftover `plan-ledger.archived-*.jsonl` / `plan-ledger.backup-*.jsonl` siblings.

### `retro-session` task ID accepted by `write-retro`

`VALID_TASK_ID` in `write-retro.ts` only matched phase-numbered retro IDs (`\d+\.\d+`) and rejected `retro-session`, which `/swarm close`'s plan-free path writes when there is no active plan. The close command emitted `retro-session` but the tool silently rejected it.

**Fix:** Broadened the regex to `/^(retro-[a-zA-Z0-9][a-zA-Z0-9_-]*|\d+\.\d+(\.\d+)*)$/`, accepting both phase-numbered and named retro IDs.

## Tests

- `tests/integration/loadplan-approved-snapshot-fallback.test.ts` — added three new integration tests covering the cross-identity guard, empty-events guard, and cross-process idempotence healing snapshot.
- `tests/integration/write-retro-session.test.ts` — new end-to-end test calling the real `executeWriteRetro` with `retro-session`.
- `tests/unit/commands/close-finalizer.test.ts` — added tests asserting ledger archive+removal and sibling sweep behavior.

All touched-area tests pass. Pre-existing test failures in unrelated suites (`tests/unit/commands/archive.test.ts`, `tests/unit/commands/rollback.test.ts`, `tests/adversarial/max_coder_revisions`, `tests/security/CheckpointConfigSchema`, etc.) are not introduced by this release and are unrelated to the touched files.

## Migration

No migration required. This is a patch release. Existing workspaces will benefit from the hardened recovery path on next `loadPlan`.

## Known caveats

Concerns flagged during review but kept out of scope for this patch are tracked in zaxbysauce/opencode-swarm#444.
