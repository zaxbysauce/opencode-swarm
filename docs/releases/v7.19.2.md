# v7.19.2

## What changed

Per-task QA-gate evidence files (`.swarm/evidence/<taskId>.json`) are now
fully system-managed. Agents can no longer delete, move, or overwrite
them via Write/Edit/apply_patch or Bash, and a malformed file is
quarantined rather than weaponised against itself.

### Closes the three-defect chain documented in issue #862

1. **`src/hooks/guardrails.ts` — universal write block.** A new helper
   `isPerTaskGateEvidencePath(path, cwd)` (driven by the shared
   `STRICT_TASK_ID_PATTERN` from `src/validation/task-id.ts`) refuses
   any Write/Edit/apply_patch call targeting `.swarm/evidence/<id>.json`
   for every agent — architect, reviewer, critic, test_engineer, docs,
   designer, explorer, sme, coder. The block runs before per-agent
   authority rules and applies inside the `apply_patch` per-target loop
   so a multi-target patch with one evidence path is rejected wholesale.
   Files with non-numeric stems (`.swarm/evidence/test.json`,
   `.swarm/evidence/notes.txt`), phase-scoped subdirectories
   (`.swarm/evidence/retro-3/evidence.json`), and
   `.swarm/evidence/final-council.json` are unaffected and continue to
   flow through their existing per-agent rules.

2. **`src/hooks/guardrails.ts` — Bash bypass closed.** New closure
   `checkBashEvidenceProtection(tool, args)`, wired next to
   `checkDestructiveCommand`, refuses Bash commands that delete, move,
   overwrite, in-place-edit, or pipe-mutate a per-task gate file —
   including `bash -c "rm …"`, `sh -c …`, `find … -delete`, `xargs rm`,
   `sed -i`, and stdout/stderr redirection. Wrapper unwrapping
   (`dcUnwrapWrappers` / `dcSplitSegments`) and the `STRICT_TASK_ID_PATTERN`
   are reused from `checkDestructiveCommand` for parity. Read-only
   commands (`cat`, `ls`, `head`, `jq`, `grep`) remain unaffected.

3. **`src/tools/update-task-status.ts` — quarantine instead of "delete
   it".** When `readTaskEvidenceRaw` throws a non-ENOENT error
   (corrupt JSON, schema mismatch, permission denied), `checkReviewerGate`
   no longer returns `{ blocked: true, reason: "… or delete it …" }`.
   It quarantines the bad file as
   `.swarm/evidence/<taskId>.corrupt-<ts>-<rand>.json` (collision-safe
   via 4-byte `randomBytes` suffix) and falls through to session-state
   evaluation. The canonical hook `recordGateEvidence` rewrites a fresh
   file on the next legitimate gate pass. Telemetry signal updated to
   `Evidence file quarantined (unreadable)`.

4. **`src/agents/architect.ts` — TIER 0 wording.** The Architect prompt
   no longer lists `.swarm/evidence/*` as agent-writable metadata. A
   note now explicitly states the files are hook-managed.

5. **`src/validation/task-id.ts`.** `STRICT_TASK_ID_PATTERN` is now
   exported so the gate-file regex stays in lockstep with production
   task-id validation rather than duplicating the pattern.

## Why

Issue #862 reports an agent that received the message
`Evidence file for task 4.4 is corrupt or unreadable. Fix the file at
.swarm/evidence/4.4.json or delete it to fall through to session state.`
and followed the instruction verbatim — `rm /…/.swarm/evidence/4.4.json`
— destroying durable, append-only QA-gate evidence. The screenshot
attached to the issue shows the deletion succeeding because:

- `DEFAULT_AGENT_AUTHORITY_RULES` has no specific block for per-task
  gate files (the `.swarm/evidence/` directory is *allowed* for
  reviewer/test_engineer to support `final-council.json`,
  `retro-N/evidence.json`, and similar non-task-scoped artefacts).
- `WRITE_TOOL_NAMES` excludes `bash`, so file-authority and lstat checks
  never see a `Bash rm` against `.swarm/evidence/4.4.json` regardless
  of which agent issued it.
- The catch block at lines 254–272 of `update-task-status.ts` told the
  agent the recovery path was *deletion*. There was no quarantine, no
  fall-through, and no positive recovery story.

This release closes all three defects independently, so the failure no
longer requires every layer to hold for the file to survive.

## Migration steps

None. The change is backwards-compatible:

- The new write/Bash blocks fire only against `.swarm/evidence/<numeric-id>.json`
  paths. No file an agent legitimately edits today is affected.
- `recordGateEvidence` writes via `bunWrite`/`renameSync` directly on
  the filesystem and never goes through the OpenCode Write tool, so the
  legitimate writer is unaffected.
- The quarantine path renames any malformed file in place; existing
  callers that previously saw `{ blocked: true, reason: '…corrupt…' }`
  will instead see the session-state-derived result. Any agent that
  was relying on the old block to halt phase progression should
  delegate to the gate agent and call `update_task_status` as normal.
