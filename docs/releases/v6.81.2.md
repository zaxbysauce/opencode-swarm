# v6.81.2

## Summary

**Fixed a critical prompt-vs-runtime discrepancy in the architect agent's plan state rules (#574).** The PLAN STATE PROTECTION section of the architect prompt incorrectly stated that direct structural writes to `.swarm/plan.md` and `.swarm/plan.json` are allowed, while the guardrail has always blocked all direct writes. This caused the architect to quote the stale rule, attempt the edit, and get a PLAN STATE VIOLATION error.

## What changed

- **Architect prompt** (`src/agents/architect.ts`): Removed the stale permissive line ("You may write to plan.md/plan.json for STRUCTURAL changes"). Replaced it with explicit guidance to use `save_plan` for all structural changes.
- **Guardrail error messages** (`src/hooks/guardrails.ts`): Improved both PLAN STATE VIOLATION messages to explicitly name `save_plan` as the path for description/dependency/task-list changes on existing plans, `update_task_status()` for status only, and `phase_complete()` for phase transitions.
- **Tool description** (`src/tools/save-plan.ts`): Updated the `save_plan` tool description to surface the structural-revision use case and document that existing task statuses are preserved by default (set `reset_statuses: true` to start fresh).
- **Tests**: Added `architect-plan-state-protection.test.ts` to pin the PLAN STATE PROTECTION contract and prevent prompt drift. Extended `guardrails-plan-md-protection.test.ts` with tests asserting the error message includes the `save_plan` recovery guidance.

## Why

`plan.md` is auto-generated from `plan.json` by PlanSyncWorker. The canonical workflow for all plan-structure edits is:
1. `save_plan` for adding/removing tasks, updating descriptions/dependencies/acceptance criteria, or renaming phases
2. `update_task_status()` for task completion status only
3. `phase_complete()` for phase transitions only

The prompt no longer contradicts the runtime. The guardrail now teaches the correct recovery path when it blocks. `save_plan` already supported the full structural-revision workflow (and preserves task statuses by default), but the tool description and error messages didn't make this discoverable.

## Migration

No user action required. The architect will now correctly use `save_plan` for structural plan changes instead of attempting illegal direct edits.
