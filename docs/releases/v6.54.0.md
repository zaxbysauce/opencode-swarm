# v6.54.0

## What changed

### PHP / Laravel framework detection

- **New export `isLarastanConfigured(directory: string): boolean`** — reads the first 4,096 bytes of `phpstan.neon` and checks for a reference to either of the two known Larastan package names (`nunomaduro/larastan` or `larastan/larastan`). Only `phpstan.neon` is scanned; `phpstan.neon.dist` is not checked. Fails gracefully (returns `false`) on errors thrown by file-open or file-read operations; `fs.existsSync` is checked first and returns `false` without throwing if the file is absent.

- **New field `staticAnalysisTool: 'larastan' | 'phpstan' | null` on `LaravelCommandOverlay`** — populated by `getLaravelCommandOverlay`. Resolves to:
  - `'larastan'` — `phpstan.neon` is present and contains a Larastan extension reference
  - `'phpstan'` — `phpstan.neon` or `phpstan.neon.dist` is present, but `phpstan.neon` contains no Larastan markers (or does not exist, leaving only `phpstan.neon.dist`)
  - `null` — neither `phpstan.neon` nor `phpstan.neon.dist` is present

  The runtime command remains `vendor/bin/phpstan analyse` in all cases; this field is metadata only.

- **Updated `dist/lang/framework-detector.d.ts`** — declaration file regenerated to match the new interface field and exported function.

- **22 new unit tests** in `tests/unit/lang/framework-detector.test.ts` — covers both `isLarastanConfigured` (missing file, both marker variants, 4,096-byte boundary, `.dist` exclusion, empty file) and `staticAnalysisTool` on the overlay (`'larastan'`, `'phpstan'`, `null` across all config-presence permutations).

## Why

`getLaravelCommandOverlay` previously had no way to distinguish a Larastan-configured project from vanilla PHPStan — both triggered `staticAnalysisCommand` purely via config-file presence with no tool-type metadata. This change surfaces that distinction to consumers (e.g. UI reporting, prompt construction) without altering any runtime behaviour.

## Migration steps

No migration required. The new `staticAnalysisTool` field is additive. Existing consumers that access `LaravelCommandOverlay` properties by name are unaffected.

## Breaking changes

None. Consumers doing exhaustive shape checks (e.g. snapshot tests of the full overlay object) will need to account for the new field.

## Known caveats

- `phpstan.neon.dist`-only projects (no `phpstan.neon`) always resolve to `staticAnalysisTool: 'phpstan'`, even if the `.dist` file contains Larastan markers. `isLarastanConfigured` only reads `phpstan.neon`; the `.dist` file is not scanned.
- Detection is content-based substring matching, not YAML parsing. A commented-out Larastan reference (e.g. `# nunomaduro/larastan`) in the first 4,096 bytes of `phpstan.neon` will be treated as an active Larastan configuration.
- Only the first 4,096 bytes of `phpstan.neon` are scanned. Larastan references beyond that offset will not be detected.
