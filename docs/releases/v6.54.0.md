# v6.54.0

## What changed

### PHP / Laravel framework detection

- **New export `isLarastanConfigured(directory: string): boolean`** — reads the first 4,096 bytes of `phpstan.neon` and checks for a reference to either of the two known Larastan package names (`nunomaduro/larastan` or `larastan/larastan`). Only `phpstan.neon` is scanned; `phpstan.neon.dist` is not checked (it is a distribution baseline that projects override locally). Fails gracefully on any I/O error.

- **New field `staticAnalysisTool: 'larastan' | 'phpstan' | null` on `LaravelCommandOverlay`** — populated by `getLaravelCommandOverlay`. Resolves to:
  - `'larastan'` — `phpstan.neon` contains a Larastan extension reference
  - `'phpstan'` — a PHPStan config is present but contains no Larastan markers
  - `null` — no PHPStan config found

  The runtime command remains `vendor/bin/phpstan analyse` in all cases; this field is metadata only.

- **Updated `dist/lang/framework-detector.d.ts`** — declaration file regenerated to match the new interface field and exported function.

## Why

`getLaravelCommandOverlay` previously could not distinguish a Larastan-configured project from vanilla PHPStan — both just detected the presence of `phpstan.neon`. This makes the distinction available to consumers (e.g. UI reporting, prompt construction) without changing any runtime behaviour.

## Migration steps

No migration required. The new `staticAnalysisTool` field is additive. Existing consumers that access `LaravelCommandOverlay` properties by name are unaffected.

## Breaking changes

None.

## Known caveats

- `phpstan.neon.dist`-only projects (no `phpstan.neon`) always resolve to `staticAnalysisTool: 'phpstan'`, even if the `.dist` file contains Larastan markers. This is intentional: `.dist` is a distribution baseline intended to be overridden by a local `phpstan.neon`.
- Detection is content-based substring matching, not YAML parsing. A commented-out Larastan reference (e.g. `# nunomaduro/larastan`) in the first 4,096 bytes of `phpstan.neon` would be treated as an active Larastan configuration.
- Only the first 4,096 bytes of `phpstan.neon` are scanned. Larastan references beyond that offset will not be detected.
