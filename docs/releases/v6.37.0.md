# Release Notes — v6.37.0

**Minor Release | 2026-03-28**

---

## Overview

This release fixes 5 trust-critical findings (1 CRITICAL, 4 HIGH) identified during code review. The fixes address a secretscan control character bypass, misleading evidence rehydration naming, non-functional model fallback, missing runtime reviewer gate enforcement, and architect prompt/config tool list drift. No breaking changes require migration.

- **Secretscan regex (C-1)**: `containsControlChars` now blocks tab and newline characters, matching all other implementations in the codebase
- **Evidence rehydration (H-29)**: Renamed `readEvidenceFromDisk` to `readGateEvidenceFromDisk` with updated docstring clarifying it reads gate evidence files only
- **Model fallback (H-34)**: Guardrails now actually applies the resolved fallback model to the agent config instead of only injecting an advisory message
- **Reviewer gate (H-35)**: New runtime enforcement blocks coder re-delegation when no reviewer has run, preventing the architect prompt self-check from being the only gate
- **Architect tools (H-30)**: YOUR TOOLS list in architect prompt now matches AGENT_TOOL_MAP exactly (was 14 discrepancies)

---

## Bug Fixes

### Secretscan Control Character Bypass (C-1 CRITICAL)

The `containsControlChars` function in `src/tools/secretscan.ts` used the regex `/[\0\r]/` which missed tab (`\t`) and newline (`\n`) characters. This allowed directory inputs containing tabs or newlines to bypass input validation. The v6.35.0 release notes claimed this was fixed, but the secretscan implementation was not updated.

**Fixed**: Regex changed to `/[\0\t\r\n]/`, matching all other `containsControlChars` implementations across `lint.ts`, `imports.ts`, `evidence-check.ts`, `todo-extract.ts`, and `complexity-hotspots.ts`.

### Evidence Rehydration Naming (H-29)

The function `readEvidenceFromDisk` in `src/state.ts` only reads gate evidence files (written by `recordGateEvidence`), but its name suggested it reads all evidence types. Investigation confirmed the validation schema (`taskId` + `required_gates`) correctly matches `recordGateEvidence` output — the issue was cosmetic, not functional.

**Fixed**: Renamed to `readGateEvidenceFromDisk` with updated docstring explicitly documenting that it reads gate evidence files and why it validates `taskId` + `required_gates`.

### Model Fallback Not Applied (H-34)

When `resolveFallbackModel()` returned a valid fallback model in `src/hooks/guardrails.ts`, the code injected an advisory message suggesting the model but never actually assigned it to the agent config. The comment in `src/agents/index.ts` claimed guardrails "modifies agents[name].config.model directly" but this never happened.

**Fixed**:
- Fallback model is now assigned to `_swarmAgents[agentName].model` when resolved
- `modelFallbackExhausted` is now only set `true` when all configured fallback models have been tried (was incorrectly set after first attempt)
- Advisory message updated from "Consider retrying with" to "Applied fallback model"
- Comment in `src/agents/index.ts` corrected to match actual behavior

### Runtime Reviewer Gate Enforcement (H-35)

The delegation gate tracked reviewer/test_engineer state in `toolAfter` but had no runtime enforcement to block coder re-delegation without reviewer. The only protection was a self-check instruction in the architect prompt.

**Fixed**: Added `toolBefore` handler to the delegation gate hook that throws `REVIEWER_GATE_VIOLATION` when:
- The delegation target is `coder`
- Any task's workflow state is `coder_delegated` (coder ran, no reviewer yet)
- Turbo mode is not active (except for Tier 3 tasks which always require reviewer)

### Architect Tool List Drift (H-30)

The YOUR TOOLS section in the architect prompt listed 20 tools while `AGENT_TOOL_MAP['architect']` in `src/config/constants.ts` had 23 tools. There were 14 discrepancies (7 in prompt only, 7 in config only).

**Fixed**: Replaced the hardcoded tool list with the exact contents of `AGENT_TOOL_MAP['architect']` plus `Task (delegation)`. Added a code comment requiring the two lists stay synchronized.
