**Patch Release | 2026-03-31**

## Overview

This release fixes two bugs that caused incorrect `.swarm/` state reads and session crashes:

- **`.swarm/` path resolution**: `completion_verify`, `phase_complete`, and `check_gate_status`
  resolved `.swarm/` relative to `process.cwd()` instead of the project root. When CWD differed
  from the actual project directory (common in multi-project workspaces), these tools silently
  read stale or missing data from the wrong directory.

- **`test_runner` session crash**: Running `test_runner` without a scope crashed the opencode
  session (GUI/TUI) due to a pipe deadlock pattern, unbounded stdout/stderr memory consumption,
  and a guard error message that coached the LLM to bypass its own safety check.

Additionally, the same pipe deadlock pattern was found and fixed in `build_check` and
`complexity_hotspots`.

---

## Bug Fixes

### `.swarm/` path resolution via `working_directory` parameter

**Root cause**: `createSwarmTool` (create-tool.ts:32) injects `directory = ctx?.directory ??
process.cwd()` into all tools. When an LLM agent's CWD differs from the project root (e.g.,
`C:\opencode\RAGAPPv2` vs `C:\opencode\RAGAPPv3`), tools read `.swarm/plan.json`,
`.swarm/evidence/`, etc. from the wrong directory.

**Fix**: Added a `working_directory` parameter to `check_gate_status`, `completion_verify`,
`phase_complete`, and `test_runner`. This follows the existing pattern used by `save_plan` and
`update_task_status`. A shared `resolveWorkingDirectory` utility validates the parameter against
null-byte injection, path traversal, Windows device paths, and verifies existence via
`fs.realpathSync`.

**Files changed**:
- `src/tools/resolve-working-directory.ts` (new ā€” shared validation utility)
- `src/tools/check-gate-status.ts`
- `src/tools/completion-verify.ts`
- `src/tools/phase-complete.ts`
- `src/tools/test-runner.ts`

### `test_runner` pipe deadlock, unbounded memory, and guard self-defeat

**Root cause 1 ā€” Pipe deadlock**: `runTests()` awaited `proc.exited` before reading
`proc.stdout`/`proc.stderr`. When test output exceeded the OS pipe buffer (~64KB), the child
process blocked on write while the parent waited for exit ā€” a classic deadlock.

**Fix**: Changed to `Promise.all([proc.exited, readBoundedStream(proc.stdout, ...), readBoundedStream(proc.stderr, ...)])` so streams are drained concurrently with the exit wait.

**Root cause 2 ā€” Unbounded memory**: `new Response(proc.stdout).text()` buffers the entire
output in memory. A test suite producing gigabytes of output would cause OOM.

**Fix**: Added `readBoundedStream()` which reads chunks up to a hard cap (512KB), then cancels
the stream remainder to release backpressure.

**Root cause 3 ā€” Guard self-defeat**: The error message for `scope: "all"` without
`allow_full_suite` said "Set allow_full_suite: true" ā€” LLMs follow instructions literally,
defeating the guard entirely.

**Fix**: Changed the error to direct toward `scope "convention"` or `scope "graph"` with a
files array. The `allow_full_suite` parameter name is not mentioned.

### Pipe deadlock in `build_check` and `complexity_hotspots`

The same sequential-read-after-exit pattern existed in two additional tools:

- `src/tools/build-check.ts`: `executeCommand()` awaited `Bun.spawn` result then read streams
  sequentially. Fixed with `Promise.all`.
- `src/tools/complexity-hotspots.ts`: `getGitChurn()` read stdout then awaited exit. Fixed with
  `Promise.all`.

## Migration

No API changes. All fixes are backward-compatible.

The `working_directory` parameter is optional. When omitted, tools fall back to the injected
`ctx.directory` (which defaults to `process.cwd()`), preserving existing behavior. Agents
operating in multi-project workspaces should pass `working_directory` to ensure correct
`.swarm/` resolution.

The `test_runner` guard error message text has changed. If you have tests asserting the old
message (`'Full-suite test execution'` or `'allow_full_suite'`), update them to assert
`'scope "all" is not allowed'`.
