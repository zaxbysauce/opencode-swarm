# v7.12.0 — Provider Error Resilience (closes #756)

## Summary

Closes [#756](https://github.com/zaxbysauce/opencode-swarm/issues/756)
("Still provider returned error message"). The swarm execution loop
previously halted when LLM providers returned common error messages
(HTTP 500, 502, 504, connection errors) because the error classification
regex was too narrow. These transient infrastructure errors were
misclassified as permanent logic errors, incrementing the circuit breaker
counter until the agent was blocked. This release expands error
classification to three tiers: transient, degraded, and permanent.

## What changed

### Three-tier error classification

The `TRANSIENT_MODEL_ERROR_PATTERN` regex in `src/hooks/guardrails.ts`
has been expanded from 9 patterns to 18+, and a new status code
extraction layer checks HTTP codes first before keyword matching.

**Transient** errors (HTTP 408, 429, 500, 502, 503, 504, 529, rate limits,
timeouts, connection errors, gateway errors, service unavailable):
- Do NOT increment `consecutiveErrors`
- Increment `transientRetryCount` (up to `max_transient_retries`, default 5)
- Trigger model fallback if configured

**Degraded** errors (context-length, token-limit, content-filter,
prompt-too-long, max-tokens, input-too-large):
- Do NOT increment `consecutiveErrors` or `transientRetryCount`
- Track model fallback exhaustion (modelFallbackExhausted flag)
- Emit advisory showing fallback model status (e.g., "Fallback model 1/2 considered")
- Differentiate content-filter (policy violation) from size errors

**Permanent** errors (auth failures, logic errors, type errors):
- Increment `consecutiveErrors` toward circuit breaker as before

### Internal helpers added

The following module-private symbols were added to `src/hooks/guardrails.ts` (not exported):

- `extractStatusCode()` — extracts HTTP status codes from error message strings
- `TRANSIENT_STATUS_CODES` — `Set` of recognized transient status codes (408, 429, 500, 502, 503, 504, 529)
- `DEGRADED_ERROR_PATTERN` — regex for degraded error classification (context-length, token-limit, etc.)
- `CONTENT_FILTER_PATTERN` — regex to distinguish content-filter (policy) errors from size errors within the degraded tier

### Advisory messages

Transient and degraded errors now emit user-visible advisory messages:
- `TRANSIENT ERROR: Provider error detected (attempt N/M). Retrying...`
- `DEGRADED: Context-limit or token-limit error detected. Fallback model N/M considered.`
- `DEGRADED: Content policy violation detected (content filter). Fallback model N/M considered.`
- `DEGRADED: Content policy violation detected (content filter). No fallback models available.`
