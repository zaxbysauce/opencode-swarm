# v6.82.0 — Mutation-Testing-Backed Test Quality Gate

## What changed

Implemented a new **Mutation Testing Gate** (Gate 4) integrated into the swarm phase completion pipeline. When enabled, the gate validates test suite quality by generating mutation patches against functions changed in each phase and running them against the test suite.

## New capabilities

**`mutation_test` QA gate flag (default: OFF)**
- Enable per-project via `set_qa_gates({ mutation_test: true })`
- When enabled, `phase_complete` blocks until mutation gate evidence is present with a `pass` verdict

**New architect tools**

- **`generate_mutants`** — Generates mutation patches for a list of source files using LLM. Supports 6 mutation types: boundary-condition shift, negate-condition, remove-call, swap-operator, swap-return, and remove-guard. Returns an empty array (SKIP verdict) when LLM context is unavailable, ensuring graceful degradation.

- **`write_mutation_evidence`** — Persists mutation gate results to `.swarm/evidence/{phase}/mutation-gate.json` using an atomic temp-file + rename pattern. Normalizes verdicts (PASS → pass, WARN → warn, FAIL → fail, SKIP → skip).

**New architect workflow step (step 5.56 in PHASE-WRAP)**
- Checks `effectiveGates.mutation_test` from the QA gate profile
- Skips entirely when turbo mode is active
- Calls `generate_mutants` for changed files, then runs `mutation_test` tool, then calls `write_mutation_evidence`
- FAIL verdict blocks phase completion; WARN verdict proceeds with a warning

## How it's built

- **Phase 1** — Extended `QaGates` interface and `set_qa_gates` tool with `mutation_test: boolean` (default: false)
- **Phase 2** — Created `src/mutation/generator.ts` (LLM-based `generateMutants()`) and both new architect tools
- **Phase 3** — Integrated Gate 4 into `phase_complete.ts`, registered tools end-to-end, updated architect prompt

## Migration steps

- No API or behavior changes to existing tools
- Projects not using `mutation_test: true` are unaffected (gate is OFF by default)
- Existing `phase_complete` calls continue to work without modification

## Breaking changes

None.

## Known caveats

- The mutation gate is expensive (LLM call per phase) — intended for use in CI or high-value feature work, not every session
- Gate evidence file (`.swarm/evidence/{phase}/mutation-gate.json`) is generated only when the gate is enabled and step 5.56 has been executed in the architect session

## Test coverage

- Unit tests: 10/10 generator tests, 11/11 mutation gate tests, 5/5 registration smoke tests
- All tests use `bun:test`
- Full 5-tier test suite run (quality, unit, integration, security, smoke)
