# v6.71.1 — scope persistence hotfix (#519)

Patch release addressing root cause B of #496: declared coder scope was held
in-memory only, so a coder session running in a separate process resolved to
"no scope declared" and every Edit/Write passed the tool-layer authority
check. v6.71.1 persists declared scope to disk, adds a plan-as-scope fallback,
and hardens the architect and coder prompts against bash bypass.

## What changed

### New: `src/scope/scope-persistence.ts`
- `writeScopeToDisk(dir, taskId, files)` — atomic temp-then-rename write at
  `.swarm/scopes/scope-{taskId}.json`, serialised by `proper-lockfile`.
- `readScopeFromDisk(dir, taskId)` — fd-based read with `O_NOFOLLOW`; rejects
  symlinked leaves, symlinked parent directories (via `realpath` containment
  on `.swarm/scopes/`), schema version ≠ 1, expired TTL (24 h default),
  `declaredAt` in the future, and `files` arrays over 10 000 entries.
- `readPlanScope(dir, taskId)` — falls back to
  `plan.json:phases[].tasks[].files_touched`; rejects plans over 10 MiB.
- `resolveScopeWithFallbacks()` — ordered chain: in-memory → disk →
  `plan.json` → pending-map.
- `clearScopeForTask()` / `clearAllScopes()` — called by `/swarm close`.

### Wired into existing hooks
- `declare_scope` tool persists on success and always returns a standing
  `SCOPE ENFORCEMENT NOTE` warning reminding callers that bash-based writes
  bypass the tool layer (syscall-layer remediation tracked in #520).
- `guardrails.ts` and `scope-guard.ts` read through the 4-layer chain.
- `/swarm close` sweeps `.swarm/scopes/` so stale scope cannot leak forward.

### Prompt hardening
- Architect gains a `SCOPE DISCIPLINE` section requiring `declare_scope`
  before every coder delegation; WRITE BLOCKED responses must re-scope
  rather than bypass via bash / eval / heredoc.
- Coder gains a rule-based `WRITE BLOCKED PROTOCOL`: if the Edit/Write/Patch
  tool denies a path, **no** other mechanism may write it. The enumeration
  (shell redirects, in-place editors/interpreters, patch/decode utilities,
  network writes, indirection wrappers) is illustrative; the rule is
  exhaustive.

### Defence-in-depth hardening (from adversarial review)
- `taskId` stored in the scope file must match the filename
  (cross-pollination blocked).
- Future `declaredAt` timestamps rejected (clock-skew / attacker-crafted).
- `files` array capped at 10 000 entries; `plan.json` capped at 10 MiB;
  scope file capped at 2 MiB (DoS caps).
- Windows reserved device names (`CON`, `NUL`, `LPT1`, trailing-dot
  variants) rejected as task IDs.

## Why

Issue #496 (write-authority bypass via bash) has three root causes. Root
cause B — in-memory-only scope — is what lets a legitimate, well-behaved
coder process quietly pass authority checks with empty scope. It was
discoverable by any agent that read the state module, and it broke the
intended defence: declared scope is supposed to be binding. This release
makes scope durable across process boundaries and makes the architect /
coder prompts explicit about the authority model. Full syscall-layer
remediation (`sed -i`, `echo >`, etc.) is tracked in #520.

## Migration steps

None. `declare_scope` arguments and semantics are unchanged. The tool now
writes a scope file as a side-effect and always returns a `warnings` array
(previously omitted when empty); callers that read `result.warnings` should
already be defensive and no regression is expected.

## Breaking changes

None.

## Known caveats

- **Bash writes still bypass the check.** This release is a tool-layer
  mitigation, not a syscall-layer enforcement. A coder process that runs
  `sed -i`, `echo > file`, `python -c 'open(…).write(…)'`, or any other
  non-Edit/Write/Patch path can still produce undeclared diffs. #520
  tracks the syscall-layer work.
- **Windows symlink coverage is partial.** `realpath` resolves Windows
  junctions, but `O_NOFOLLOW` is a no-op on Windows (falls back to 0).
  Parent-directory containment remains the primary guard on that
  platform; leaf-file TOCTOU on Windows is not fully closed.
- **Stale lockfile DoS window is 30 s.** A crashed writer leaves a lock
  for up to `LOCK_STALE_MS`; concurrent `declare_scope` calls during that
  window fail silently and the architect relies on in-memory state. This
  is acceptable because in-memory state remains authoritative inside the
  live process; disk is a fallback.
- **Temp files from crashed rename are bounded only by `/swarm close`.**
  If Bun crashes between `Bun.write(tmp)` and `renameSync`, the temp file
  lingers. `/swarm close` rm -rf's `.swarm/scopes/` so accumulation is
  capped at one session.
