# OpenCode Swarm v6.16.1 Release Notes

## Summary

Version 6.16.1 delivers **Spec Lifecycle Fixes** that improve the specification workflow throughout the project lifecycle. This release addresses edge cases in spec command handling, stale spec detection, spec archival, and plan ingestion.

- **Explicit spec command override**: `/swarm specify` and `/swarm clarify` now bypass RESUME on projects with incomplete plans
- **Stale spec detection**: Three-option prompt when entering PLAN with a spec that appears to be from a different feature
- **Spec archival**: Old spec.md is preserved to `.swarm/spec-archive/` before any overwrite
- **Plan ingestion spec gate**: Soft gate when a pre-written plan is provided without a spec
- **Issue #17**: Confirmed guardrail delegation-awareness already present in v6.16.0 (no regression)
- **Issue #22**: Confirmed placeholder detection already present and working (fixed in v6.14.1)

---

## Detailed Features

### 1. Explicit Spec Command Override

**Problem**: RESUME unconditionally won over SPECIFY — users with an incomplete plan.md could not trigger `/swarm specify` or `/swarm clarify`, even when they wanted to start fresh or create a new spec.

**How it works**: Added EXPLICIT COMMAND OVERRIDE as priority 0 in MODE DETECTION. Explicit spec commands (`/swarm specify`, `/swarm clarify`, "specify", "write a spec", "define requirements", "create a spec") now bypass RESUME. RESUME is demoted to priority 1 and is gated on the user NOT having issued an explicit spec command.

---

### 2. Stale Spec Detection

**Problem**: When starting a new plan on a project with an old spec.md from a different feature, the Architect had no way to detect or warn about a scope mismatch.

**How it works**: In MODE: PLAN, if spec.md exists, the Architect now heuristically compares spec headings against the current work context. If the spec appears stale or mismatched, the Architect offers three options:

1. **Archive the existing spec and start fresh** — moves current spec to archive, proceeds to SPECIFY mode
2. **Keep the existing spec** — proceeds with current spec as-is
3. **Skip the spec gate and plan directly** — bypasses spec entirely, plans directly

---

### 3. Spec Archival

**Problem**: On the "overwrite" path in MODE: SPECIFY, the old spec was silently deleted with no way to recover it.

**How it works**: Before overwriting, the Architect now reads the existing spec, extracts the version from the spec heading or package.json, copies the current spec to `.swarm/spec-archive/spec-v{version}.md` (fallback: `spec-{YYYY-MM-DD}.md`), logs the archive location, then proceeds with new spec generation.

---

### 4. Plan Ingestion Spec Gate

**Problem**: When users pasted an existing implementation plan (e.g., "implement this plan"), the Architect would proceed directly to planning with no offer to generate a spec from the provided plan.

**How it works**: In MODE: PLAN, if the input looks like a plan (markdown with phase/task structure, or explicit ingestion phrases like "ingest this plan" / "implement this plan" / "prepare for implementation") AND no spec.md exists, the Architect now soft-gates with two options:

1. **Generate a spec from the provided plan** — reverse-engineers requirements from the plan
2. **Skip and proceed directly** — planning proceeds exactly as before

This is a soft gate — the user can always skip if they prefer to plan without a spec.

---

### 5. Issue #17 Fix: Guardrail False Positive

**Problem**: In v6.16.0, the guardrails hook incorrectly triggered a self-coding warning when the Architect edited source files via coder delegation, because the delegation state was not checked before firing the warning.

**How it works**: Verified that delegation-awareness was already present in v6.16.0 (`src/hooks/guardrails.ts` lines 228-231). The `delegationActive` flag suppresses the warning when a coder delegation is active. No code change was needed — this was a verification-only task.

---

### 6. Issue #22 Fix: Placeholder Plan Content

**Problem**: Reports that plan.md/plan.json was generated with placeholder bracket content like `[Project]`, `[task]`, `[N]` instead of real content.

**How it works**: Confirmed fixed in v6.14.1. The fix is fully present in v6.16. `detectPlaceholderContent()` in `src/tools/save-plan.ts` rejects plans with placeholder patterns before any file is written. The architect prompt also contains explicit warnings against using placeholder text.

---

## Upgrade Notes

- **Backward compatible**: All existing behavior preserved. No breaking changes.
- **Explicit spec commands**: `/swarm specify` and `/swarm clarify` now work even when a partial plan.md exists — no need to delete the plan first.
- **Stale spec detection**: When entering PLAN mode with an existing spec.md, you may be prompted to verify the spec is relevant to your current task.
- **Spec archival**: When overwriting an existing spec.md, the old version is automatically archived to `.swarm/spec-archive/`.
- **Plan ingestion**: When pasting a pre-written plan without a spec.md, you'll see a one-time offer to generate a spec first. Choose "Skip" to proceed as before.
- **Issue #17**: No action needed — guardrail delegation-awareness is already present.
- **Issue #22**: No action needed — placeholder detection is already present and working.

---

## Release Type

This is a **prompt-only release**. All changes in Tasks 1.1–1.4 are implemented through updated agent prompts (Architect). Tasks 2.1 and 1.5 were verification-only — no code changes were required.
