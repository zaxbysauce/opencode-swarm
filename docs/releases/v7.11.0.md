# v7.11.0 — Curator + knowledge-to-skill + skill_improver + spec_writer (closes #629)

## Summary

Closes [#629](https://github.com/zaxbysauce/opencode-swarm/issues/629)
("Agent for skill improver with very few requests"). Ships an end-to-end
behavioural bridge that makes stored swarm knowledge actually affect swarm
behaviour: actionable directives, action-aware retrieval, an enforce/warn
gate wired into runtime, a knowledge-to-skill compiler, a quota-bounded
expensive-model `skill_improver` agent that performs **real LLM dispatch**
on an ephemeral OpenCode session, and a separate `spec_writer` agent for
`.swarm/spec.md` authorship.

## What changed

### Knowledge schema v2 — actionable directives

`KnowledgeEntryBase` carries optional `triggers`, `required_actions`,
`forbidden_actions`, `applies_to_agents`, `applies_to_tools`,
`verification_checks`, `source_refs`, `source_knowledge_ids`,
`generated_skill_slug`, `generated_skill_path`, `directive_priority`,
`last_applied_at`, `last_acknowledged_at`. Existing v1 JSONL keeps reading;
entries are normalised on read.

Retrieval-outcome counters now distinguish `shown_count`,
`acknowledged_count`, `applied_explicit_count`, `ignored_count`,
`violated_count`, `succeeded_after_shown_count`,
`failed_after_shown_count`. Legacy `applied_count`,
`succeeded_after_count`, `failed_after_count` are FROZEN — preserved on
disk for backward compatibility but no longer auto-incremented. The v1→v2
normalizer back-fills `shown_count` from the legacy `applied_count` so
existing on-disk data still trips the low-utility audit.

### Action-aware retrieval

`readContextualKnowledge(directory, config, ctx)` ranks against current
tool, target agent, task id, file paths, mode, recent failures. Critical
directives whose trigger / tool / agent matches are forced into the top-N.
Cache key: `phase | tool | action | targetAgent | taskId | files`. Task id
flows in via the new `getCurrentTaskId(plan)` helper. `lastUserMessage` is
deliberately NOT in the cache key (would invalidate every turn).

### Active injection contract

Architect receives a structured `<swarm_knowledge_directives>` block with
id, confidence, priority, trigger, required, forbidden, skill,
verification per record. Architect prompt requires emitting
`KNOWLEDGE_APPLIED: <id>`, `KNOWLEDGE_IGNORED: <id> reason=...`, or
`KNOWLEDGE_VIOLATED: <id> reason=...` before high-risk actions. The
`knowledge_ack` tool is the deterministic alternative.

### Enforce / warn gate wired into runtime

- `experimental.chat.messages.transform` integration scans the latest
  architect-authored message for ack markers via a dedup-aware path
  (per-(sessionId, knowledgeId, result, dayKey) key) so inline markers and
  `knowledge_ack` do not double-count.
- A new fail-closed `tool.execute.before` hook **throws**
  `KNOWLEDGE_ENFORCE_GATE_DENY` (correct OpenCode plugin contract; not
  `output.error`) when the architect attempts a high-risk action
  (`save_plan`, `update_task_status`, `phase_complete`, `Task`) with an
  unacknowledged critical directive in `enforce` mode. `warn` mode appends
  to `.swarm/events.jsonl` and returns. In-memory
  `swarmState.currentCriticalShownIds` (populated at injection) prevents
  O(swarm × hive) re-scan per toolBefore.

### Knowledge-to-skill compiler

New `src/services/skill-generator.ts` and four tools: `skill_generate`,
`skill_list`, `skill_apply`, `skill_inspect`.

- Drafts → `.swarm/skills/proposals/<slug>.md` with `status: draft`.
- Active → `.opencode/skills/generated/<slug>/SKILL.md` with `status: active`.
- Slug sanitiser; path traversal rejected.
- Generator-stamped skills are not overwritten without `force=true`.
- `activateProposal` parses draft frontmatter via a bounded YAML parser and
  stamps source knowledge entries with `generated_skill_slug` /
  `generated_skill_path`. Malformed frontmatter activates but does NOT
  stamp — knowledge is never mutated from bad metadata.

### Curator integration

Curator output parser extracts strict-JSON `knowledge_application_findings`
and `skill_candidates` blocks. Malformed JSON is silently dropped — no
knowledge or skill writes happen on bad output. When
`curator.skill_generation_enabled` is true (default), high-confidence
candidates emit draft SKILL.md proposals.

`normalizeAgentName` no longer hard-codes
`(mega|paid|local|lowtier|modelrelay)_`. It calls
`getCanonicalAgentRole(name, swarmState.generatedAgentNames)` so arbitrary
swarm IDs (`banana_coder`, `acme-prod_reviewer`,
`customer123_test_engineer`) resolve correctly when registered;
`not_an_architect` does NOT collapse to `architect`. Empty registry falls
back to a permissive suffix scan against `ALL_AGENT_NAMES`.

### `skill_improver` agent — real LLM dispatch (closes #629)

- New agent registered in all maps. Independent model under
  `agents.skill_improver.model`.
- New `src/hooks/skill-improver-llm-factory.ts` mirrors curator-llm-factory:
  ephemeral OpenCode session per call, abort/timeout/NotFoundError-on-abort
  handling, response is `parts[].text` joined.
- New `swarmState.skillImproverAgentNames` registry resolves the right
  prefixed agent under multi-swarm configs.
- Quota: `.swarm/skill-improver-quota.json` under `proper-lockfile`. New
  `releaseQuota` helper for pre-network failures. **Anti-flake:** once a
  network call has begun, the slot stays consumed even on failure.
- Pre-flight refusal (`no_llm_client`) when no client wired and
  `allow_deterministic_fallback: false` — quota untouched.
- Proposal frontmatter tagged `source: llm | deterministic_fallback`.

```jsonc
"skill_improver": {
  "enabled": true,
  "max_calls_per_day": 10,
  "trigger": "manual",
  "targets": ["skills", "spec", "architect_prompt", "knowledge"],
  "write_mode": "proposal",
  "require_user_approval": true,
  "quota_window": "utc",
  "allow_deterministic_fallback": true
}
// Set the actual model under agents.skill_improver.model
```

### `spec_writer` agent

New independently-configurable agent for `.swarm/spec.md`. Uses the safe
`spec_write` tool (256 KiB cap, top-level heading required, atomic write).
Cannot edit source files. Architect prompt routes substantive spec authoring
here while Architect stays cheap.

### Configuration precedence

| Agent | Set model under | Deprecated location |
|-------|----------------|---------------------|
| `skill_improver` | `agents.skill_improver.model` | top-level `skill_improver.model` (deprecated, no effect) |
| `spec_writer` | `agents.spec_writer.model` | top-level `spec_writer.model` (deprecated, no effect) |

### Baseline fixes

- `curator-analyze` accepts `rewrite` action.
- Curator phase parser accepts both `event` and `type` fields on incoming
  phase events.

## Why

Pre-existing knowledge was passive prose. Architect received terse lessons
like `[S] lesson text ✓✓` with no IDs, triggers, required actions, forbidden
actions, or acknowledgment contract. Retrieval outcome tracking conflated
"shown" with "applied". Mature lessons stayed as text and never compiled
into reusable skills. Issue #629 specifically asked for a low-frequency,
expensive-model improver agent under a daily budget; the codebase had no
path for that.

## Migration steps

Zero-touch. v1 JSONL keeps reading; legacy `retrieval_outcomes` fields are
frozen and back-filled into v2 fields on read. New agent model overrides
should be set under `agents.<name>.model` going forward; the old top-level
fields no-op (config-doctor will surface a warning).

## Breaking changes

None. Two behavioural shifts to be aware of:

- Architect-emitted `KNOWLEDGE_APPLIED:` markers are now recorded in
  `.swarm/knowledge-application.jsonl` and bump per-entry counters.
  Architects unaware of the contract will get warnings in `warn` mode (the
  default); the gate only blocks in `enforce` mode.
- Legacy `applied_count` no longer increments on phase outcomes. New code
  should read `applied_explicit_count` (for "actually applied") or
  `shown_count` (for "saw it").

## Known caveats

- **CI cannot exercise the live OpenCode `client.session.prompt` path.**
  `swarmState.opencodeClient` is only wired by the OpenCode runtime; tests
  use a mocked delegate to assert dispatch shape, quota policy, and source
  tagging. Live-LLM verification of `skill_improve` requires a manual smoke
  run from an OpenCode session with the agent registered and a real model.
- `allow_deterministic_fallback` defaults to `true` in this release as a
  one-minor compatibility window. It will flip to `false` in the next
  minor — set it explicitly to lock behaviour.
- `applied_count` and friends remain on disk as frozen legacy fields. If
  you have downstream analytics reading them, migrate to `shown_count` or
  `applied_explicit_count`.
