# v7.8.2 — Fix: session not found unhandled rejection

## Bug Fix

### Session not found error surfacing in TUI/terminal

**Root cause**: When the curator LLM delegate timed out, an `AbortController` fired
which triggered two concurrent actions: (1) the timeout `Promise` in `curator.ts`
rejected with `CURATOR_LLM_TIMEOUT`, settling the `Promise.race()`, and (2) the
abort listener in `curator-llm-factory.ts` called `cleanup()`, deleting the
ephemeral OpenCode session. The still-in-flight `client.session.prompt()` call then
received a `NotFoundError: Session not found` from the SDK because the session was
already deleted. Since `Promise.race()` had already settled, this rejection became
an **unhandled promise rejection** that surfaced as a visible error in the
OpenCode TUI/terminal.

**Fix (two parts)**:

1. **`src/hooks/curator-llm-factory.ts`** — Added a catch block around
   `client.session.prompt()` that converts any error thrown while
   `signal.aborted === true` into `CURATOR_LLM_TIMEOUT`. This gives a clean,
   expected error instead of a confusing `NotFoundError`.

2. **`src/hooks/curator.ts`** — In both the `CURATOR_INIT` and `CURATOR_PHASE`
   timeout race patterns, hoisted the `llmDelegate` promise and attached
   `void delegatePromise.catch(() => {})` before passing it to `Promise.race()`.
   This marks the race loser's eventual rejection as handled, preventing it from
   surfacing as an unhandled rejection in Bun/Node.js.

**Tests**: Added two regression tests in
`tests/unit/hooks/curator-llm-factory.test.ts`:
- `NotFoundError from prompt is converted to CURATOR_LLM_TIMEOUT when signal is aborted`
- `non-abort NotFoundError from prompt is re-thrown unchanged`
