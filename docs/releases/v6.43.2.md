# v6.43.2

## What changed

### Curator re-trigger loop fix — prevents session leak and digest ballooning

`runCuratorPhase` had no idempotency guard. Each call for the same phase appended a duplicate digest entry to `curator-summary.json`, re-emitted compliance events to `events.jsonl` (which were re-read on the next run, amplifying the count), and created a new ephemeral LLM session that could time out and leak. In multi-swarm deployments, this caused dozens of orphaned sessions and a ballooning `PRIOR_DIGEST` with 15+ duplicate "Phase N completed" entries.

Added a deduplication guard that checks `phase_digests` before proceeding. If the phase was already digested, returns cached data with `summary_updated: false`.

### Curator LLM timeout — configurable and increased from 30s

The hardcoded 30-second timeout for curator LLM delegation was insufficient. The budget must cover ephemeral session creation overhead plus a full model response, which varies by model and queue depth.

- Default increased from 30s to 300s (5 minutes)
- Now configurable via `curator.llm_timeout_ms` in swarm config (range: 5000–600000ms)
- Follows the precedent set by `incremental_verify.timeoutMs`

### Ephemeral session leak on timeout — AbortController cleanup

When `Promise.race` timed out, the ephemeral session's SDK call continued running and the `finally` cleanup block only fired when the abandoned promise eventually settled. Added `AbortController` support to `CuratorLLMDelegate` — when the timeout fires, the abort signal triggers immediate session cleanup rather than waiting for the SDK call to settle.

### Task counting bug — "0 tasks recorded" despite completed tasks

`runCuratorPhase` counted completed tasks by filtering `events.jsonl` for `type === 'task.completed'`, but that event type is never emitted. Task status lives exclusively in `plan.json`. Changed to read `plan.json` via `loadPlanJsonOnly` and count tasks with `status === 'completed'` for the target phase. The digest summary now shows accurate counts (e.g., "13/13 tasks completed").

## Why

- **Session leak**: Users reported dozens of orphaned ephemeral curator sessions in TUI, consuming resources
- **Digest ballooning**: Repeated `phase_complete` calls for the same phase caused the `PRIOR_DIGEST` to grow unboundedly, degrading LLM context quality
- **Timeout failures**: 30s was unrealistic for session creation + queue wait + model response, especially with non-fast models
- **Incorrect counts**: "0 tasks recorded" was misleading when all tasks were actually completed

## Migration

No changes required. The new `curator.llm_timeout_ms` config field is optional with a 300s default. Users with slow curator models can increase it:

```json
{
  "curator": {
    "llm_timeout_ms": 600000
  }
}
```

## Known caveats

- `session.delete()` on timeout may not cancel an in-flight prompt server-side (depends on SDK server semantics). The client-side cleanup is correct, but the server may continue processing until it finishes or times out naturally.
- The dedup guard prevents re-running LLM analysis for an already-digested phase. The `curator_analyze` tool's knowledge application still works (it's independent of the digest), but fresh LLM recommendations are not regenerated on re-analysis of the same phase.
