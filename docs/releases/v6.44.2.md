# v6.44.2 — CI stability: remove `continue-on-error` and fix 50+ pre-existing test failures

## What changed

### CI workflow
- Removed `continue-on-error: true` from two unit-test steps that were masking 43 pre-existing failures
- Added per-file subprocess isolation to `Unit tests (commands, config)` step to prevent `--smol` module-cache poisoning across test files (#330 pattern)
- All CI steps now fail fast on real regressions

### New module: `src/knowledge/hive-promoter.ts`
- Created to satisfy `tests/unit/knowledge/hive-promoter-task2-2.test.ts` which imported from a non-existent path
- Exports `validateLesson()`, `getHiveFilePath()`, `promoteToHive()`, `promoteFromSwarm()`
- `validateLesson()` uses negative lookbehind (`(?<!\.)`) on `exec`/`eval` patterns so method calls like `process.exec()` are not falsely flagged as dangerous

### `src/commands/promote.ts` refactor
- `handlePromoteCommand` now imports `promoteToHive`, `promoteFromSwarm`, and `validateLesson` from `src/knowledge/hive-promoter` instead of the hooks layer
- Fixes the `--category` flag accepting arbitrary categories (previously rejected by the hooks validator's strict category enum)

## Why

The CI unit job was running with `continue-on-error: true` on two steps, hiding 43+ test failures. This made it impossible to detect regressions. This release resolves all masked failures so CI can enforce correctness going forward.

The `parallel-wiring` and `checkpoint` failures were caused by the async `tryAcquireLock` migration (commit `6a750bb`) and the global git signing config respectively — neither was caught because the steps masked errors.

The Bun 1.3.9 event loop saturation bug (after many `fs.watch` create/destroy cycles) was fixed by converting async I/O in background test setup/teardown to synchronous operations.

## Migration steps

None. All changes are test infrastructure and internal module reorganization. The `handlePromoteCommand` public API is unchanged.

None.
