# OpenCode Swarm v6.33.1 Release Notes

## Summary

Version **6.33.1** is a stabilization hotfix that resolves three critical bugs in scope containment and session state management, adds session stability improvements (snapshot debounce, diagnostics, stale state recovery), and introduces performance mode configuration.

---

## Critical Bug Fixes

### 1. Scope containment was non-functional (CRIT-1)

`declaredCoderScope` was written to the architect session in `messagesTransform` but read from the coder session in `scope-guard`. Since architect and coder sessions are different objects, scope containment never triggered. Fixed by adding a `pendingCoderScopeByTaskId` fallback Map in delegation-gate that scope-guard checks when the coder session's scope is null.

### 2. Stored input args leaked when delegation-gate disabled (CRIT-2)

`setStoredInputArgs` fired unconditionally in guardrails `toolBefore`, but cleanup (`deleteStoredInputArgs`) only ran inside delegation-gate's `toolAfter`. When delegation-gate was disabled, stored args accumulated indefinitely. Fixed by moving cleanup to the end of the `tool.execute.after` chain in index.ts, after all handlers have completed.

### 3. Scope-guard resolved paths against wrong directory (CRIT-3)

`isFileInScope` used bare `path.resolve(filePath)` which resolves against `process.cwd()` — not the workspace directory. The `_directory` parameter was received but unused. Fixed by renaming to `directory` and passing it through to all `path.resolve` calls.

---

## Session Stability

### Snapshot write debounce

Snapshot writes fired on every single `tool.execute.after` call, causing massive I/O pressure that could overwhelm the SSE drain loop and lead to session loss. Added a 2-second trailing-edge debounce that coalesces rapid tool calls into a single write. A `flushPendingSnapshot()` function is called on critical transitions (phase-complete, handoff) to ensure immediate persistence.

### Session-loss diagnostics

- `writeSnapshot` catch block now logs failures with `console.warn` instead of silently swallowing errors
- Added `console.debug` tracing at the start of `tool.execute.after` and `chat.message` handlers for session lifecycle debugging

---

## Stale State Recovery

### Snapshot field validation

Malformed agent sessions in `state.json` (missing required fields like `agentName`, `lastToolCallTime`, `delegationActive`) are now skipped during rehydration instead of being injected into memory, which previously caused confusing downstream errors.

### No-op work detector

A new advisory fires when an agent makes 15+ consecutive tool calls without any file modifications (configurable via `guardrails.no_op_warning_threshold`). Fires once per session to avoid noise.

### `/swarm reset-session` command

New command that deletes `.swarm/session/state.json` and clears in-memory agent sessions while preserving plan, evidence, and knowledge. Use after a corrupted session to start fresh without losing project state.

---

## Performance

### `execution_mode` configuration

New `execution_mode` option (`strict`, `balanced`, `fast`) gates optional quality hooks:

| Mode | Behavior |
|------|----------|
| `strict` | Runs all hooks — slop-detector, incremental-verify, compaction (v6.32 behavior) |
| `balanced` (default) | Skips optional quality hooks for faster iteration |
| `fast` | Same as balanced — reserved for future aggressive optimizations |

```jsonc
{
  "execution_mode": "fast"
}
```

---

## Migration Notes

- No breaking changes. All new features have backward-compatible defaults.
- Existing `state.json` files are fully compatible — no version bump, no migration needed.
- The `/swarm turbo` slash command already existed and now has a documented config equivalent.
