# v7.17.0 — Language-Agnostic Plugin

## What changed

Seven phases of work to lift opencode-swarm out of its TypeScript-first hardcoding and onto a registry-driven `LanguageBackend` extension seam. New languages now ship as a single profile entry plus an optional backend file.

### Phase 0 — `isCommandAvailable` Invariant 3 fix
- `src/build/discovery.ts:isCommandAvailable` now sets explicit `cwd`, `stdin: 'ignore'`, `timeout: 3000ms`, and `'ignore'` stdio. Pre-fix it omitted three of the four invariant-3 properties; every caller (13 production sites) inherited the violation.
- DI seam (`_internals.spawnSyncImpl`) added so the new `tests/unit/build/discovery.test.ts` "Invariant 3 — subprocess properties" suite asserts the contract directly via spy, not just by reading source.

### Phase 1 — `LanguageProfile` enrichment + drift guards
- `LanguageProfile.treeSitter.commentNodes` added (optional; populated for all 12 production profiles using values from `src/lang/registry.ts`). Source of truth lives in profiles; parser registry is the projection.
- `LanguageRegistry.register()` now THROWS on duplicate id and on duplicate non-`parserOnly` extension claim (was silent overwrite — hid configuration drift).
- `LanguageRegistry.unregister(id)` added so tests that mutate the global singleton can clean up after themselves (Bun's per-file-but-shared-process runner otherwise leaks fixture profiles across test files).
- `mocha` framework added to the TypeScript profile (`detect: .mocharc.json`, `cmd: npx mocha`, priority 7) so the dispatch path can resolve mocha-driven JS projects.

### Phase 2 — `LanguageBackend` interface, default backend, TypeScript backend
- New `src/lang/backend.ts` interface. Hook surface: `detectProject`, `selectTestFramework`, `buildTestCommand`, `parseTestOutput`, `testFilesFor`, `extractImports`, `selectBuildCommand`, `selectFramework`, `selectEntryPoints`.
- `src/lang/default-backend.ts` provides registry-driven defaults plus the full 14-framework `buildTestCommand` and `parseTestOutput` switches (ported verbatim from the legacy logic that previously lived in `src/tools/test-runner.ts`). Coverage flags, scope-dependent file inclusion, platform-specific python/python3, pester `-EncodedCommand`, gradlew probing, ctest build-dir probing, flutter-vs-dart selection, bundle/rspec detection, minitest `require_relative` — all there.
- `src/lang/dispatch.ts` adds `pickBackend(dir)`. Walks up to the manifest root, picks the dominant detected language, returns the registered or defaulted backend. Bounded LRU cache (capacity 64) keyed by manifest hash (`size:mtimeMs:ino`).
- Concrete backends `src/lang/backends/typescript.ts`, `python.ts`, `go.ts`. All three start from `defaultBackendFor(profile)` and override only their language-specific hooks (test-framework selection that honors `package.json#scripts.test`, language-specific import extractors, framework + entry-point detectors).
- `tests/unit/lang/backend-purity.test.ts`: static-analysis test failing PRs that import `bun:...`, reference `Bun.*`, or use spawn primitives in any backend file. Backends never spawn — the single spawn site stays in `src/tools/test-runner.ts` and `src/build/discovery.ts:isCommandAvailable`.

### Phase 3b — full test-runner dispatch
- `src/tools/test-runner.ts` routes `detectTestFramework`, `buildTestCommand`, AND `parseTestOutput` through the dispatch path. **Dispatch is the default**; `SWARM_LANG_BACKEND=legacy` is the opt-out rollback for one release.
- When dispatch returns `'none'` (no profile, missing binary), the test-runner FALLS BACK to the legacy switch as a backstop. This eliminates the Phase-3-era heuristic divergence for Rust/Python and means PowerShell/Pester users — for whom there is no LanguageProfile — keep working through the legacy detector.
- `tests/unit/tools/test-runner-dispatch-parity.test.ts` extended with build/parse parity assertions for vitest, bun, pytest, go-test plus the framework=none null-return contract.

### Phase 4b — full architect prompt template substitution
- `src/agents/template.ts`: strict `{{KEY}}` renderer. Throws on unknown placeholders; auto-escapes backticks + `${...}` for safe inclusion in TS template literals.
- `src/agents/project-context.ts`: `buildProjectContext(directory)` resolves `pickBackend` and queries every behavior hook. Lint-command glob patterns (`*.csproj`, `*.xcodeproj`) now resolve correctly (previously failed via literal `fs.accessSync('*.csproj')`).
- Every `ProjectContext` field is now substituted in the architect prompt: `PROJECT_LANGUAGE`, `PROJECT_FRAMEWORK`, `BUILD_CMD`, `TEST_CMD`, `LINT_CMD`, `ENTRY_POINTS`, `CODER_CONSTRAINTS`, `TEST_CONSTRAINTS`, `REVIEWER_CHECKLIST`, `PROJECT_CONTEXT_SECONDARY_LANGUAGES`. The PR #825 adversarial review flagged the last four as computed but never substituted; that gap is closed.
- `PROJECT_FRAMEWORK` and `ENTRY_POINTS` populated from backend hooks. TypeScript reads `package.json` deps + `bin`/`main`/`module`/`exports`. Python reads `pyproject.toml`/`requirements.txt`/`setup.py` and `[project.scripts]` + common entry filenames. Go reads `go.mod` requires + `main.go` / `cmd/*/main.go`.
- `src/index.ts:initializeOpenCodeSwarm` calls `buildProjectContext` between `ensureSwarmGitExcluded` and `getAgentConfigs`, wrapped in `withTimeout(2000ms)` with `.catch(() => null)` fail-open per Invariant 1.
- `tests/unit/agents/init-fail-open.test.ts` is the regression guard for v7.0.3 issue #704 + v7.3.3 git-hygiene precedent: a hung `pickBackend` cannot block the manifest return.

### Phase 5 — Python and Go backends + full impact-graph routing
- `src/lang/backends/python.ts`: `import x`, `import x.y`, `import x as y`, `import a, b`, comma-list with intermediate aliases (`import os, sys as system, pathlib` — all three captured), `from x import y` (relative `.`/`.foo` and dotted `x.y`), multi-line `from x import (\n a,\n b\n)`. Aliases keep the source module per analyzer convention.
- `src/lang/backends/go.ts`: single-line `import "x"`, aliased / side-effect / dot variants, and grouped `import (\n "a"\n "b"\n)` with mixed alias forms inside.
- `src/test-impact/analyzer.ts` walks Python (`test_*.py`, `*_test.py`, `tests/*.py`) and Go (`*_test.go`) test files in addition to TS/JS. Per-file extension routing through `lang/backends/python._internals` and `lang/backends/go._internals` builds cross-language impact graphs. Python relative imports resolve via dotted-path walk + `__init__.py` probing; Go relative imports resolve via package-directory enumeration (excluding `*_test.go`).
- Multi-line TS/JS ES imports (`import {\n  a,\n  b\n} from 'mod'`) now captured by both the analyzer and the TS backend — pre-Phase-5 the regex used `.` which doesn't cross newlines.
- `tests/unit/test-impact/analyzer-multi-lang.test.ts`: 8 new tests for Python+Go test walking, relative-import resolution, and mixed-language repo handling.
- `tests/unit/lang/framework-and-entry-points.test.ts`: 14 new tests for `selectFramework`/`selectEntryPoints` across all three concrete backends.

### Phase 6 — extension-seam documentation
- `docs/adding-a-language.md`: contributor guide covering the three-registry architecture, step-by-step profile/parser/backend additions, backend invariants, common pitfalls, rollout checklist.
- README.md and `docs/index.md` updated with cross-references.

## Why

Three motivations:

1. **Project-context substitution** — the architect prompt had shipped with literal `{{PROJECT_LANGUAGE}}` / `{{BUILD_CMD}}` placeholders for months. The model received unrendered curly braces, both confusing and a wasted opportunity to ground the agent in the user's actual project. Every placeholder now resolves to a detected value or the explicit `'unresolved (run /swarm preflight)'` sentinel that the architect's existing DISCOVER mode picks up.
2. **Invariant 3 hygiene** — `isCommandAvailable` was a known anti-pattern (no cwd, no stdin, no timeout) that several past regressions traced back to. Phase 0 closes it as a prerequisite for the dispatch layer's binary-availability checks.
3. **Extensibility** — adding a language used to mean editing `src/tools/test-runner.ts:1858` (a 14-way switch on framework detection), `src/agents/coder.ts:125`, `src/agents/test-engineer.ts:67`, and several hooks. Now it's a single profile entry and an optional backend file.

## Migration

No action required for end users. The dispatch path is now the default — `SWARM_LANG_BACKEND=legacy` opts back to the inline switches for one release as a rollback. The architect prompt substitutes its placeholders with detected values; when detection fails (no manifest, slow filesystem, etc.), placeholders resolve to the sentinel `unresolved (run /swarm preflight)` and the architect's existing DISCOVER mode handles them the same way it always has.

For plugin developers extending opencode-swarm to a new language, see [`docs/adding-a-language.md`](../adding-a-language.md).

## Breaking changes

None. Every change is additive to public APIs:

- `getAgentConfigs(config?, directory?, sessionId?, projectContext?)` adds an optional 4th argument; existing 3-arg callers unaffected.
- `createAgents(config?, projectContext?)` adds an optional 2nd argument.
- `LanguageProfile.treeSitter.commentNodes?: string[]` is optional; existing test fixtures and externally-defined profiles type-check.
- `LanguageRegistry.register()` now throws on duplicate ids and extension collisions (was silent overwrite). Three internal tier1/tier2 tests that asserted the old silent-overwrite behavior were updated to assert the new throw.
- `LanguageBackend.buildTestCommand` signature accepts an optional `opts` parameter (`{ scope, coverage }`); existing zero-opt callers unaffected.
- `LanguageBackend.parseTestOutput` returns a `TestRunSummary` with optional `total` and `coveragePercent` fields added.
- `SWARM_LANG_BACKEND=legacy` opts out of the new dispatch path; default is dispatch.
