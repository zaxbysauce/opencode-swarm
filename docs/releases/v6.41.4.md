# Release Notes — v6.41.4

**Patch Release | 2026-03-31**

---

## Overview

This release fixes three bugs in the dark matter knowledge pipeline introduced in v6.41.0,
and adds Claude skill tooling to enforce the qa-sweep protocol and contributing guidelines
automatically.

---

## Bug Fixes

### `scope: 'project'` prevented dark matter entries from reaching the architect

**Root cause**: `darkMatterToKnowledgeEntries` wrote knowledge entries with `scope: 'project'`.
The knowledge reader's `scope_filter` defaults to `['global']` and uses exact string matching,
so all dark matter entries were silently filtered out and never injected into the architect's
context. The entries accumulated in `.swarm/knowledge.jsonl` but were never used.

A unit test asserting `scope === 'global'` had existed since v6.31.0 and was passing before
v6.41.0 introduced the regression.

**Fix**: Restored `scope: 'global'` in `darkMatterToKnowledgeEntries`
(`src/tools/co-change-analyzer.ts`).

### Retroactive repair for existing `scope: 'project'` entries

**Root cause**: Repos that ran a dark matter scan under v6.41.0–v6.41.3 have orphaned
`scope: 'project'` entries in their `.swarm/knowledge.jsonl` that will never be injected
under the default config. Deleting and rescanning would replace the entries, but existing
knowledge is lost.

**Fix**: Added a non-blocking repair pass in `system-enhancer.ts` that runs on every session
start. It scans for auto-generated dark-matter entries with `scope: 'project'` and re-scopes
them to `'global'`. The repair is idempotent and only rewrites the store when stale entries
are found.

### `npmiThreshold: 0.3` override caused dark matter count to always be exactly 10

**Root cause**: The system-enhancer passed `npmiThreshold: 0.3` to `detectDarkMatter`, but
the design specification (`docs/releases/v6.17.0.md`) requires NPMI ≥ 0.5, which is also
`detectDarkMatter`'s own default. At 0.3, even weakly-correlated file pairs qualify, so
virtually any repo with 20+ commits produces 20+ qualifying pairs — always hitting the
10-entry cap in `darkMatterToKnowledgeEntries`. The entry count appeared fixed at exactly 10
regardless of the codebase's actual hidden coupling density.

**Fix**: Removed the `npmiThreshold: 0.3` override so `detectDarkMatter` uses its designed
default of 0.5. Repos with fewer genuine hidden couplings will now produce fewer than 10
entries; the count reflects actual signal rather than noise.

## New: Claude Skills

Added two `.claude/skills/` entries and a `SessionStart` hook:

- **`qa-sweep`** (updated): Description now explicitly includes "debugging errors,
  investigating failures, tracing root causes" so the skill triggers on diagnostic sessions,
  not only implementation sessions.

- **`commit-pr`** (new): Encodes all contributing guidelines — conventional commit format,
  mandatory `docs/releases/v{NEXT_VERSION}.md` on every PR, 5-tier test suite before pushing,
  SHA-pinning for workflow changes, and PR body format. Triggers automatically on commit/push/PR
  operations.

- **`SessionStart` hook**: Added to `.claude/settings.json` so both skills are loaded at
  session start rather than relying solely on intent-matching.

---

## Migration

No API or configuration changes.

**Affected repos** (ran dark matter scan under v6.41.0–v6.41.3): The retroactive repair in
`system-enhancer.ts` will automatically fix `scope: 'project'` entries on the next session
start. No manual action required. A `[system-enhancer] Repaired N dark matter knowledge
entries` warning will appear in the TUI when the repair runs.

None.
