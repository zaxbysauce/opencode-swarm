# v6.84.7 (2026-04-26)

## What changed

- **Redacted OpenCode key variants in shell audit logs** — The `redactShellCommand()` regex patterns were expanded to catch:
  1. Environment variables ending in `_KEY` (e.g., `OPENCODE_KEY=sk-xxx`). Previously only compound keywords like `API_KEY` and `ACCESS_KEY` were caught.
  2. Custom HTTP headers without the `X-` prefix (e.g., `-H "Opencode-Key: sk-xxx"`, `-H "Api-Secret: value"`). The pattern now matches any dash-separated header ending in `key`, `token`, `secret`, `auth`, or `credential`.
- **Warn if `.swarm/` is not gitignored** — On plugin init, checks if `.swarm/` is covered by `.gitignore` or `.git/info/exclude`. Emits a one-time warning if not, preventing accidental commits of shell audit logs containing secrets.
- **Fixed repo-graph first-initialization race** — Added defensive `mkdir` before temp file write in `saveGraph()`.
- **Hardened repo-graph symbol extraction** — Wrapped symbol extraction in try/catch to skip malformed files gracefully instead of aborting the entire graph build.
- **Downgraded repo-graph errors to warnings** — Non-fatal graph errors use `console.warn` instead of `console.error`.
- **Added council gate AND semantics** — `checkCouncilGate()` now requires BOTH `council.enabled` in plugin config AND `council_mode` in the QA gate profile before blocking. Prevents the gate from activating at the plan level when council is globally enabled but not plan-specific.

## Why

**Issue #618** reported that the OpenCode API key was appearing unredacted in `.swarm/session/shell-audit.jsonl`. Two gaps in the redaction logic allowed keys to leak:
1. Environment variable assignments using a bare `_KEY` suffix (e.g., `OPENCODE_KEY=value`) were not being redacted because the keyword list only included compound patterns like `API_KEY`.
2. Custom HTTP headers not following the `X-` prefix convention were not caught by the header redaction pattern.

Additionally, the `.swarm/` directory was being created in user projects but was not automatically added to their `.gitignore`.

## Migration steps

None required. The changes are backward-compatible and do not modify any APIs, config schemas, or command syntax.

## Known caveats

- The gitignore warning fires at most once per process (module-level deduplication).
- The redaction logic is conservative and may redact header values in non-credential headers that happen to end in `key`, `token`, `secret`, etc.
