# v6.45.0

## What changed

### search — structured workspace search tool

Added `src/tools/search.ts`, a workspace-scoped ripgrep-style text search tool that
produces structured JSON output. It supports literal and regex modes, glob filtering,
result limits, and falls back to `rg-not-found` output when ripgrep is unavailable.
Registered for: architect, coder, reviewer, explorer, test_engineer.

Test coverage: 35 unit tests + 99 adversarial tests.

### suggest_patch — reviewer-safe patch suggestion tool

Added `src/tools/suggest-patch.ts`, a context-anchored patch suggestion tool that
generates diff hunks without modifying files. It supports multi-hunk patches, stale
context detection, and partial failure handling when some hunks cannot be generated.
Registered for: reviewer, architect (NOT coder — read-only operation).

Test coverage: 27 unit tests + ~74 adversarial tests.

### batch_symbols — batched symbol extraction tool

Added `src/tools/batch-symbols.ts`, which extracts exported symbols from multiple files
in a single call. Each file is processed independently with per-file error isolation,
so a parse failure in one file does not affect others. Benchmarks show 75–98% call
reduction versus sequential single-file calls. Reuses the existing `symbols.ts` parsing
logic. Registered for: architect, explorer, reviewer.

Test coverage: 27 unit tests.

### Tool-name normalization helper

Added `src/hooks/normalize-tool-name.ts` providing `normalizeToolName()` and
`normalizeToolNameLowerCase()` for stripping namespace prefixes from tool names (e.g.
`mega:search` → `search`). This consolidates 13 inline `replace(/^[^:]+[:.]/, '')` regex patterns
patterns that were duplicated across 6 files: guardrails.ts, scope-guard.ts, index.ts,
delegation-gate.ts, self-review.ts, and delegation-ledger.ts. The regex handles both `:` and `.`
separators (e.g. `mega:search` → `search` or `mega.search` → `search`).

### Test drift detection pipeline step

Added step 5l-ter to the architect EXECUTE workflow. The step fires conditionally when
changes involve: command behaviour, parsing or routing logic, user-visible output,
public contracts, assertion-heavy areas, or helper lifecycle changes. It validates
that tests still align with current behaviour before changes land. 16 new tests were
added to `tests/unit/agents/architect.test.ts`.

All Phase 2–4 tests: 188+ tests passing.

## Why

These additions address three recurring pain points. First, agents lacked a
structured, workspace-scoped search that returns machine-readable results, forcing
workarounds with shell commands. Second, the reviewer agent could not surface
contextual patch suggestions without triggering file writes, limiting its utility in
read-only review passes. Third, symbol extraction required one round-trip per file,
creating unnecessary overhead when analysing large workspaces. The normalization helper
removes a latent source of inconsistency where the same namespace-stripping logic was
implemented slightly differently across six files. The test drift detection step closes
a gap where test behaviour could diverge from implementation without triggering a
mandatory review.

## Migration steps

None — all changes are additive with no impact on existing APIs or runtime behaviour.

## Breaking changes

None.

## Known caveats

The `suggest_patch` tool is intentionally read-only and registered only for reviewer
and architect agents. Coder agents will not receive this tool in their toolkit.
A write-capable `apply_patch` tool is explicitly deferred. `suggest_patch` addresses the reviewer→coder handoff without requiring it — reviewers can propose structured patches without the tool having write capability.
