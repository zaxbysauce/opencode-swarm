# v6.86.14 — Enforce continuation on temporary LLM errors

## What changed

**Issue #691 — Swarm now continues on temporary API errors instead of stopping**

When a temporary LLM API error occurs (429 rate limit, 503 service unavailable, 529 overloaded, timeout, etc.), the swarm will now retry up to 5 times before counting errors toward the circuit breaker.

Previously, if an agent had no `fallback_models` configured, a single transient error would:
1. Immediately exhaust the model fallback mechanism
2. Cause all subsequent errors (including more transient errors) to count toward `consecutiveErrors`
3. Trigger the circuit breaker hard stop after 5 total errors

**Now:** The circuit-breaker bypass is independent of model fallback state. Each invocation window gets a dedicated `transientRetryCount` (0-5) that resets per invocation. Transient errors increment this counter instead of `consecutiveErrors` until the budget is exhausted, allowing the agent to recover during temporary outages.

## Why

Temporary infrastructure failures are not agent logic errors. When an API is rate-limited or briefly overloaded, agents should be allowed to retry, especially in contexts where the swarm can handle eventual success (like automated code review or test execution). The previous behavior was too aggressive — a single temporary error could cascade into a session-blocking stop.

## Technical detail

- **src/state.ts**: `InvocationWindow` now has `transientRetryCount: number` (initialized to 0), auto-resets per invocation.
- **src/config/schema.ts**: `max_transient_retries` added to both `GuardrailsConfigSchema` (global default 5) and `GuardrailsProfileSchema` (per-agent optional override).
- **src/hooks/guardrails.ts**:
  - Split `isTransientPatternMatch` (pure regex check) from `isTransient` (adds `window.transientRetryCount < cfg.max_transient_retries` guard).
  - Added `529` (HTTP overloaded) to `TRANSIENT_MODEL_ERROR_PATTERN`.
  - Model-fallback gate unchanged — continues to use `!session.modelFallbackExhausted` independently.
  - On success: reset both `consecutiveErrors` and `transientRetryCount` to 0.

## Migration steps

No migration required. The feature is transparent:
- Agents without `fallback_models` now get 5 free retries on transient errors (up from 1).
- Agents with `fallback_models` continue to work as before — they get fallback switching + transient retry bypass.
- Per-agent override via `GuardrailsProfile.max_transient_retries` if custom retry budgets are desired.
