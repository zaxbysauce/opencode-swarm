# v6.75.0 — Dark Foundation for Stacked Parallelization (PR 1)

## What changed

### Config & telemetry scaffolding (Phase 1)
- Added `ParallelizationConfigSchema` to plugin config with safe defaults: `enabled: false`, `maxConcurrentTasks: 1`, `evidenceLockTimeoutMs: 60000`
- Added 4 new telemetry event names required by this PR: `evidence_lock_acquired`, `evidence_lock_contended`, `evidence_lock_stale_recovered`, `plan_ledger_cas_retry`
- Added `docs/dev/pr1-foundation.md` design note documenting what is dark, what is deferred, and process-global variable audit

### Evidence write locking (Phase 2)
- New `src/evidence/lock.ts`: `withEvidenceLock<T>` helper using `proper-lockfile` with exponential backoff and a typed `EvidenceLockTimeoutError`
- Wired lock around every read-modify-write path in `src/evidence/manager.ts` (`saveEvidence`) and `src/gate-evidence.ts` (`recordGateEvidence`, `recordAgentDispatch`)
- The legacy flat-retrospective write-back in `loadEvidence` is now also guarded by the lock
- Atomic temp-file-plus-rename semantics preserved; locking is an additive safety layer
- `withEvidenceLock` and `EvidenceLockTimeoutError` exported via `src/parallel/index.ts`

### AgentRunContext foundation (Phase 3)
- New `src/state/agent-run-context.ts`: generic `AgentRunContext` class holding per-run isolated Maps (`activeToolCalls`, `activeAgent`, `delegationChains`, `agentSessions`, `environmentProfiles`) plus a shared reference to process-global `toolAggregates`
- `swarmState` per-run Maps are now backed by `defaultRunContext` (same object references — zero semantic change for all existing callers)
- New `getRunContext(runId?)` API; no-arg returns `defaultRunContext`

### Plan ledger CAS backoff (Phase 4)
- New `retryCasWithBackoff` helper extracted in `src/plan/manager.ts`: exponential backoff (5ms start, 2× per attempt, 250ms cap, ±25% jitter), emits `plan_ledger_cas_retry` telemetry with 8-character hash prefixes only (never full hashes)

### Dark dispatcher skeleton (Phase 5)
- `src/parallel/dispatcher/types.ts`: type-only surfaces (`DispatcherConfig`, `RunSlot`, `DispatchDecision`, `TaskExecutionHandle`)
- `src/parallel/dispatcher/noop-dispatcher.ts`: `createNoopDispatcher` factory — always returns `reject`/`parallelization_disabled`, no side effects
- No production code imports the dispatcher

## Why

This PR lands safe, dependency-free foundations that later parallelization PRs need without activating any parallel execution. All new capabilities are default-disabled and do not alter the behavior of any existing single-run workload.

The evidence locking closes a real concurrent write-loss bug: a baseline race test showed 15 of 16 concurrent `saveEvidence` writes lost to read-modify-write interleaving before the fix.

## Migration steps

No migration required. All new config is optional with defaults that reproduce current behavior exactly. No existing call sites need updating.

## Breaking changes

None.

## Known caveats

- `evidence_lock_stale_recovered` is emitted as a heuristic when `withEvidenceLock` acquires after prior contention (`attempt > 0`). `proper-lockfile` handles stale cleanup internally without surfacing it to callers, so the event fires on normal high-contention scenarios as well. This will be refined in a future PR when `proper-lockfile` exposes stale-recovery signals.
- `parallelization.enabled` is wired to config but no production path reads it yet. PR 2 will activate the enabled path.
- Dispatcher barrel (`src/parallel/dispatcher/index.ts`) is not re-exported by `src/parallel/index.ts` — it is intentionally dark for this PR.
