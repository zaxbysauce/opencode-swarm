# v7.19.0

## What changed

### Part 1: `save_plan` is now non-destructive (issue #853)

`savePlan` previously replaced a plan's `phases` verbatim — any task present
in the prior plan but absent from the new save was silently deleted. After
this release, both the manager-level `savePlan` and the architect-facing
`save_plan` tool reject such saves unless the caller explicitly acknowledges
every missing task id.

- New typed error `PlanTaskRemovalNotAcknowledgedError` thrown by the
  manager when removals are unacknowledged. The save-plan tool returns
  `PLAN_TASK_REMOVAL_NOT_ACKNOWLEDGED` in the standard `SavePlanResult`
  shape.
- New `save_plan` tool args:
  - `removed_task_ids: string[]` — every task id missing from `phases` must
    be enumerated here.
  - `removal_reason: string` — non-empty, recorded on every audit event.
  - `confirm_destructive_reset: boolean` — required when `reset_statuses:
    true` would drop tasks. When set alongside `reset_statuses`, the tool
    auto-populates `removed_task_ids` from the computed missing set.
- New ledger event `task_removed` (schema bumped 1.0.0 → 1.1.0).
  **Functional on replay**: `applyEventToPlan` splices the task out of
  the active phase so a crash between the ledger append and the
  `plan.json` atomic rename does not resurrect the removed task on the
  next `loadPlan`. The event's `plan_hash_after` matches the post-removal
  plan, so determinism is preserved. (An earlier draft of this release
  shipped `task_removed` as audit-only; the post-merge review caught the
  crash-window resurrection bug and the replay semantics were corrected
  before any non-prerelease tag.)
- New helper `savePlanWithAutoAcknowledgedRemovals(dir, plan, source,
  reason)` for legitimate recovery paths that need to replace the task
  set without per-id user confirmation: `loadPlan` migrate-from-md,
  ledger replay rebuild, approved-snapshot recovery, `importCheckpoint`,
  `phase-complete` rebuild. Each call site tags the event with a
  caller-identifying `source`.
- `RuntimePlan` gains `_midLoadRemovals?: { count, source }` so the
  system-enhancer can disclose recovery-time removals to the model
  without re-reading the ledger.

### Part 2: spec drift is now surfaced proactively (issue #853 secondary)

Three independent layers act on `.swarm/spec-staleness.json`:

- **Layer A** (`src/hooks/system-enhancer.ts`): injects a
  `[spec-drift]` advisory into the system prompt after every `loadPlan`
  call inside `experimental.chat.system.transform`. The advisory names
  the resolution commands (`/swarm clarify`, `/swarm
  acknowledge-spec-drift`), the blocked tool set, and any
  `_midLoadRemovals` count.
- **Layer B** (`src/hooks/guardrails.ts`): structural block in
  `toolBefore` for the `SPEC_DRIFT_BLOCKED_TOOLS` set —
  `save_plan`, `update_task_status`, `phase_complete`,
  `lean_turbo_run_phase`, `lean_turbo_acquire_locks`. Returns
  `SPEC_DRIFT_BLOCK` while the staleness file exists. Read-only
  exploration tools are intentionally NOT blocked. No cache — deletion
  is reflected immediately on the next call.
- **Layer C** (`src/services/status-service.ts`): `/swarm status`
  renders a `**Spec drift detected**` line with stored/current hashes.

### Architect guidance

Section 6k of the Architect prompt was extended with explicit guidance:

- Re-saving a plan after spec drift MUST enumerate dropped tasks in
  `removed_task_ids` with a `removal_reason`. Unfinished tasks (status
  pending / in_progress / blocked) MUST NOT be removed without explicit
  user confirmation.
- The runtime structurally blocks `SPEC_DRIFT_BLOCKED_TOOLS` while
  `.swarm/spec-staleness.json` exists. A `SPEC_DRIFT_BLOCK` response is
  not retryable.

## Why

Issue #853 reported two symptoms after the assistant detected spec drift:

1. **Silent task deletion.** The assistant re-derived the plan from the
   updated spec and called `save_plan` with a smaller task set. The
   missing tasks vanished without a ledger event, leaving the user with
   half a plan and no recovery path. Root cause: `executeSavePlan`
   replaced `phases` verbatim; `savePlan` had a `preserveCompletedStatuses`
   guard but no missing-task detection; the ledger had no `task_removed`
   event type.
2. **Drift detection happened late.** `loadPlan` wrote
   `.swarm/spec-staleness.json` and set `_specStale` on the runtime
   plan, but no hook consumed those signals. The architect only
   discovered drift when the user explicitly asked. Root cause: no
   surface (system-prompt / guardrail / status) consumed the staleness
   signal.

## Migration steps

After upgrading, **restart any running OpenCode session**. Older
in-memory `applyEventToPlan` readers throw on unknown event types — once
the new plugin starts writing `task_removed` events, an older reader
will throw during replay until the process is restarted. The on-disk
ledger format is forward-compatible; only the in-memory reader needs to
be refreshed.

No data migration is required. Existing plans continue to load. The
first `save_plan` call after upgrade that would drop tasks will fail
with `PLAN_TASK_REMOVAL_NOT_ACKNOWLEDGED` — the recovery_guidance in
the result tells the caller (or architect) what to pass.
