# v6.86.8 — Plugin portability fix + cache refresh command

## What changed

### Plugin bundle is now portable to Node ESM hosts

The published `dist/index.js` bundle previously contained a bare top-level
`import { Database } from "bun:sqlite"`. ESM hoists static imports to
module-evaluation time, which means any host whose `await import(pluginEntry)`
goes through Node's ESM resolver throws `ERR_UNSUPPORTED_ESM_URL_SCHEME` before
any plugin code runs. OpenCode catches and silently drops the error — leaving
the plugin listed in the sidebar but with zero agents or commands.

The fix is two-part:

1. **Build target changed to `--target node`** for the main bundle. Bun now
   emits `var __require = createRequire(import.meta.url)` instead of
   `var __require = import.meta.require`. The former works under Node; the
   latter is Bun-only and causes `__require is not a function` on Node.

2. **`bun:sqlite` is now resolved lazily** in `src/db/project-db.ts` and
   `src/db/global-db.ts` via `createRequire(import.meta.url)('bun:sqlite')`
   at call time rather than at module-evaluation time. Under Bun the lazy call
   succeeds as before; under Node it only runs when the SQLite layer is
   actually touched, which is fine because OpenCode invokes the plugin under
   Bun's runtime.

The CLI bundle (`dist/cli/index.js`) keeps `--target bun` because it runs
exclusively via `bunx`.

### New `bunx opencode-swarm update` command

OpenCode never refreshes a plugin it has already installed. Once cached, users
silently stay on the same version until they manually clear the cache. This
release adds an `update` subcommand that clears both known cache layouts:

- **XDG cache layout** — `$XDG_CACHE_HOME/opencode/packages/opencode-swarm@latest/`
  (used on some macOS and Windows builds)
- **node_modules layout** — `~/.config/opencode/node_modules/opencode-swarm/`
  (used on Linux and devcontainers)

Running `bunx opencode-swarm update` clears whichever layouts are present (or
both) and reports what it cleared. If neither exists the command exits 0 with a
"nothing to do" message listing the locations it checked. Running it twice is
safe (idempotent). It never touches `opencode.json`, `opencode-swarm.json`,
custom prompts, or any other user config.

### Version staleness check at startup

The plugin now performs a non-blocking, once-per-24h check against npm to see
if a newer version of `opencode-swarm` is available. If one is found, a one-line
warning is printed at startup (or deferred until after the session if
`quiet: true` is set). Results are cached at
`~/.cache/opencode-swarm/version-check.json`. The check is skipped entirely
when `version_check: false` is set in `opencode-swarm.json`.

### Diagnose shows version staleness

`/swarm diagnose` now includes a "Version check" row that shows the currently
running version, the npm-published version (from cache), the cache age, and a
⚠️ flag when the running version is behind.

### Defensive plugin init wrapper

The plugin entry point now wraps initialization in a try/catch that logs
`[opencode-swarm] FATAL: plugin initialization failed` with the full stack trace
to stderr before re-throwing. Previously, any unhandled throw in init was
silently swallowed by OpenCode's plugin loader, making failures invisible.

### New regression tests

- `tests/unit/build/bundle-portability.test.ts` — scans `dist/index.js` and
  `dist/cli/index.js` for top-level `bun:` imports; fails CI if one slips back in.
- `tests/unit/build/bundle-node-load.test.ts` — spawns `node --input-type=module`
  to actually import the bundle; asserts no `ERR_UNSUPPORTED_ESM_URL_SCHEME`.
- `tests/unit/cli/update-command.test.ts` — 7 tests covering both cache layouts,
  simultaneous eviction, idempotence, and config non-mutation.
- `tests/unit/services/version-check.test.ts` — 18 tests for version comparison,
  scheduling throttle, network silence, latch idempotence, and cache I/O.

## Why

Issue #675: users on Linux devcontainers reported the plugin not loading at all
after a recent OpenCode update. The static `bun:sqlite` import was the root
cause. A secondary factor is that OpenCode's cache-pin behavior means users
never automatically receive the fixed version once it's published — hence the
`update` command.

## Migration steps

- **No config changes required.**
- If the plugin currently fails to load: run `bunx opencode-swarm update` (once
  this version is published) and restart OpenCode.
- To suppress version-check warnings: set `"version_check": false` in
  `opencode-swarm.json`.
- To opt into quiet deferral of version warnings: set `"quiet": true` in
  `opencode-swarm.json`. The warning will be shown once, at session end.

## Breaking changes

None. The `--target node` switch is internal to the build; the exported API and
plugin registration are unchanged.

## Known caveats

- The version check requires outbound HTTPS to the npm registry. It is
  best-effort: network failures are silently ignored and the cached result
  (if any) is used.
- `bunx opencode-swarm update` clears the cache but does not re-install. OpenCode
  performs the fresh install automatically on next start.
