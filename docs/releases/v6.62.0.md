# v6.62.0

## What changed

**Critic drift verifier can now detect silent plan mutations** -- the `get_approved_plan` tool exposes the immutable critic-approved snapshot store as an actively queryable tool for the `phase_drift_verifier` agent.

- New `get_approved_plan` tool (`src/tools/get-approved-plan.ts`) wraps `loadLastApprovedPlan` from the ledger as a tool callable by the critic agent
- Tool derives `plan_id` internally from current `plan.json` for cross-identity safety -- the LLM does not need to know the plan identity format
- Returns both the approved baseline plan and the current plan with hash-based drift detection (`drift_detected: true/false/"unknown"`)
- Supports `summary_only` mode that returns structural metadata only (phase counts, task counts) to reduce token budget for large plans
- Handles edge cases: no approved snapshot, missing/corrupt plan.json, cross-identity filtering
- Wired into `AGENT_TOOL_MAP` for `critic_drift_verifier` with a `TOOL_DESCRIPTIONS` entry
- `PHASE_DRIFT_VERIFIER_PROMPT` updated with mandatory `BASELINE COMPARISON` section before per-task rubric and `BASELINE DRIFT` output format
- Registered in tool-names, barrel export, and plugin tool map

### Repo graph: hardening and hook wiring (issue #470)

This release adds a complete repo dependency graph system with automatic refresh, security hardening, and full test coverage.

**Graph core (Phase 1)**
- Added `repo-graph.ts` — workspace dependency graph with node/edge model, safe load/save with atomic writes, symlink boundary validation, and cross-platform path normalization
- Added `repo-graph-builder.ts` — startup hook that builds the graph and write-triggered hook for incremental updates
- Exported graph utilities via `src/tools/index.ts` barrel

**Automatic refresh (Phase 2)**
- Startup `init()` hook builds or refreshes the graph when a session starts
- `toolAfter` hook incrementally updates the graph when write tools modify `.ts`, `.tsx`, `.js`, `.jsx`, `.mjs`, `.cjs`, or `.py` files
- Large-repo truncation with configurable `maxFiles` and `maxFileSizeBytes` options

**Hardening and validation (Phase 3)**
- Symlink boundary validation prevents workspace-escaping attacks via `realpathSync`
- Path normalization (`normalizeGraphPath`) ensures consistent graph keys across Windows/macOS/Linux
- `repo_graph` removed from tool registrations (no tool handler exists — graph is maintained via hooks)
- Import parsing expanded to cover re-export patterns (`export { x } from './y'`, `export * from './y'`) and dynamic `import()` expressions
- Tree-sitter evaluation documented (deferred — no implementation)

**Review council fixes (Phase 4)**
- Fixed `scanFile` call in `updateGraphForFiles` to pass raw file path instead of normalized key
- Added workspace boundary validation in `toolAfter` hook to reject paths outside workspace root
- Wired `repoGraphHook.toolAfter` into `src/index.ts` handler chain via `safeHook()`

**Test suite stabilization (Phase 5)**
- Eliminated `mock.module` global leakage by adding dependency injection (`RepoGraphDeps` interface) to `createRepoGraphBuilderHook`
- Fixed Windows test failures caused by fake Unix paths (`/fake/workspace` → real temp directories)
- Added `clearCache` calls for test isolation across all graph test files
- 148 tests pass across 7 test files (0 failures, 1 intentional Windows skip)
- 54 adversarial tests covering path traversal, encoding bypass, oversized inputs, prototype pollution

## Why

**Critic drift verifier:** The critic-approved snapshot store was implemented for passive crash recovery only (`loadPlan` Step 4b). The `phase_drift_verifier` had no way to compare the immutable approved baseline against the current plan to detect unauthorized plan mutations introduced between phases. This meant scope creep or silent plan changes after critic approval went undetected. Exposing the snapshot store as a tool maximizes the value of immutable snapshots for active drift checking, not just recovery. Closes #449.

**Repo graph:** The repo graph provides Claude Code with real-time awareness of workspace file dependencies, enabling smarter context injection and navigation. The implementation was hardened based on a review council audit (issue #474) covering security, architecture, and test quality.

## Additional fixes

- **syntax-check WASM stability**: Reduced `MAX_FILE_SIZE` from 5MB to 2MB to prevent WASM tree-sitter `abort()` crashes that corrupted all subsequent parse operations in the same process. Added `isMissing` node detection and `rootNode.hasError` fallback for more robust error reporting. This fixes a pre-existing test failure in `syntax-check.test.ts`.

## Known caveats

- The `get_approved_plan` tool is wired only into `critic_drift_verifier` -- other critic roles (`plan_critic`, `sounding_board`, `critic_oversight`) do not have it in their whitelist since their prompts are not updated to use it
- Pre-existing issue: `req_coverage` may have a similar registration gap in `src/index.ts` plugin tool block (out of scope for this PR)
- `repo-graph.ts` is a large module (~1459 lines) that could benefit from splitting into submodules in a future refactor
- Concurrent sessions updating the same workspace may experience lost graph updates (no inter-process locking)
- URL-encoded paths in tool arguments are not decoded before workspace boundary check (coincidentally safe, not intentionally validated)
- See issue #474 for the full list of post-hardening improvements identified by the review council
