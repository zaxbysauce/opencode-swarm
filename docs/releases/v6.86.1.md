## Overview

This release fixes a structural durability gap in the plan system: `.swarm/plan.md` was silently going stale during long swarm runs, leaving task checkboxes as `[ ]` even after agents had finished their work. Two related rendering bugs and a silent failure mode are also fixed in this patch.

---

## Bug Fixes

### `plan.md` task status not updating during swarm runs (primary fix)

The plan system uses a two-layer model:
- **Layer 1 (in-memory):** `session.taskWorkflowStates` â€” the per-session state machine used by gate checks (`idle â†’ coder_delegated â†’ pre_check_passed â†’ reviewer_run â†’ tests_run â†’ complete`)
- **Layer 2 (durable):** `plan.json` `task.status` â€” the persistent record projected to `plan.md`

Layer 1 advanced automatically when delegation milestones fired (including the council APPROVE â†’ `complete` transition). Layer 2 only changed when the architect agent explicitly called the `update_task_status` tool. Nothing bridged the two layers, so when the architect skipped the call â€” common in long runs â€” `plan.md` silently showed stale `[ ]` checkboxes for completed tasks.

**Fix:** A new async helper `advanceTaskStateAndPersist(session, taskId, newState, directory)` in `src/state.ts` wraps `advanceTaskState` and persists at the two meaningful workflow boundaries:
- `coder_delegated` â†’ marks task `in_progress` in `plan.json`
- `complete` â†’ marks task `completed` in `plan.json` and regenerates `plan.md` with `[x]`

The helper is wired into `src/hooks/delegation-gate.ts` at the council APPROVE fast-path and coder delegation call sites. Persistence errors are logged and swallowed so transient disk failures do not break the in-memory state machine.

### Phase header rendering incorrect when using `completed` status alias

`derivePlanMarkdown`'s `statusMap` only contained `complete` as a key, but `PhaseStatusSchema` accepts both `complete` and `completed` as valid phase statuses (the latter as an alias). Phases stored with `completed` rendered as `[PENDING]` in the markdown header. `closed` was also unmapped.

**Fix:** Added `completed: 'COMPLETE'` and `closed: 'CLOSED'` aliases to `statusMap` in `derivePlanMarkdown`.

### Silent `plan.md` write failures inside `savePlan`

`savePlan` intentionally treats `plan.md` writes as non-fatal (plan.json is authoritative), but the failure was only logged at warn level. If a write failed silently, the markdown projection went stale with no observable signal.

**Fix:** A `plan_md_write_failed` telemetry event is now emitted on plan.md write failure, giving operators and diagnostics tooling (e.g. `/swarm status`) an observable signal for silent staleness.

## Migration Notes

No configuration or API changes. No migration steps required.

If you use telemetry consumers that subscribe to swarm events, the new `plan_md_write_failed` event will appear in your telemetry stream when plan.md writes fail. It carries `directory`, `error`, and `timestamp` fields.
