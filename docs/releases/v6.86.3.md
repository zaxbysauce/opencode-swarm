# v6.86.3

## What changed

- **Fixed ledger corruption handling (plan audit F-01):** Wired the existing `replayWithIntegrity` function into the production `replayFromLedger` path so that ledger corruption detection and quarantine actually execute at runtime. Previously the quarantine was dead code — only tests exercised it. Also added a `console.warn` when `readLedgerEvents` skips malformed JSON lines (previously silent discard).

- **Fixed `/swarm rollback` ledger bypass (plan audit F-02):** The rollback command now excludes `plan-ledger.jsonl` from checkpoint file copies, deletes any existing ledger unconditionally after restore, and initializes a fresh ledger via `initLedger` + appends a `plan_rebuilt` ledger event. Copy failures now produce detailed error messages instead of continuing silently.

- **Fixed stale tests (plan audit F-10):** Updated two `replayFromLedger` tests that expected null when plan.json is deleted. Since the #444 self-sufficient ledger feature embeds plan data in the `plan_created` event, `replayFromLedger` can reconstruct plans from the ledger alone — the tests now simulate legacy ledger format to correctly test the "no plan data available" path.

## Why

These changes resolve the three HIGH-severity findings from the comprehensive planning-system audit (`planning-system-audit.md`). The audit identified 14 source-backed defects; these three required attention before shipping.

## Migration steps

None. All changes are backward-compatible internal improvements.

## Known caveats

The `rollback` fix assumes the checkpoint directory contains a valid `plan.json`. If a checkpoint lacks `plan.json`, files are restored but the ledger is not re-initialized (user is directed to `/swarm reset-session`).
