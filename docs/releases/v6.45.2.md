# v6.45.2

## What changed

### Per-agent file write authority fixes

- Prefixed agents now inherit their canonical authority rules when no exact override exists.
- Mixed-case authority config keys are normalized before lookup.
- Architect direct writes are now checked in the runtime `toolBefore` hook.
- An explicit empty `allowedPrefix: []` now denies all writes instead of bypassing the allowlist.
- README authority documentation corrected to accurately describe merge behavior for known vs custom agents, and the three-state custom agent authority model.

### Documentation accuracy fixes

- README Merge Behavior section now correctly distinguishes between known agents (omitted fields fall back to hardcoded defaults via `??`) and custom agents (omitted fields are `undefined` — no defaults to inherit).
- `allowedPrefix: []` vs omitting `allowedPrefix` distinction is now explicitly documented: empty array denies all writes; omitting means no allowlist restriction applies.
- README Custom Agents section now describes three distinct states: not in config (denied as Unknown agent), in config without `allowedPrefix` (no allowlist restriction — only explicit blocklist rules apply), and in config with `allowedPrefix: []` (all writes denied).

### Test coverage

- Self-contradictory comment in authority test suite corrected to match the actual `??` fallback behavior it was describing.
- 19 new edge case tests added covering: fast path correctness for empty/disabled/undefined config, behavioral equivalence between hook path and direct call, map isolation across hook instances, `enabled` flag interaction, and custom agent authority via pre-computed path.

### Performance

- Effective authority rules are now pre-computed once at hook initialization (`createGuardrailsHooks`) instead of being rebuilt on every `toolBefore` invocation. The common case of zero user rules now takes a fast path with no allocation. `authorityConfig` is immutable after plugin init so there is no behavioral difference.

## Why

These fixes close correctness gaps in the per-agent file write authority system so prefixed swarm agents, runtime architect writes, and config overrides all behave consistently. The documentation corrections ensure operators configure custom agent authority with an accurate mental model of what each field combination actually does.

## Known caveats

Custom agents that are **not present in `authority.rules` at all** are denied with an `Unknown agent` error — they are not in the effective rules map and therefore blocked by default. Custom agents that **are** present in `authority.rules` but have no `allowedPrefix` set are subject only to whatever `blockedPrefix`, `blockedZones`, or `readOnly` rules you explicitly configure — no allowlist restriction is applied by default. To restrict a custom agent to specific paths, always set `allowedPrefix` explicitly.
