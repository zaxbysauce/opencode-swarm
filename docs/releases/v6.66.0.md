# v6.66.0

## What changed

### Work Complete Council — parallel five-member verification gate

A new architect-only tool `convene_council` that synthesizes verdicts from
critic, reviewer, sme, test_engineer, and explorer into a single veto-aware
decision before a task is allowed to advance to `complete`. Any single
`REJECT` blocks advancement (veto, not voting). The architect receives one
unified feedback markdown document — the coder never gets contradictory
instructions from different members. Explorer specializes in anti-slop
detection (hallucinated APIs, cargo-cult patterns, lazy abstractions, spec
drift).

The feature is **off by default** (`council.enabled: false`). Existing
configurations are unaffected.

#### New surface (all under new paths — zero impact on existing flows)
- `src/council/types.ts` — flat, stable data contracts (verdicts, findings,
  synthesis, criteria, config + defaults)
- `src/council/council-service.ts` — pure synchronous synthesis: veto
  detection, conflict reconciliation (add-vs-remove heuristic), required-fix
  vs advisory classification, mandatory-criteria assessment, and the unified
  feedback markdown builder
- `src/council/criteria-store.ts` — writer/reader for pre-declared acceptance
  criteria at `.swarm/council/{taskId}.json`; malformed JSON returns `null`
  without crashing
- `src/council/council-evidence-writer.ts` — stamps the council outcome
  into `.swarm/evidence/{taskId}.json` under `gates.council` (the shape
  `check_gate_status` and `update-task-status` actually consume — they
  read `evidence.gates[gateName]`, not a top-level `evidence.council`).
  Standard GateInfo fields (`sessionId`, `timestamp`, `agent`) are written
  alongside council-specific extras (`verdict`, `vetoedBy`, `roundNumber`,
  `allCriteriaMet`). Pre-existing `gates[*]` entries and top-level keys
  are preserved across the write. Defense-in-depth measures: the taskId
  regex rejects malformed IDs before any FS op, and spread-merge filters
  the `__proto__` / `constructor` / `prototype` keys defensively (prototype
  pollution hardening)
- `src/tools/convene-council.ts` — the architect-only tool, following the
  `check-gate-status.ts` pattern: `createSwarmTool` factory, zod-validated
  args, `resolveWorkingDirectory`, config gate runs before any filesystem
  side effect, structured JSON on bad inputs (never throws)

#### Wiring (minimal, surgical)
- `src/tools/tool-names.ts` — `convene_council` added to `ToolName` union
  and `TOOL_NAMES`
- `src/tools/index.ts` — barrel export (alphabetical)
- `src/config/constants.ts` — added to `AGENT_TOOL_MAP.architect` only (no
  other agent); `TOOL_DESCRIPTIONS` entry
- `src/config/schema.ts` — `CouncilConfigSchema` (strict; `enabled: false`
  default, `maxRounds: 3`, `parallelTimeoutMs: 30_000`, `vetoPriority: true`)
  plus optional `council` key on `PluginConfigSchema`

#### Tests (39 new, 0 regressions)
- `tests/unit/council/council-service.test.ts` — 13 tests: veto logic
  (unanimous APPROVE, single REJECT, test_engineer REJECT parity, CONCERNS
  without REJECT, vetoPriority=false downgrade), feedback markdown (APPROVE
  message, REJECT veto + required fixes, max-rounds escalation), conflict
  detection, criteria assessment
- `tests/unit/council/criteria-store.test.ts` — 5 tests: round-trip,
  unknown taskId, dot encoding, malformed JSON, path-traversal safety
- `tests/unit/council/convene-council.test.ts` — 9 tests: registration
  (`TOOL_NAMES`, `TOOL_NAME_SET`, `AGENT_TOOL_MAP.architect` only, no other
  agent, barrel export, valid tool shape, args surface), config gate
  (disabled message default, invalid-args structured error), happy path
  (writes evidence at `1.1.json` and returns APPROVE)
- `tests/unit/council/adversarial.test.ts` — 12 tests: poisoned verdicts

## Why

Cross-cutting completeness verification was previously the architect's
implicit responsibility, with the individual reviewer/test_engineer gates
running in series inside the main context. This encouraged single-context
self-approval on high-risk work. The council gives the architect a
parallel verification path with four independent contexts, each restricted
to its role's domain, synthesized into one decision with veto semantics.
Quality is the goal; speed is preserved by running the four members in
parallel.

## Migration steps

None. The feature is opt-in via config:

```json
{
  "council": {
    "enabled": true
  }
}
```

Defaults when `council.enabled: true`:
- `maxRounds: 3` (architect must escalate to user when the council cannot
  clear a task in three rounds)
- `parallelTimeoutMs: 30000`
- `vetoPriority: true` (any single `REJECT` blocks; set to `false` for
  downgrade-to-`CONCERNS` mode)

The tool is architect-only and enforced by the existing `AGENT_TOOL_MAP`
permission check.

## Breaking changes

None.

## Known caveats

- **`synthesizeCouncilVerdicts` returns `APPROVE` when given an empty
  verdicts array.** This cannot happen through the tool (zod schema
  enforces `.min(1)`) but direct callers of the service should be aware.
- **Conflict detection is a best-effort heuristic** (add/include/insert
  vs remove/delete/eliminate directives at the same location). Surfaced
  to the architect as advisory — it is not a proof.
- **Advisory delivery channel is architect self-echo, not
  architect→coder.** `pushCouncilAdvisory` and the guardrails
  `messagesTransform` hook inject advisories into the ARCHITECT session's
  system prompt only; non-architect sessions drain silently. The
  architect is therefore expected to render `unifiedFeedbackMd` into the
  coder's delegation payload explicitly (per the prompt instructions).
  A dedicated architect→coder advisory primitive is a potential future
  improvement.
- **Pre-declared criteria are prompt-driven, not enforced at plan save
  time.** `save_plan` does not call `declare_council_criteria`; if the
  architect skips Phase 0, the council falls through to `criteria=null`
  and `allCriteriaMet` defaults to `true`. Criteria declaration is an
  instruction in the architect prompt, not a runtime invariant.
- **Full-auto mode × maxRounds-REJECT** produces a "surface to user and
  HALT" instruction in the unified feedback. No user exists in full-auto
  mode, and `critic_oversight` is not wired into the council escalation
  path. `escalateOnMaxRounds` is declared for forward compatibility with
  no behavior yet.
- **Retry protocol's "verify prior findings resolved"** is prompt-level
  guidance only — there is no diff-gating, finding-status tracking, or
  prior-synthesis injection into member context. Council members see
  the full candidate each round; they are instructed (but not required)
  to focus on unresolved findings.

## Phase 6 additions (PR #477, continuation)

Phase 6 closes the loop between the council tool contract shipped above and
the architect workflow. All additions remain opt-in via `council.enabled`;
users who have not set the flag see zero behavioral change.

### Explorer council member (fifth role)

- `CouncilAgent` union extended with `'explorer'` — the council now has five
  members: `critic`, `reviewer`, `sme`, `test_engineer`, `explorer`.
- Explorer specializes in anti-slop detection. New `CouncilFindingCategory`
  values surface its domain: `slop_pattern`, `hallucinated_api`,
  `lazy_abstraction`, `cargo_cult`, `spec_drift`.
- Dispatched in parallel alongside the other four members during synthesis.

### `declare_council_criteria` tool

- New architect-only, config-gated tool (`src/tools/declare-council-criteria.ts`).
- Persists acceptance criteria at plan time to `.swarm/council/{safeId}.json`
  so reviewers assess a stable, pre-committed contract rather than criteria
  invented at review time.
- Criterion shape: `id` matches `^C\d+$`, `description` 10–500 chars,
  `mandatory: boolean`. Between 1 and 20 criteria per declaration.
- Idempotent overwrite — repeated declarations replace the previous file
  and the response includes `replaced: true` for observability.

### Architect prompt workflow (four phases)

The architect prompt is wired (Phase C) to drive the council through four
explicit phases:
1. **Pre-declare** — at plan time, call `declare_council_criteria`.
2. **Parallel dispatch** — after the coder signals complete, dispatch all
   five members simultaneously with isolated role-specific context.
3. **Synthesize** — call `convene_council(taskId, swarmId, verdicts[], roundNumber)`.
4. **Act** — interpret `overallVerdict`: `APPROVE` advances the task;
   `CONCERNS` sends `unifiedFeedbackMd` as a single non-blocking advisory
   and advances; `REJECT` blocks and retries up to `maxRounds`; reaching
   `maxRounds` without APPROVE escalates to the user.

The workflow is gated by `council.enabled`. When disabled, the architect
falls through to the previous reviewer-gate path unchanged.

### Advisory routing — `pushCouncilAdvisory`

- New helper (`src/council/council-advisory.ts`) routes
  `synthesis.unifiedFeedbackMd` into the architect session's
  `pendingAdvisoryMessages` queue so the feedback is surfaced as an
  `[ADVISORIES]` block on the next `messagesTransform` pass.
- Dedup key: `council:${taskId}:${roundNumber}`. Repeated calls within the
  same round are no-ops.
- Blocking semantics are carried in the header: `blocking=true` on REJECT,
  `blocking=false` on CONCERNS. APPROVE with no advisory findings is a
  no-op — the architect already received the verdict as the tool return.

### `update_task_status` council gate enforcement

- `checkCouncilGate(workingDir, taskId)` is invoked in the `'completed'`
  branch after the existing reviewer-gate.
- Blocks advancement when `council.enabled === true` AND
  (`gates.council` is absent OR `gates.council.verdict === 'REJECT'`).
- `APPROVE` and `CONCERNS` allow advancement. Config-load failures degrade
  to "not blocked" to avoid regressing users who have not opted in.

### New `CouncilConfig` fields

- `requireAllMembers: boolean` — default `false`. When `true`,
  `convene_council` returns a structured error if fewer than five member
  verdicts are supplied.
- `escalateOnMaxRounds?: string` — optional. Declared for forward
  compatibility (webhook URL or handler name). No runtime behavior is
  wired yet; reserved for a follow-up.
