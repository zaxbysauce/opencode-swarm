# v6.47.2 Release Notes

## What Changed

### [HIGH] Curator new-entry loop now enforces the knowledge validation gate

The `applyCuratorKnowledgeUpdates` function handles `promote new` recommendations where `entry_id` is undefined by creating new `SwarmKnowledgeEntry` records. Previously this path wrote entries directly without consulting the three-layer validation gate, making it the only knowledge ingestion path that did not call `validateLesson()`. Every other path (`knowledge-add`, `knowledge-migrator`, `hive-promoter`) calls `validateLesson()` and respects the `validation_enabled` configuration.

The fix loads `existingLessons` from the store before the loop as an array of lesson strings. Inside the loop, an explicit exact-match deduplication check runs before validation: if the lesson is already in `existingLessons`, it is skipped. When `validation_enabled !== false`, `validateLesson()` is called and the entry is skipped if validation returns false. Each successfully written lesson is pushed into `existingLessons` so that later entries in the same batch are caught by the deduplication check.

### [MEDIUM] `curator_analyze` now rejects invalid `entry_id` values instead of silently coercing

Previously, any non-UUID string supplied as `entry_id` was silently coerced to `undefined`, causing stale or typo'd IDs to create new entries unintentionally rather than surfacing an error to the caller. This made it impossible for direct tool callers to detect a bad entry reference.

The tool now validates `entry_id` in the early-validation block before config load and the LLM delegate call. Undefined is allowed and means create new. A valid UUID v4 is allowed and means update existing. Any other non-empty string returns a JSON error response: `{ "error": "Invalid entry_id '...': must be a UUID v4 or omitted." }`.

The LLM text-parse path via `parseKnowledgeRecommendations` intentionally retains silent normalisation because hallucinated slugs are an expected occurrence when parsing free-form LLM output, and this is the standard path used by the agent loop.

## Why

The `promote new` path in `applyCuratorKnowledgeUpdates` was the only knowledge ingestion path that did not call `validateLesson()`, creating an inconsistency that allowed the curator to persist lessons that every other path would reject. This was a follow-up finding from the validation gate work in v6.45.1 and PR #381.

The entry_id coercion issue meant direct tool callers had no way to detect a bad entry reference — it silently became an unintended new entry. The pre-flight validation block now fail-fast on invalid input before any expensive work is done.

## Migration Steps

None for most users. Auto-generated curator lessons that previously bypassed validation will now be silently skipped if they fail the gate. The only observable change for standard users is that harmful or low-quality lessons are no longer persisted by the curator.

## Breaking Changes

Direct callers of the `curator_analyze` tool that previously supplied non-UUID strings as `entry_id` values will now receive an error response instead of having the value silently coerced to a new-entry creation. Callers that use LLM-sourced recommendations via `parseKnowledgeRecommendations` are unaffected — that path continues to silently normalise hallucinated slugs as it always has.

## Tests Added

`tests/unit/hooks/curator.test.ts` gained three new tests covering the validation gate: SC-001 verifies that a dangerous lesson is skipped when `validation_enabled` is true, SC-002 verifies that the same lesson is allowed when `validation_enabled` is false, and SC-003 verifies that identical lessons within the same call are deduplicated.

`tests/unit/tools/curator-analyze.test.ts` is a new test file with eight tests covering the entry_id contract: rejects non-UUID entry_id, accepts undefined entry_id (new entry), accepts valid UUID v4 (update existing), and adversarial cases including empty string, slug-style strings, early short-circuit on first bad ID in a batch, all-undefined batch, and mixed batch with both valid and invalid entries.

### Bundled v6.47.1 plan/ledger changes (dist only)

The `dist/` artifact in this release also contains plan and ledger logic that was
authored in v6.47.1 (PR #388) but had not previously been compiled into `dist/`.
These are not v6.47.2 changes — the source code was already present in `main` before
this PR. They are included here because the prior `dist/` was stale.

The four affected functions are: `replayFromLedger`, which now filters events to a
single plan identity using `targetPlanId = events[0].plan_id` so that cross-identity
events from a renamed plan are ignored during replay; `savePlan`, which archives the
old ledger file via `renameSync` to a timestamped path before reinitialising when a
plan identity mismatch is detected; `loadPlan`, which checks the ledger's first event
`plan_id` against the current plan identity before triggering a ledger rebuild,
skipping the rebuild when the ledger belongs to a different identity; and
`applyEventToPlan`, which validates the incoming task status through
`TaskStatusSchema.safeParse` before applying it, returning the current plan unchanged
on validation failure instead of corrupting state with an undefined status.
