# Skill Maturity Unblock + Outcome Accrual Fix (#1477)

## What changed

- **High-priority directive maturity path**: `critical`/`high` directive entries confirmed in at least one phase and at or above a 0.6 confidence floor (`HIGH_PRIORITY_SKILL_MIN_CONFIDENCE`) now satisfy the skill-maturity gate. This decouples the confidence floor from the 2-distinct-phase confirmation requirement for the highest-value knowledge, which previously stranded those entries permanently. Ordinary (`medium`/`low`) entries are unaffected, and any entry with a net-negative outcome signal still blocks.
- **Outcome accrual fixed (root cause)**: `effectiveRetrievalOutcomes` merged stored entry counts with the event-derived rollup using a plain spread (`{ ...base, ...rollup }`), so the always-zero production rollup silently overwrote real `succeeded_after_shown_count` / `failed_after_shown_count` values written to the entry. Outcomes were written but never read, starving the maturity gate, search ranking, the curator, and learning metrics of all outcome signal. The merge is now **additive** for the outcome-count fields, preserving historical entry counts without a data migration and without double-counting.
- **Outcome events wired**: `updateRetrievalOutcome` now records each shownâ†’phase outcome as an immutable `'outcome'` `KnowledgeEvent` (the single source of truth folded into the rollup) instead of mutating entry counters in place. This supplies the previously-missing producer for the existing, test-covered outcome fold consumed by learning metrics.
- **Corrupt-baseline resilience**: `readCounterBaseline` now fails open scoped to the baseline â€” a corrupted `knowledge-counter-baseline.json` no longer propagates to the outer rollup catch (which would discard the live event log too). The folded historical baseline is dropped while live events are still replayed, so outcome accrual and skill maturation survive a corrupt baseline file instead of every entry's outcome signal silently dropping to zero.
- **Stale-skill self-healing (Part C)**: Closed a detection blind spot where an active `SKILL.md` with no parseable frontmatter was silently skipped (now surfaced as `unparseable_frontmatter`); added a deterministic `reconcileStaleActiveSkills` pass (regenerate-or-retire) wired into the skill_improver cadence; hand-authored compatibility shims marked `skill_origin: shim` are now skipped. The committed `guardrail-patterns` skill was converted to proper generated frontmatter, and `pr-review-fix` was marked as a shim.

## Why

Issue #1477 reported the knowledgeâ†’skill pipeline stalled at "0 of 102 entries eligible" with all outcome counters reading zero. Root-cause tracing showed three compounding defects: the maturity gate offered no priority path for high-value directives (RC-1/RC-4), a lossy read-merge discarded every written outcome (RC-OUTCOMES, the primary cause), and stale active skills could never be detected or repaired. These changes restore legitimate maturation for high-value knowledge, make outcomes actually accrue and reach every consumer, and add a durable repair path for stale skills â€” without loosening quality gates for ordinary entries.
