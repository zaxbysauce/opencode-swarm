# v6.44.0

## What changed

### Durable plan ledger — append-only source of truth

- Added `.swarm/plan-ledger.jsonl`: an append-only JSONL ledger that is the durable runtime record for plan status mutations. `plan.json` and `plan.md` are now projections derived from the ledger, not the source of truth.
- All four plan mutation paths (savePlan, updateTaskStatus, phase_complete, executeSavePlan) now write ledger events first before updating projections.
- `loadPlan()` is now ledger-aware: detects hash mismatch between `plan.json` and the latest ledger event, and auto-rebuilds the projection from the ledger before returning.
- Snapshot events are appended to the ledger every 50 events and on `phase_complete`. Each snapshot embeds the full Plan payload and `payload_hash`. `replayFromLedger()` uses the latest snapshot as base state for fast recovery. Corrupted ledger suffixes are quarantined to `.swarm/plan-ledger.quarantine` and replay continues from the last valid event.

### Checkpoint artifact — SWARM_PLAN.md and SWARM_PLAN.json

- `writeCheckpoint()` writes `SWARM_PLAN.md` and `SWARM_PLAN.json` at the project root on every `save_plan`, `phase_complete`, and `/swarm close`. These are export-only artifacts — not live runtime sources of truth.
- `importCheckpoint()` reads `SWARM_PLAN.json` and reseeds the live plan + ledger from it. Useful for starting fresh in a new repo with an existing checkpoint.
- Both files added to `.gitignore`.

### Capability metadata — single source of truth

- Architect prompt `YOUR TOOLS` and `Available Tools` sections are now auto-generated at prompt-construction time from `AGENT_TOOL_MAP.architect` in `constants.ts`. No more hand-maintained lists that drift.
- Added `TOOL_DESCRIPTIONS` map in `constants.ts` for human-readable tool descriptions.
- Added `WRITE_TOOL_NAMES` shared constant used by both `scope-guard.ts` and `guardrails.ts`. Fixes a bug where `guardrails.ts` was missing `append` and `prepend` from its write-tool detection.
- Added 25 capability drift guard and hygiene tests that fail if tool lists drift from `AGENT_TOOL_MAP`.

### Verification contract hardening

- Fixed prompt ambiguity: architect prompt now unambiguously states the critic returns a verdict and the architect calls `write_drift_evidence` — the critic does not write files.
- Added first-pass phase completion e2e test proving `phase_complete` succeeds on a single invocation without a second call (closes the historical must-call-twice bug).
- Added no-deadlock SAST integration test: coder is not blocked by pre-existing HIGH findings on unchanged files.

### Documentation

- `docs/plan-durability.md`: migration notes for the ledger-based plan durability model, file roles, rebuild/import/export mechanics, snapshot system, corruption handling.
- `docs/troubleshooting/recovery-guide.md`: operator recovery runbook with 6 scenarios (missing plan.json, stale session, ledger mismatch, when to use reset-session, rebuild vs import, ledger corruption).

## Why

- **Crash recovery**: Before this change, a session crash or interrupted write could leave `plan.json` in an inconsistent state with no recovery path. The ledger provides a durable audit trail that survives crashes.
- **Drift prevention**: Hand-maintained tool lists in the architect prompt drifted from `AGENT_TOOL_MAP` silently. Auto-generation eliminates the drift surface.
- **Security fix**: `guardrails.ts` was missing `append` and `prepend` from its write-tool detection, meaning those tools bypassed the architect self-coding gate.

## Migration

No action required for existing projects. The ledger is created automatically on the first `save_plan` call after upgrading. Existing `plan.json` files are read as-is; the ledger is initialized from them.

If you want to export a checkpoint of your current plan state, call `writeCheckpoint(directory)` or trigger any `save_plan` / `phase_complete` operation — the checkpoint files will be written automatically.

## Breaking changes

None.

## Known caveats

- Rebuild after file loss: if `plan.json` is missing but a snapshot event exists in the ledger, `replayFromLedger()` will use the latest snapshot as base state. Without a snapshot event, `plan.json` is still required as the base state. If both `plan.json` and the ledger are lost, restore from `SWARM_PLAN.json`.
- The `plan_hash_after` field in ledger events now stores the post-mutation hash (the hash after the mutation is applied), not the pre-mutation hash. This corrects an off-by-one error where the stored hash referred to the state before the mutation.
- The `importCheckpoint()` function is programmatic only — there is no `/swarm import` CLI command in this release.
