# OpenCode Swarm v6.17.3 Release Notes

## Summary

Version 6.17.3 is a **patch release** delivering two diagnostic signal fidelity fixes:

- **BUG-1** ‚Äî Two startup messages previously emitted at `warn()` level (which bleeds into the OpenCode TUI stdout) have been reclassified to `log()` (debug-gated, only visible when `OPENCODE_SWARM_DEBUG=1` is set).
- **BUG-2** ‚Äî `loadEvidence()` previously returned `null` for both "file not found" and "file found but invalid schema", making the two failure modes indistinguishable. It now returns a discriminated union result. `phase_complete` now emits actionable error messages that name the specific failing Zod fields and include a minimum valid retrospective template when a file is missing.

---

## Detailed Fixes

### 1. BUG-1: Log Level Reclassification (`warn()` ‚Üí `log()`)

**Problem**: Two informational startup diagnostics were tagged at `warn()` level, causing them to appear in the OpenCode TUI stdout during normal operation (GitHub Issue #40). `warn()` is reserved for degraded or uncertain states ‚Äî a successfully resolved model limit is the opposite.

The two affected messages:
- `[model-limits] Resolved limit for <model>@<provider>: <N> (source: ...)` ‚Äî emitted by `src/hooks/model-limits.ts`
- `[swarm] Context budget: model=... provider=... limit=...` ‚Äî emitted by `src/hooks/context-budget.ts`

**Fix**: Both messages reclassified from `warn()` to `log()`. The `log()` function is debug-gated ‚Äî messages only appear when `OPENCODE_SWARM_DEBUG=1` is set.

**For Issue #40 reporters**: No configuration change is needed. The fix takes effect automatically on the next `npm update`. Both messages will disappear from TUI stdout after upgrading to v6.17.3.

---

### 2. BUG-2: `loadEvidence` Discriminated Union Return Type

**Problem**: `loadEvidence()` returned `EvidenceBundle | null` ‚Äî silently returning `null` for both "file not found" and "file found but failed Zod schema validation". When an agent wrote a retrospective manually and omitted required fields (e.g., `phase_number`, `total_tool_calls`, `task_count`), `loadEvidence` would catch the Zod error and return `null`. `phase_complete` would then emit the generic "no valid retrospective evidence found" error, making a schema-invalid file indistinguishable from a missing file.

**Fix A ‚Äî Discriminated return type**: `loadEvidence()` now returns a `LoadEvidenceResult` discriminated union:
```typescript
export type LoadEvidenceResult =
  | { status: 'found'; bundle: EvidenceBundle }
  | { status: 'not_found' }
  | { status: 'invalid_schema'; errors: string[] };
```

Status semantics:
- `not_found` ‚Äî file does not exist on disk
- `invalid_schema` ‚Äî file exists but JSON.parse threw OR Zod schema validation failed; `errors` array contains field-path messages (e.g., `"entries.0.phase_number: Required"`)
- `found` ‚Äî file exists and passed full Zod validation; `bundle` contains the parsed `EvidenceBundle`

**Fix B ‚Äî `phase_complete` actionable error messages**: When `loadEvidence` returns `invalid_schema`, `phase_complete` now emits an error naming each failing Zod field:

```
Phase N retrospective found at .swarm/evidence/retro-N/evidence.json but failed schema validation.
Missing or invalid fields: entries.0.phase_number: Required, entries.0.total_tool_calls: Required, ...
```

**Fix C ‚Äî Minimum valid retrospective template**: When `loadEvidence` returns `not_found` after exhausting all fallback scan positions, `phase_complete` now includes a minimum valid template in the error output so agents can construct a valid retro without consulting external documentation.

**For agents that hit the `phase_complete` catch-22**: If you previously received "no valid retrospective evidence found" even though the file existed ‚Äî the retro file was valid JSON but failed `EvidenceBundle` schema validation. After upgrading to v6.17.3, the error message will name the missing fields, allowing you to fix the retro file and retry.

## Upgrade Notes

- **Backward compatible**: All callers of `loadEvidence()` have been updated to handle the new discriminated union. No existing successful workflows are affected.
- **No migration required**: Existing valid retrospective evidence files continue to work unchanged.
- **`LoadEvidenceResult` type exported**: Available from `src/evidence/index.ts` for consumers that need to handle all three cases explicitly.
