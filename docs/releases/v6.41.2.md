# Release Notes — v6.41.2

**Patch Release | 2026-03-30**

---

## Overview

This release fixes a path authority enforcement bug where absolute file paths caused all coder write operations to be blocked, even when the target file was within allowed directories. The issue was most visible on Windows (where IDEs and tools routinely emit absolute paths with drive letters) but affected all platforms.

- **checkFileAuthority**: Absolute paths now correctly normalized to project-relative before prefix matching
- **declare_scope**: Absolute paths normalized at declaration time with warnings; out-of-project paths rejected
- **isInDeclaredScope**: Now resolves paths against the project directory instead of `process.cwd()`

Reported in #259 by @asakura42, root cause analysis by @LordVandal1976.

---

## Bug Fixes

### Path authority enforcement breaks on absolute paths (`src/hooks/guardrails.ts`)

`checkFileAuthority` used string prefix matching (`startsWith('src/')`) against relative prefixes, but the file path parameter could be an absolute path (e.g., `C:/Users/dev/project/src/file.ts` or `/home/dev/project/src/file.ts`). Since absolute paths never start with `src/`, every coder write was blocked with `"Path not in allowed list for coder"`.

**Fix**: The function now resolves the incoming path to absolute via `path.resolve(cwd, filePath)`, then converts back to project-relative via `path.relative(cwd, resolved)` before prefix comparison. This is fully backward-compatible — relative paths pass through unchanged.

### declare_scope silently accepts absolute paths (`src/tools/declare-scope.ts`)

`declare_scope` accepted and stored absolute paths without error or warning. The scope declaration appeared successful, but downstream `checkFileAuthority` calls would block every write — with no indication that path format was the problem.

**Fix**: Absolute paths are now normalized to project-relative at declaration time. A `warnings` array in the response notifies the caller of each normalization. Absolute paths that resolve outside the project directory are rejected with an explicit error message.

### isInDeclaredScope resolves against process.cwd() instead of project directory (`src/hooks/guardrails.ts`)

The post-execution scope validation function `isInDeclaredScope` used `path.resolve(filePath)` without a base directory, resolving against `process.cwd()` instead of the project's working directory. If `process.cwd()` differed from the project directory, scope containment checks could produce incorrect results.

**Fix**: `isInDeclaredScope` now accepts an optional `cwd` parameter, and the call site passes the project `directory` from the `createGuardrailsHooks` closure.

## Breaking Changes

None. The `DeclareScopeResult` interface gained an optional `warnings?: string[]` field, which is additive only.
