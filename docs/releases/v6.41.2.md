# Release Notes — v6.41.2

**Patch Release | 2026-03-30**

---

## Overview

This release fixes two bugs that caused Kimi K2.5 (and potentially other LLMs) to enter infinite
loops or get permanently blocked when using the swarm tools:

- **`save_plan` infinite loop**: Kimi K2.5 kept generating negative phase IDs because the JSON
  Schema for the `phase.id` field used `{"exclusiveMinimum": 0}` (Zod v4 Draft 2020-12 output
  for `.positive()`), which some models misinterpret. The recovery guidance was also too vague
  to allow the model to self-correct.

- **`completion_verify` blocks Phase 1 research tasks**: Research and inventory tasks (codebase
  audit, CI review, dependency analysis, etc.) produce knowledge artifacts rather than source
  files. The tool's fail-closed logic blocked any completed task with no `files_touched` and no
  file paths in the description, making Phase 1 completion impossible without turbo mode.

Additionally, three consistency gaps identified by adversarial review are addressed: the same
`.positive()` JSON Schema issue in `write_drift_evidence` and `write_retro`, and a missing
`.int().min(1)` constraint on the `phase_complete` tool's phase argument.

---

## Bug Fixes

### `save_plan` loop with negative phase IDs (`src/tools/save-plan.ts`)

**Root cause**: `tool.schema.number().int().positive()` generates JSON Schema
`{"type": "integer", "exclusiveMinimum": 0}` (JSON Schema Draft 2020-12). Kimi K2.5
misinterprets this constraint and produces negative integers. With the previous vague
`recovery_guidance` ("Use save_plan with corrected inputs"), the model could not determine
what value to use and looped indefinitely.

**Fix 1**: Changed `.positive()` to `.min(1)` on the `phases[].id` schema field. This generates
`{"type": "integer", "minimum": 1}` — unambiguous across all JSON Schema drafts and LLMs.

**Fix 2**: Expanded `recovery_guidance` from a generic instruction to an explicit example:
> "Phase IDs must be positive integers: 1, 2, 3 (not 0, -1, or decimals). Task IDs must use
> N.M format: '1.1', '2.3', '3.1'. Call save_plan again with corrected ids."

Concrete examples break the loop regardless of how the model interprets the schema constraint.

### `completion_verify` blocks research/inventory tasks (`src/tools/completion-verify.ts`)

**Root cause**: The tool used fail-closed logic for completed tasks with no file targets:
if a task had `status: 'completed'` but no `files_touched` and no file paths parseable from
its description, it was added to `blockedTasks` (preventing phase completion). This is correct
for implementation tasks but wrong for Phase 1 research tasks. Also, `tasksSkipped` was
declared `const` so it could never be incremented even if skip logic had existed.

**Fix 1**: Changed `const tasksSkipped = 0` to `let tasksSkipped = 0`.

**Fix 2**: Changed the "no file targets" branch from `BLOCKED` to `SKIPPED`. Absence of file
targets is not evidence of incompleteness — research tasks are unverifiable by this tool, but
that is not the same as being incomplete. The authoritative completion gate for any task is the
`update_task_status` reviewer enforcement (requires reviewer + test_engineer delegation).

### Schema consistency: `.positive()` → `.min(1)` across all phase-number args

The same `exclusiveMinimum: 0` issue existed in two other tools:

- `write_drift_evidence` (`src/tools/write-drift-evidence.ts`): `phase` arg changed from
  `.positive()` to `.min(1)`, description updated to add "(e.g., 1, 2, 3)"
- `write_retro` (`src/tools/write-retro.ts`): same change

### `phase_complete` schema hardening (`src/tools/phase-complete.ts`)

The `phase` argument used plain `.number()` with no `.int()` or minimum constraint, accepting
floats and negative values at the JSON Schema level (runtime validation still caught them, but
the schema advertised a looser contract than was enforced). Changed to `.number().int().min(1)`
for consistency with all other phase-number schemas.

## Migration

No API changes. All fixes are backward-compatible: valid inputs continue to work identically.

The behavior change in `completion_verify` (BLOCKED → SKIPPED for no-file-target tasks) is
intentional. If you have tests asserting `tasksBlocked > 0` for research tasks, update them to
assert `tasksSkipped > 0` and `tasksBlocked === 0`.
