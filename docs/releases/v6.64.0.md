# v6.64.0

## What changed

### Repo Map / Code Graph (#460, #461)

A new structural-awareness layer for swarm agents. The graph is persisted at `.swarm/repo-graph.json` and queried via the new `repo_map` tool plus auto-injection into the coder and reviewer system prompts.

#### Phase 1 — graph infrastructure
- `src/graph/import-extractor.ts` — regex-based import edge extraction for TypeScript, JavaScript, Python, Go, and Rust. Resolves relative specifiers (with `.ts`/`.js`/`.tsx`/`.jsx`/`/index.*` fallbacks); external packages are recorded as edges with empty `target` so they can be filtered out cheaply.
- `src/graph/symbol-extractor.ts` — exported-symbol extraction (functions, classes, interfaces, types, variables, enums) for the same five languages.
- `src/graph/graph-builder.ts` — concurrent file walker with `p-limit` and a `maxFiles` cap; skips `node_modules`, `dist`, `.git`, `.swarm`, build artefacts.
- `src/graph/graph-store.ts` — atomic save (`tmp + renameSync`, with tmpfile cleanup on failure), load with schema-version check, and `updateGraphIncremental` for in-place per-file refresh. `isGraphFresh` exposes a 5-minute freshness window callers can use as a staleness signal.
- `src/graph/graph-query.ts` — `getImporters`, `getDependencies`, `getSymbolConsumers` (handles namespace wildcards), `getBlastRadius` (BFS over reverse edges with configurable depth + risk classification), `getKeyFiles`, `getLocalizationContext`. A reverse-edge index is cached keyed on graph identity and invalidated explicitly on incremental update.
- `src/tools/repo-map.ts` — `createSwarmTool` wrapper exposing six actions (`build | importers | dependencies | blast_radius | localization | key_files`). All errors return a structured `{ success: false, action, error }` JSON envelope; the `build` branch is wrapped in `try/catch` so a failed `mkdirSync` or `renameSync` no longer escapes as an uncaught throw.
- Path-security validation rejects control characters, path traversal, POSIX absolute paths, and Windows drive-letter absolute paths before the graph is consulted.
- Aliased TS/JS and Python imports (`import { add as sum }`, `from x import add as sum`) record the *original* exported name (`add`), not the local binding (`sum`), so symbol-consumer queries like `getSymbolConsumers(graph, file, "add")` correctly find aliased callers.
- Import-like text inside string literals or template literals (codegen templates, embedded examples, prose) no longer creates phantom graph edges. The TS/JS extractor tracks string-literal ranges and skips matches whose `import`/`require` keyword falls inside one — only the source-string of a real `from '…'` / `require('…')` / `import('…')` remains visible to the regex.
- The "is this string an import source?" lookback now requires a literal word-bounded `from` / `require(` / `import(` immediately before the quote. The previous over-permissive `(`-only lookback would treat any `(`-preceded string (e.g. `console.log("./foo")`) as an import source.
- Import line numbers are now computed from the `import` keyword offset rather than the regex match start, which previously included a consumed leading `\n` and reported every non-first import as `line - 1`.
- `normalizeGraphPath` now strips all leading `./` segments (`././util.ts` → `util.ts`) instead of only one.
- `getBlastRadius` reports `depthReached` correctly for `maxDepth=1` — direct importers are visited at depth 1 and the counter now reflects that even when no further BFS expansion happens.
- `buildRepoGraph` now applies a `DEFAULT_MAX_FILES = 10_000` cap when no explicit `maxFiles` is supplied, bounding worst-case memory on extremely large monorepos.
- `updateGraphIncremental` validates each `changedRelativePaths` entry against the workspace root and silently skips absolute paths, NUL-containing strings, and paths that resolve outside the workspace — defending against graph-injection from malicious incremental update callers.
- Index-file resolution now uses `path.join` so `./lib` resolves to `lib\index.ts` on Windows (previously `lib/index.ts`, which would not match the host's path separator).
- `saveGraph` now refuses to write through a symlinked `.swarm/` directory (`fs.lstatSync` check before `mkdirSync`), preventing a TOCTOU symlink redirect of the graph file. The tmp filename is now `.tmp.${crypto.randomUUID()}` instead of `.tmp.${pid}.${Date.now()}`, eliminating predictable-tmpfile attacks.

#### Phase 2 — agent integration
- `src/hooks/repo-graph-injection.ts` — `buildCoderLocalizationBlock` and `buildReviewerBlastRadiusBlock` produce compact (~300–500 char) context blocks from the cached graph. The cache key is `(mtimeMs, size)` per workspace directory so an mtime collision within the same millisecond no longer serves a stale graph.
- `src/hooks/system-enhancer.ts` — the coder agent now receives a localization block for its declared scope (importers, dependencies, blast radius). The reviewer agent receives a blast-radius block for the coder's last declared scope so it can spot-check unseen consumers. Both injections are silent no-ops when no graph exists.
- Tool registration: `repo_map` is exposed to the architect, explorer, coder, reviewer, and critic agents via `AGENT_TOOL_MAP`.

## Why

The swarm's most expensive failure mode was edits made without structural awareness — the coder didn't know which functions depended on the code it was editing, leading to broken callers, reviewer churn, and architect-coder loops. This change gives every reasoning agent first-class access to "who imports X?", "what does X depend on?", and "what's the blast radius of changing X?" without forcing them to grep.

## Migration steps

No migration required. The graph is opt-in: agents must call `repo_map action="build"` once per workspace to populate `.swarm/repo-graph.json`. Until they do, the system-enhancer injection helpers return `null` and the experience is unchanged.

## Breaking changes

None.

## Known caveats

- The graph is not auto-built on session start. Agents must invoke `repo_map action="build"` (typically the architect during planning); the staleness signal is surfaced via `stale: true` on every query response so consumers can decide when to rebuild.
- Import resolution is regex-based, not Tree-sitter-based. This was a deliberate performance trade-off (a 999-file / 5089-edge build completes in ~2 s); rare edge cases involving dynamic imports or unusual quoting may be missed.
- External (npm/pip/cargo) packages are recorded as edges with empty `target` and are filtered out of all query results by design.
- Bun's full-test runner can OOM on this repo when running every tier sequentially in one process — this is a pre-existing Bun runtime issue unrelated to this PR. Per-file loops (the pattern documented in `commit-pr`) work fine.
