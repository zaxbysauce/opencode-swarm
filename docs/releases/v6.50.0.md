# v6.50.0 — Knowledge System + Plan Sync Overhaul

## What changed

### Knowledge injector: three-regime budget system

The binary 75k character threshold that silently skipped knowledge injection at ~47% context usage has been replaced with a three-regime headroom calculation based on the 128k token model limit (~387k chars):

- **High regime** (>60% headroom): full `inject_char_budget` (default 2,000 chars)
- **Moderate regime** (20-60% headroom): 50% of budget
- **Low regime** (<20% headroom): 25% of budget
- **Skip**: only when headroom drops below 300 chars

This means knowledge injection continues working at 181k chars (previously silently skipped) and stays active through ~97% of the context window.

Additional injector changes:
- Compact display format: `[S]/[H] lesson text` with confirmation indicators replacing verbose star ratings and tier labels
- Priority-ordered assembly: Lessons > Run Memory > Drift > Rejected Warnings > Curator Briefing
- Recency position injection: knowledge block placed before the last user message instead of after the system message
- New config fields: `inject_char_budget` (default 2,000) and `max_lesson_display_chars` (default 120)

### Curator: rewrite action and real entry UUIDs

- New `rewrite` action allows the curator to replace verbose or stale lesson text in-place (15-280 char validation, confidence reduced by 0.05)
- Curator LLM now receives real knowledge entry UUIDs (up to 30 entries, sorted by `updated_at`) instead of always using the `"new"` token
- Updated curator prompts in both `runCuratorPhase` and `runCuratorInit` with rewrite documentation and `KNOWLEDGE_ENTRIES` input format

### Plan sync: merge-mode save and startup-only rebuild

- **Merge-mode `save_plan`**: when a plan already exists, task statuses from matching IDs are preserved across plan revisions (prevents status wipes on re-saves)
- **Startup-only ledger rebuild**: the hash-mismatch rebuild in `loadPlan()` is now gated to the first call per process lifetime, preventing destructive overwrites during active sessions from concurrent `save_plan` + `update_task_status` writes
- **Ledger snapshot**: a snapshot event is written to the ledger after every `save_plan` call
- **Durable evidence seed**: when a task transitions to `in_progress`, a minimal evidence file is written to `.swarm/evidence/{task_id}.json` to persist gate tracking across session restarts
- **PlanSyncWorker improvements**: temp file events (`.tmp.` / `.rebuild`) are filtered out, plans with zero phases skip markdown regeneration, debounce increased from 300ms to 500ms

### Architect: auto-generated slash commands

The hardcoded slash command list in the architect prompt is replaced with a `{{SLASH_COMMANDS}}` template token generated from `COMMAND_REGISTRY` at runtime, eliminating drift when commands are added or removed.

## Why

Three interconnected failure modes were reported:
1. Knowledge injector's budget check silently skipping injection in long sessions (binary threshold too aggressive)
2. Curator unable to act on knowledge it cannot see (no entry UUIDs passed)
3. Plan sync pipeline corrupting task state during `save_plan` calls and ledger rebuilds (no merge mode, no startup gate)

## Migration steps

No breaking changes. The new config fields (`inject_char_budget`, `max_lesson_display_chars`) have sensible defaults and are optional. Existing `.swarm/` directories and knowledge stores are fully compatible.

## Known caveats

- Cross-file test execution with `bun:test` and `vitest` mock systems can cause mock contamination (pre-existing infrastructure issue). All test files pass when run individually.
- The PlanSyncWorker has 10 pre-existing timing-dependent test failures (reduced from 14 on the previous release).
