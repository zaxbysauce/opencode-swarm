# v7.4.2 — Compaction State Recovery & Task Marker Parity

## What changed

**Two critical bug fixes for post-compaction agent session continuity:**

### Bug Fix 1: Task Marker Parity in extractIncompleteTasksFromPlan

- **`src/hooks/extractors.ts`** — `extractIncompleteTasksFromPlan()` now emits the `← CURRENT` marker for tasks with `status: "in_progress"`, matching the behavior of `extractCurrentTaskFromPlan()`. Previously, after compaction the architect could not distinguish the active task from pending backlog tasks, causing session recovery to lose workflow context.

### Bug Fix 2: Rehydration Cache Initialization After Compaction

- **`src/hooks/compaction-customizer.ts`** — The `experimental.session.compacting` handler now calls `buildRehydrationCache(directory)` at the end of the compaction flow. This ensures new agent sessions created after a compaction start with workflow state derived from the current `plan.json`, not the stale startup snapshot. Previously, post-compaction sessions would rehydrate from outdated state, causing drift between the plan and the agent's working context.

## Why

Compaction is designed to collapse long-running swarm sessions into a compact state while preserving workflow continuity. Without these fixes:

1. **Lost task context**: After compaction, the architect could not identify which task was actively in progress. The `← CURRENT` marker is the contract between the plan ledger and the architect agent — its absence breaks the session recovery protocol.

2. **Stale rehydration cache**: New sessions starting after compaction would load workflow state from the pre-compaction snapshot instead of the updated `plan.json`. This caused agents to work against outdated task lists, acceptance criteria, and phase boundaries, creating silent state drift that compounded over time.

Both issues undermined the core contract of compaction: that a compacted session can resume with full context fidelity.

## Migration steps

No migration required. These are bug fixes that restore expected behavior:

- Existing compacted sessions will automatically benefit from the corrected task marker logic on next load
- The rehydration cache will now build correctly for all new sessions post-compaction

## Known caveats

- None. These fixes are backward-compatible and require no configuration changes.
