# OpenCode Swarm v6.33.4 Release Notes

## Summary

Version **6.33.4** fixes session corruption that caused "Session not found" errors and 5-10 second hangs after upgrading to v6.33.3. The root cause was stale timestamps on rehydrated sessions triggering immediate eviction and guardrail circuit breakers on startup.

---

## Bug Fixes

### 1. Rehydrated sessions immediately evicted on startup

Sessions restored from `.swarm/session/state.json` kept their original `lastToolCallTime` from disk. If the plugin process was restarted after sessions had been idle for 2+ hours, `startAgentSession`'s stale eviction sweep deleted every rehydrated session on the first hook call. OpenCode's GUI still referenced those session IDs from the snapshot, producing "Session not found: ses_XXXXX" errors.

**Fix:** `rehydrateState()` now refreshes `lastToolCallTime` and `lastAgentEventTime` to `Date.now()` for every deserialized session before inserting it into `swarmState.agentSessions`.

### 2. Guardrail circuit breakers fired immediately after restart

`InvocationWindow.startedAtMs` and `lastSuccessTimeMs` were also preserved from disk with stale values. After a process restart, the guardrails duration-limit check (`guardrails.ts:778`) and idle-timeout check (`guardrails.ts:832`) computed artificially large elapsed times, immediately triggering circuit breakers and blocking valid agent work.

**Fix:** `rehydrateState()` now refreshes `startedAtMs` and `lastSuccessTimeMs` on every `InvocationWindow` within each deserialized session.

---

## Symptoms This Fixes

- TUI showing "Session not found: ses_XXXXX" error
- Sessions pausing/hanging within 5-10 seconds of starting, requiring terminal restart
- On first launch after upgrade, crash sound + red dot notifications on sessions
- Second launch partially recovering (architects reappear but sessions are fragile)

---

## Migration Notes

- No config changes required
- Existing `.swarm/session/state.json` snapshots (v1 and v2) are fully compatible
- Sessions from previous versions will have their timestamps refreshed on first load
