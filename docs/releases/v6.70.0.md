# v6.70.0

## What changed

### Transparent write authority + lstat symlink guard + universal DENY prefixes + declared-scope override

Extends the write-authority enforcement introduced in PR #378 in four directions:

#### 1. Transparent authority for all agents

Previously, direct Write/Edit tool calls from non-architect sessions were **not**
authority-checked (only delegated writes and architect writes were checked).
A coder session with no `delegationActive` could call `write` with any file path and
bypass the `allowedPrefix`/`blockedPrefix` rules entirely.

Now every Write/Edit call from **any** session is authority-checked using the same
per-agent rule set, before the write executes.

- `handlePlanAndScopeProtection` remains architect-only (no change to that gate).
- `apply_patch` and `patch` tools follow the same extended path.

#### 2. lstat symlink guard on Write/Edit

Writing through a symlink can redirect the write outside the declared working directory,
bypassing scope containment. Example: an agent creates `src/` as a symlink to `/etc/`,
then writes `src/config.ts` — which lands in `/etc/config.ts`.

Every Write/Edit call now lstat-walks the target path from the target up to the working
directory. If any node in the chain is a symlink or junction, the write is blocked before
it executes:

```
WRITE BLOCKED: "src" is a symlink — writing through a symlink could redirect the write
outside the working directory
```

The same guard applies to `apply_patch`/`patch` targets and to `declare_scope` file
declarations (early rejection at scope-declaration time, before coder execution).

#### 3. Universal deny prefixes

`AuthorityConfigSchema` gains a new field:

```jsonc
{
  "authority": {
    "universal_deny_prefixes": [".env", "secrets/", ".git/config"]
  }
}
```

Paths matching any of these prefixes are denied to **all agents** regardless of their
per-agent rules. Checked before per-agent rules; cannot be overridden by `allowedExact`
or `allowedPrefix`. Matching is **case-insensitive** (`.ENV` is blocked by `.env`).

#### 4. Declared-scope override for `allowedPrefix` (issue #496)

Fixes an architectural gap where the `declare_scope` tool set
`session.declaredCoderScope` but that state was **never consulted** by the write
authority check. A coder on a Rails/Python/Go project was blocked from files
the architect had explicitly scoped — because coder's hardcoded
`allowedPrefix` is `['src/', 'tests/', 'docs/', 'scripts/']` (JS/TS-centric).

With PR #501 making the authority check **transparent** for all agents, that
same JS/TS-centric whitelist would now block direct `Write`/`Edit` calls too —
worsening the issue for Rails/Python/Go users. This release closes the gap by
wiring the declared scope into the authority check itself.

`checkFileAuthorityWithRules` gains an optional `options.declaredScope`
parameter. When a path falls inside the declared scope (exact match or
directory containment), the `allowedPrefix` whitelist (Step 7) is **bypassed**
for that path. The guard now resolves the scope from two sources, preferring
the session state and falling back to the per-task map:

```
session.declaredCoderScope
  ?? pendingCoderScopeByTaskId.get(session.currentTaskId)
  ?? null
```

The fallback handles the `/swarm close` → new-task flow where the fresh
session hasn't inherited `declaredCoderScope` yet but the pending map still
has the architect's declaration.

**Security: DENY rules remain enforced.** Declared scope only relaxes Step 7
(`allowedPrefix`). All of the following still block the write even if the
target path is in declared scope:

- `universal_deny_prefixes` (checked upstream in `toolBefore`)
- `readOnly` (Step 1)
- `blockedExact` (Step 2)
- `blockedGlobs` (Step 3)
- `blockedPrefix` (Step 6)
- `blockedZones` (Step 8)

So a declared scope cannot grant writes into `.env`, `.git/`, `secrets/`,
`.swarm/plan.json`, or any path in a blocked zone — the architect's scope
declaration is a positive authorisation, not a security bypass.

Example — Rails project with a coder delegated to fix a config file:

```
architect → declare_scope(['config/environments/test.rb'])
coder    → Write('config/environments/test.rb', ...)   ✅ allowed
coder    → Write('config/database.yml', ...)           ❌ blocked (config zone)
coder    → Write('.env', ...)                          ❌ blocked (universal deny)
coder    → Write('.swarm/plan.json', ...)              ❌ blocked (blockedPrefix)
coder    → Write('app/models/user.rb', ...)            ❌ blocked (outside scope)
```

#### Gap-closure over e95bbce

A post-implementation adversarial review of the #496 fix (commit e95bbce)
found three further gaps that have now been closed:

1. **Delegated-write call sites** — `checkFileAuthorityWithRules` is called
   from four sites in `guardrails.ts`. The initial fix only passed
   `options.declaredScope` at the two transparent-authority sites. The two
   delegated-write sites inside `handleDelegatedWriteTracking` (Write/Edit and
   `apply_patch`) still used pre-fix behaviour, so the primary
   architect→coder workflow (`session.delegationActive === true`) continued
   to WRITE BLOCK Rails/Python/Go paths declared via `declare_scope` — the
   exact #496 scenario. All four call sites now share a `resolveDeclaredScope(sessionID)`
   helper that reads from `session.declaredCoderScope` and falls back to
   `pendingCoderScopeByTaskId` identically.

2. **`pendingCoderScopeByTaskId` cleanup on reset** — the fallback map is
   module-scoped in `src/hooks/delegation-gate.ts` (not inside `swarmState`),
   and `resetSwarmState` did not clear it. After `/swarm close`, a new
   session with a colliding taskId (e.g. `"1.1"`) could inherit stale scope
   from the previous swarm via the fallback added in e95bbce. `resetSwarmState`
   now invokes a new `clearPendingCoderScope()` helper exported from
   `delegation-gate.ts` (helper indirection avoids a circular import).

3. **Windows case-insensitivity for scope matching** — `isInDeclaredScope`
   did byte-exact string comparison on resolved paths. On Windows (a
   case-insensitive filesystem) a scope of `config/` would not match a write
   to `Config/foo.rb` even though the OS treats them as the same path.
   The comparison now lowercases both sides when `process.platform === 'win32'`
   and stays case-sensitive on POSIX.

### Security hardening (adversarial review findings)

Two HIGH-severity findings from the post-implementation adversarial review were fixed:

- **Unknown agent fail-closed**: When `swarmState.activeAgent.get(sessionID)` returns
  `undefined`, the code previously fell back to `'architect'` (broad permissions).
  It now throws `WRITE BLOCKED: No active agent registered for session "..."` instead.
  Agents must call `startAgentSession` before issuing write tool calls.

- **Case-insensitive universal deny**: Universal deny prefix matching now uses
  `.toLowerCase()` on both sides, preventing `.ENV` from bypassing an `.env` prefix
  rule on case-insensitive filesystems (Windows/macOS).

One known TOCTOU window remains (architectural): the lstat check runs in `toolBefore`
while the actual write runs later in the write tool. A concurrent process could swap
a real file for a symlink between the two. This is inherent to the pre-hook architecture
and documented as a known caveat below.

## Why

The K2.6 incident (covered in v6.69.0-a, PR #499) showed that symlink/junction
manipulation can defeat scope containment for destructive commands. The same class of
attack applies to writes: if an agent can write through a symlink, it can exfiltrate
or corrupt files outside the declared working directory, even when the declared scope
looks correct.

The "transparent authority" gap was a silent regression in the authority model — coder
and reviewer agents could write any file via direct Write/Edit calls, making the
`allowedPrefix` rules ineffective unless agents were operating in delegation mode.

Issue #496 revealed a second, complementary gap: the `declare_scope` tool was only
consulted by post-hoc scope-violation logging, not by the write guard itself. That
was a no-op on JS/TS projects (where coder's hardcoded `allowedPrefix` already covers
`src/` and `tests/`) but silently broke Rails, Python, Go, and every other project
using non-JS conventions. Making the authority check transparent without wiring
`declare_scope` into it would have shipped that breakage to all users. The
declared-scope override closes both gaps together.

## Migration steps

- **No breaking changes** for standard usage. The new authority check applies only
  when `authority.enabled` (default: `true`). Set `authority.enabled = false` to
  restore the pre-PR behavior (not recommended).
- If any non-architect session issues Write/Edit calls without `startAgentSession`
  being called first, those writes will now fail. Call `startAgentSession(sessionID, agentType)`
  before delegating to any agent.
- `universal_deny_prefixes` defaults to `[]` — no existing paths are blocked unless
  explicitly configured.
- **Rails / Python / Go projects:** the architect must call `declare_scope` with
  the list of files/directories the coder will touch before delegating. Without
  a declared scope, coder's hardcoded `allowedPrefix` (`src/`, `tests/`, `docs/`,
  `scripts/`) still applies and paths outside that whitelist will be blocked.
  This is how the existing architect/coder workflow is designed to operate.

## Breaking changes

None.

## Known caveats

- **TOCTOU lstat window**: The lstat check runs in the `toolBefore` hook, which fires
  before the write tool executes. A concurrent process could create a symlink at the
  target path in the interval between lstat and write. This is an inherent limitation
  of the pre-hook architecture. Defense-in-depth: OS-level file permissions and the
  scope containment checks in `checkFileAuthorityWithRules` provide additional layers.
- **declare_scope lstat is early but not final**: The declare_scope lstat check rejects
  symlink-backed paths at scope-declaration time, giving the architect early feedback.
  The Write/Edit lstat check provides the final enforcement at write time.
- **Tool name filter**: Only tools in `WRITE_TOOL_NAMES` are subject to the new checks.
  Custom tools with non-standard names registered outside the plugin are not covered.
