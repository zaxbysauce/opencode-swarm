# Release v6.73.0

## Summary

Implements runtime enforcement for the `hallucination_guard` QA gate flag, completing issue #491. The gate existed with full type/tooling/persistence infrastructure but had zero runtime enforcement — nothing ever read `effectiveGates.hallucination_guard` to block execution. This release adds a dedicated `critic_hallucination_verifier` sub-agent and a mandatory `phase_complete` gate that blocks phase completion if hallucination verification is enabled but evidence is missing or rejected.

## What changed

### Core enforcement
- **`src/tools/phase-complete.ts` Gate 3** — New conditional gate that reads `getEffectiveGates().hallucination_guard`; blocks phase completion with `HALLUCINATION_VERIFICATION_MISSING` if evidence file is absent, or `HALLUCINATION_VERIFICATION_REJECTED` if verdict is rejected; turbo mode bypasses the gate at parity with Gate 2 (drift verifier)
- **`src/tools/write-hallucination-evidence.ts`** (new) — Dedicated tool for writing hallucination verification evidence; atomic write (temp file + rename); normalizes verdict values; **intentionally omits** `takeSnapshotEvent` and `lockProfile` calls (profile locking is reserved for drift verifier approval)

### Agent architecture
- **`src/agents/critic.ts`** — Extended `CriticRole` union with `'hallucination_verifier'`; added `HALLUCINATION_VERIFIER_PROMPT` constant with PRESSURE IMMUNITY block, IDENTITY section, 4-axis rubric (API Existence: VERIFIED|FABRICATED, Signature Accuracy: ACCURATE|DRIFTED, Doc/Spec Claims: SUPPORTED|UNSUPPORTED, Citation Integrity: RESOLVED|BROKEN), per-artifact verification format, PHASE VERDICT (APPROVED|NEEDS_REVISION), and read-only enforcement RULES
- **`src/agents/index.ts`** — Registered `critic_hallucination_verifier` agent using the `createCriticAgent` factory with `'hallucination_verifier'` role

### Configuration and tooling
- **`src/config/constants.ts`** — Added `'critic_hallucination_verifier'` to `ALL_SUBAGENT_NAMES`; added `AGENT_TOOL_MAP.critic_hallucination_verifier` whitelist (11 tools: complexity_hotspots, detect_domains, imports, retrieve_summary, symbols, batch_symbols, search, pkg_audit, knowledge_recall, req_coverage, repo_map); added `'write_hallucination_evidence'` to architect's tool whitelist; extended `TOOL_DESCRIPTIONS` and `DEFAULT_MODELS`
- **`src/config/agent-categories.ts`** — Categorized `critic_hallucination_verifier` as a `'qa'` agent
- **`src/tools/tool-names.ts` + `src/tools/index.ts`** — Registered `'write_hallucination_evidence'` as a tool

### Architect prompt
- **`src/agents/architect.ts`** — Added mandatory PHASE-WRAP step 5.55 (conditional hallucination verification): architect delegates to `critic_hallucination_verifier` when `hallucination_guard` is enabled in the effective QA profile; if verifier returns NEEDS_REVISION, architect stops and does NOT proceed to `phase_complete`; step is skipped if QA profile is already locked; extended step 5.6 to require `.swarm/evidence/{phase}/hallucination-guard.json` with APPROVED verdict when gate is enabled; updated `buildQaGateSelectionDialogue()` from advisory language to mandatory enforcement language

### Documentation
- **`src/db/qa-gate-profile.ts`** — Updated JSDoc for `getEffectiveGates()`: removed outdated "tracked as follow-up work" sentence; added enumerated list of active enforcement consumers including hallucination_guard enforcement in `phase_complete.ts`

## Why

The `hallucination_guard` flag was added in v6.72.x to allow plan review processes to opt into claim verification. However, it was never wired to block execution, making it a declaration without teeth. Users could enable the gate, but `phase_complete` would succeed regardless of whether verification evidence existed or was approved.

This release follows the established `critic_drift_verifier` pattern to complete the enforcement loop: when enabled, hallucination verification becomes mandatory per phase, and phase_complete blocks until evidence is written with an APPROVED verdict. This provides concrete integrity guarantees for claim-heavy and research-heavy work.

## Migration steps

No migration needed. The gate defaults to `false` (disabled), maintaining backward-compatible behavior. To activate hallucination verification:

1. Run `/swarm plan` and elect `hallucination_guard` during QA gate selection dialogue
2. At PHASE-WRAP, architect will delegate to `critic_hallucination_verifier`
3. Review the verifier's findings; if rejected, fix claims and re-delegate
4. Once verifier approves, write evidence with `write_hallucination_evidence` and proceed to `phase_complete`

Alternatively, enable via session override:
```
/swarm qa-gates override hallucination_guard
```

## Breaking changes

None. The feature is opt-in (default disabled) and adds no API changes.

## Known caveats

- **Spec.md optional**: Unlike `critic_drift_verifier`, the hallucination verifier fires even when `.swarm/spec.md` is missing. Claims can exist in plan.md or implicit in code without a formal spec. This is intentional — spec is guidance, not a prerequisite for claim verification.
- **Profile lock ordering**: If `critic_drift_verifier` runs first (step 5.5) and locks the QA profile, and the hallucination gate was never elected during initial QA GATE SELECTION, step 5.55 skips silently. A new plan cycle is required to retroactively enable hallucination verification. This matches profile lock semantics — gates must be elected upfront.
- **Turbo mode bypass**: When turbo mode is active, both drift and hallucination gates are bypassed in `phase_complete`. This is intentional — turbo mode trades thoroughness for speed.
- **No double-lock**: `write_hallucination_evidence` intentionally does NOT call `takeSnapshotEvent` or `lockProfile`. Only drift verifier approval locks the profile. This prevents double-locking and ensures hallucination verification is always reversible if a new plan cycle is needed.
