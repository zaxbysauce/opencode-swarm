# v7.20.1

## What changed

Architect sessions now receive next-turn recovery guidance when the latest
assistant message visibly ended with a transient provider outage such as
`provider_unavailable` or `Network connection lost`.

The guidance is injected through the existing model-only advisory surface and is
deduplicated with a session-scoped fingerprint, so the same interrupted
transcript does not repeatedly inject the recovery instruction.

## Why

Issue #856 showed that a brief OpenRouter outage could stop an active swarm.
The structured provider-error classification fix handles retry accounting for
tool/subagent failures, but an interrupted Architect response is only visible to
the plugin once it appears in transcript history. The safest recovery path is to
make the following Architect turn continue from the last stable step instead of
treating the provider interruption as task completion.

## Migration steps

No configuration changes are required.

## Breaking changes

None.

## Known caveats

- Architect recovery is next-turn guidance, not an automatic retry of an
  interrupted provider response.
- The recovery detector is intentionally text-pattern based and bounded to one
  advisory per interrupted transcript. Transcript text that literally includes
  provider outage markers can trigger the advisory even when quoted for
  discussion.
