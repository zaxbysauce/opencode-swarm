# v6.84.4

## What changed

### Bug fix: council gate no longer blocks when `council_mode` is disabled

`checkCouncilGate` (called by `update_task_status` on every completion transition)
previously blocked task advancement whenever `pluginConfig.council.enabled === true`,
regardless of the `council_mode` flag in the per-plan QA gate profile.

This was inconsistent with `isCouncilGateActive` in `state.ts`, which already
implements correct AND semantics: the council gate is only authoritative when
**both** `council.enabled === true` **and** `council_mode === true`.

The result was that architects who had left `council_mode` at its default (`false`)
were still forced to run `declare_council_criteria` → `convene_council` on every
task completion, wasting cycles and blocking the workflow.

`checkCouncilGate` now applies the same AND gate as `isCouncilGateActive`:

| `council.enabled` | `council_mode` | gate active? |
|---|---|---|
| `true` | `true` | ✅ yes — council evidence required |
| `true` | `false` (default) | ❌ no — gate inactive |
| `false` | any | ❌ no — feature off |

When `plan.json` is missing or the QA gate profile DB does not exist, the gate
falls back to inactive (same behaviour as `isCouncilGateActive`).

## Why

Reported via a swarm architect trace where the architect had `council_mode: false`
in the QA gate profile but was still repeatedly forced to convene councils in order
to mark tasks complete. The root cause was the missing `council_mode` check in
`checkCouncilGate`.

## Migration steps

None. This is a pure bug fix. Projects that have both `council.enabled: true` **and**
`council_mode: true` in the QA gate profile will continue to require council evidence
exactly as before. Projects with `council_mode: false` (the default) will no longer
be blocked.

None.
