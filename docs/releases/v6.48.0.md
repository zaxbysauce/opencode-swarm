# v6.48.0 — Tool Audit, Regression Sweep Fixes & Doctor Expansion

## What changed

### Tool Registration (Phase 1)
- Registered 6 previously missing tools in the plugin `tool: {}` block: `syntax_check`, `placeholder_scan`, `quality_budget`, `sast_scan`, `sbom_generate`, `build_check`
- Added `createSwarmTool()` wrappers for `syntax_check`, `placeholder_scan`, and `quality_budget` (were plain function exports)
- Updated `src/tools/index.ts` barrel to export both tool definitions and raw functions for backward compatibility
- Added conformance test (`src/tools/verify-six-tools-registration.test.ts`) that asserts every `TOOL_NAMES` entry has a matching key in the registered `tool: {}` block — prevents future registration gaps

### test_runner Ambiguous Signals (Phase 2)
- Added `RegressionOutcome` type (`pass` | `skip` | `regression` | `scope_exceeded` | `error`) to `src/tools/test-runner.ts`
- Added `outcome` field to all `TestSuccessResult` and `TestErrorResult` return sites
- Zero-files guard now returns `outcome: 'skip'` with `graphFallbackReason` and `attempted_scope: 'graph'` — no longer returns a spurious error
- Too-many-files guard now returns `outcome: 'scope_exceeded'` with a SKIP instruction — no longer stalls the architect in a retry loop
- Updated architect step 5l-bis to route on `outcome` values instead of inferring signal from error text
- Updated test-runner tests with outcome field assertions

### Concurrent Write Safety (Phase 3)
- Added `tryAcquireLock` (via `proper-lockfile`) to `executeUpdateTaskStatus` before writing `plan.json` — prevents lost updates under concurrent load
- Added `tryAcquireLock` to `phase_complete` events.jsonl write — serializes concurrent appends where possible; if lock unavailable, write proceeds with a warning (advisory locking — `events.jsonl` is append-only, duplicate entries do not corrupt state)
- Lock failure returns `success: false` with `recovery_guidance: "retry"` rather than silently dropping the write
- Added concurrent-write test suite verifying: no `plan.json` corruption, exactly 1 lock winner per batch, losers return actionable errors, sequential calls all succeed

### Knowledge Injection & pre_check_batch (Phase 4)
- `knowledge-injector.ts`: replaced silent `return` at 75K char threshold with `console.warn` including the actual char count — skips are now visible in logs
- Architect `pre_check_batch` handling: if ALL four tools (`lint`, `secretscan`, `sast_scan`, `quality_budget`) have `ran === false`, treat as SKIP not failure — prevents spurious coder rejections when tools can't run
- `guardrails.ts`: `gates_passed: false` check now excludes skip conditions (`ran === false`) from triggering `lastGateFailure`

### `/swarm doctor tools` Command (Phase 5)
- Added new subcommand `/swarm doctor tools` (registered in `src/commands/registry.ts`)
- Runs three checks:
  1. **Tool registration coherence** — every `TOOL_NAMES` entry has a key in the plugin `tool: {}` block (errors as `severity: error`)
  2. **AGENT_TOOL_MAP alignment** — every tool assigned to an agent is registered in the plugin (warns as `severity: warn`)
  3. **Class 3 binary readiness** — checks `ruff`, `cargo`, `golangci-lint`, `mvn`, `gradle`, `dotnet`, `swift`, `swiftlint`, `dart`, `flutter`, `biome`, `eslint` on PATH (warns as `severity: warn`)
- New `src/services/tool-doctor.ts` service implementing all three checks

## Why

- **GitHub #395**: 6 tools were registered in `TOOL_NAMES` and `AGENT_TOOL_MAP` but absent from the plugin `tool: {}` block — agents that attempted to call them would get "tool not found" errors
- **GitHub #396**: `test_runner` with `scope: "graph"` would return an ambiguous error signal when no test files resolved, causing the architect to loop on retry instead of treating it as a skip
- Race conditions in `update_task_status` and `phase_complete` were causing silent lost writes when multiple swarm agents ran in parallel
- Silent knowledge injection skips made debugging context budget issues very difficult
- `pre_check_batch` returning `gates_passed: false` when tools simply couldn't run (no toolchain detected) was causing unnecessary coder rejections

## Migration steps

No breaking changes. All new fields (`outcome`) are additive. Existing integrations continue to work.

If you have custom architect prompts that inspect `test_runner` output by text matching, update them to use the `outcome` field:
- `outcome: "skip"` → no test files found, treat as SKIP
- `outcome: "scope_exceeded"` → too many files, treat as SKIP
- `outcome: "regression"` → tests ran and failed, trigger retry
- `outcome: "pass"` → all tests passed
- `outcome: "error"` → infra/timeout failure, treat as SKIP

## Known caveats

- `tryAcquireLock` uses `retries: 0` (fail-fast). For `update_task_status` (hard lock), concurrent callers that lose the lock receive `success: false` with `recovery_guidance` advising a retry. There is no automatic backoff or queuing — callers must retry explicitly.
- `phase_complete` uses advisory locking for `events.jsonl`. If two agents complete a phase simultaneously, both succeed and both events are written. This is by design — `events.jsonl` is append-only and duplicate phase events are recoverable.
- `/swarm doctor tools` binary readiness checks are advisory (`severity: warn`) — a missing binary does not prevent the command from completing.
