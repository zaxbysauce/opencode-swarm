# OpenCode Swarm v6.30.1 Release Notes

## Summary

Version **6.30.1** is a targeted bug-fix release addressing seven confirmed issues surfaced by the post-merge QA audit of v6.29.0–v6.30.0. All fixes were verified against the live v6.30.0 source before implementation. The release also includes two critical reliability fixes: a spawnAsync OOM vulnerability (infinite-output commands could accumulate hundreds of MB before the timeout fired) and three pre-existing flaky test races in the snapshot/rehydration test suite.

---

## Bug Fixes

### 1. Package manager detection uses lockfile — not always `bun` (FR-001)

**Affected:** `src/hooks/incremental-verify.ts` — `detectTypecheckCommand`

**Problem:** The `typecheck` and `type-check` script branches always returned `['bun', 'run', ...]` regardless of the project's actual package manager. Projects using npm, pnpm, or yarn received ENOENT failures silently — the incremental-verify hook appeared to run but did nothing.

**Fix:** A new private `detectPackageManager(projectDir)` helper checks for lockfiles in priority order: `bun.lockb` → bun, `pnpm-lock.yaml` → pnpm, `yarn.lock` → yarn, `package-lock.json` → npm. Falls back to bun when no lockfile is found (preserves existing default).

**Files changed:**
- `src/hooks/incremental-verify.ts` — added `detectPackageManager`, updated two returns in `detectTypecheckCommand`
- `src/hooks/incremental-verify.test.ts` — 10 new tests covering all four lockfile types, priority ordering, and no-lockfile fallback

---

### 2. String command with spaces in path now warns instead of silently breaking (FR-002)

**Affected:** `src/hooks/incremental-verify.ts` — string command parsing

**Problem:** `config.command.split(' ')` on a string like `"python -m mypy C:\\My Projects\\src"` produces `['python', '-m', 'mypy', 'C:\\My', 'Projects\\src']` — a malformed command array that causes silent failures.

**Fix:** When `config.command` is a string, each token from `split(' ')` is checked for path separators (`/` or `\`). If a space-separated string contains path separators, an advisory is injected recommending the array form and execution is skipped. Single-token paths (e.g. `"./scripts/check.sh"`) are not flagged.

---

### 3. Windows spawn now resolves npm, npx, pnpm, yarn correctly (FR-003)

**Affected:** `src/hooks/spawn-helper.ts` — `spawnAsync`

**Problem:** On Windows, `spawn('npx', ...)` fails with ENOENT because the binary is installed as `npx.cmd`. Without `.cmd`, Node.js's spawn cannot find it.

**Fix:** A module-level constant `WIN32_CMD_BINARIES = new Set(['npm', 'npx', 'pnpm', 'yarn'])` is checked at spawn time. When `process.platform === 'win32'` and the command matches with no existing extension, `.cmd` is appended. Non-Windows and all other commands are unchanged.

**Files changed:**
- `src/hooks/spawn-helper.ts` — `WIN32_CMD_BINARIES` constant + platform guard
- `src/hooks/spawn-helper.test.ts` — 6 new tests for `.cmd` behavior with platform mocking

---

### 4. spawnAsync OOM vulnerability: 512KB output cap (hotfix)

**Affected:** `src/hooks/spawn-helper.ts` — `spawnAsync`

**Problem:** The stdout/stderr data handlers accumulated output into strings without any cap (`stdout += d`). Commands producing infinite output (e.g. `cat /dev/zero`) could accumulate hundreds of megabytes before the timeout timer fired, causing `RangeError: Invalid string length` and crashing the process. Measured: **372MB in 80ms** on this machine.

**Fix:** Both handlers now enforce a 512KB cap. Once reached, the string is sliced and the stream is destroyed immediately — the timeout and kill logic remain unchanged.

```typescript
const MAX_OUTPUT = 512 * 1024;
proc.stdout.on('data', (d: Buffer) => {
    if (stdout.length < MAX_OUTPUT) {
        stdout += d;
        if (stdout.length >= MAX_OUTPUT) {
            stdout = stdout.slice(0, MAX_OUTPUT);
            try { proc.stdout.destroy(); } catch { /* ignore */ }
        }
    }
});
```

**Related test fix:** The timeout test in `incremental-verify.test.ts` used `cat /dev/zero` as a "blocking" command. It was replaced with `sleep 30`, which blocks without producing output.

### 5. Curator knowledge updates now receive populated KnowledgeConfig (FR-004)

**Affected:** `src/tools/phase-complete.ts` — `applyCuratorKnowledgeUpdates` call

**Problem:** `applyCuratorKnowledgeUpdates` was called with `{} as KnowledgeConfig`, disabling all config-driven features (dedup threshold, max entries, scope filter, etc.). The fully-populated `knowledgeConfig` object was constructed 60 lines earlier but went out of scope.

**Fix:** The `knowledgeConfig` construction was hoisted above the `retroFound` conditional block so it is in scope at the `applyCuratorKnowledgeUpdates` call. Values are unchanged; only the variable scope changed.

---

### 6. Advisory messages no longer accumulate in coder sessions (FR-006)

**Affected:** `src/hooks/guardrails.ts` — `messagesTransform`

**Problem:** The `pendingAdvisoryMessages` queue was only cleared inside the `isArchitectSession` branch. In long-lived coder sessions, advisories pushed by slop-detector and compaction accumulated indefinitely until the 2-hour TTL evicted the session.

**Fix:** An `else if` branch clears the queue for non-architect sessions without injecting the messages (advisories are architect-only). This drains the queue on every `messagesTransform` call.

---

### 7. Rehydration race guard: concurrent loadSnapshot no longer silently drops workflow state (FR-005)

**Affected:** `src/state.ts`, `src/session/snapshot-reader.ts`

**Problem:** `startAgentSession` fired `rehydrateSessionFromDisk` as fire-and-forget. If `loadSnapshot` → `rehydrateState` ran concurrently and called `agentSessions.clear()` before the rehydration completed, the in-flight workflow state was silently lost.

**Fix:** A `pendingRehydrations: Set<Promise<void>>` field on `swarmState` tracks in-flight rehydrations. `startAgentSession` registers each rehydration promise and removes it in a `finally` block. `rehydrateState` (now async) calls `await Promise.allSettled([...pendingRehydrations])` before clearing `agentSessions`, ensuring all in-flight rehydrations complete first. Errors remain swallowed (non-fatal contract preserved).

---

### 8. CompactionState cross-session bleed documented (FR-007)

**Affected:** `src/services/compaction-service.ts`

**Problem:** Module-level `CompactionState` is shared across all concurrent sessions. When two architect sessions run simultaneously, session A's compaction hysteresis can suppress session B's advisories.

**Change:** Added a `KNOWN LIMITATION` comment above the module-level `state` declaration explaining the incorrect suppression scenario and a `TODO` for per-sessionId keying in a future release. No functional change.

---

### 9. Pre-existing flaky test races fixed

Three pre-existing test reliability issues were fixed as part of this release:

| File | Problem | Fix |
|------|---------|-----|
| `tests/unit/session/snapshot-integration.test.ts` | Unawaited `Bun.write(...)` created an unhandled rejection when afterEach deleted tmpDir before the async write completed | Replaced with synchronous `mkdirSync + writeFileSync` |
| `src/state.session-restart.test.ts` | `afterEach` only deleted the test session from agentSessions but didn't call `resetSwarmState()`, allowing stale state to leak to concurrently-running test files | Changed to `resetSwarmState()` in afterEach |
| `src/state.rehydration-integration.test.ts` | Missing `resetSwarmState()` in beforeEach allowed cross-file contamination via shared swarmState singleton | Added `resetSwarmState()` to beforeEach |

Additionally, all `rehydrateState()` callers in the snapshot-reader test files were updated to `await rehydrateState()` after the function was made async (Task 7). The adversarial test assertion `.resolves.not.toThrow()` was corrected to `.resolves.toBeUndefined()` for `Promise<void>` functions.

## Upgrade Notes

- **Backward compatible** — no breaking changes, no API surface changes, no configuration changes required
- **Package manager detection** — if your project uses npm, pnpm, or yarn, `incremental_verify` will now automatically use the correct CLI. No configuration change needed if you have a lockfile.
- **Windows users** — `incremental_verify` now correctly resolves `npx.cmd`, `npm.cmd`, `pnpm.cmd`, and `yarn.cmd` on Windows without any configuration change.
- **spawnAsync output cap** — commands producing more than 512KB of output will be truncated. For typecheck/lint tools this is not a concern; only adversarial or debugging scenarios produce that volume.
- **Rehydration** — `rehydrateState` in `snapshot-reader.ts` is now async. Any direct callers outside tests should `await` it.
