# OpenCode Swarm v6.33.7 Release Notes

## Summary

Version **6.33.7** fixes the critical session freeze that occurred when a sub-agent (coder, test_engineer, etc.) completed its work via the Task tool but the architect never resumed ŌĆö leaving the TUI completely unresponsive.

---

## Bug Fixes

### 1. Task handoff ran after hooks ŌĆö any hook hang froze the session

**Root cause:** The Task tool completion handoff (restoring `activeAgent` to architect and setting `delegationActive = false`) was positioned at the **very end** of the `tool.execute.after` hook chain ŌĆö after 12+ sequential async hooks. If any hook produced an unresolved promise (due to Windows file-system stalls, virus scanner locks on `Bun.write`/`renameSync`, or `fs.promises.readFile` hanging on locked files), the handoff never executed. Since OpenCode `await`s `tool.execute.after`, the entire TUI froze.

The v6.33.1 conversion of `getEvidenceTaskId` from synchronous (`readFileSync`) to asynchronous (`fs.promises.readFile`) introduced a new await point inside `delegationGateHooks.toolAfter` that is susceptible to Windows NTFS file-locking stalls. Combined with the handoff-at-end architecture, any stall in this code path caused a permanent session freeze.

**Fix (two-part):**

1. **Early handoff:** Moved the Task tool handoff to the **very first operation** in `tool.execute.after`, before any hooks run. The architect identity is now restored immediately, regardless of hook behavior.

2. **Timeout protection:** Wrapped the entire hook chain in a 30-second `Promise.race` timeout. If any hook hangs, the toolAfter function force-returns instead of blocking forever. Diagnostic logging identifies which hook timed out.

### 2. `getEvidenceTaskId` re-threw unexpected errors outside evidence try-catch

The `getEvidenceTaskId` function (converted from sync to async in v6.33.2) re-threw unexpected file-system errors (EPERM, EBUSY, etc.) at lines 402-406. These calls were positioned **outside** the evidence recording try-catch block, meaning the re-thrown errors propagated up to `safeHook` instead of being handled locally.

On Windows, virus scanner interference or NTFS metadata locks produce EBUSY errors that were re-thrown, causing the entire `delegationGateHooks.toolAfter` to abort via safeHook rather than gracefully continuing.

**Fix:**
- `getEvidenceTaskId` now never throws ŌĆö all errors return `null` (evidence task ID lookup is best-effort)
- Both evidence recording blocks (stored-args path and delegation-chain fallback path) are now fully wrapped in try-catch, including the `getEvidenceTaskId` call

---

## Architecture Change

The `tool.execute.after` hook now follows a **handoff-first, hooks-second** pattern:

```
BEFORE (v6.33.0ŌĆōv6.33.6):
  1. Run 12+ hooks sequentially (any can hang)
  2. Task handoff (set activeAgent ŌåÆ architect)

AFTER (v6.33.7):
  1. Task handoff (set activeAgent ŌåÆ architect)  ŌåÉ immediate, no I/O
  2. Run hooks with 30s timeout protection
```

This ensures the architect identity is always restored, even if hooks encounter I/O stalls.
