# v6.55.0

## What changed

### Conflict resolution mechanisms (Issue #414)

Full implementation of the Conflict Resolution Mechanisms Audit Report. This release adds new infrastructure types, a centralized conflict-resolution hook, an adversarial detection pattern, delegation-reason tracking, a typed sounding-board parser — and live-wires all of it into two runtime paths.

---

#### New event types (`src/types/events.ts`)

- **`AgentConflictDetectedEvent`** — telemetry event emitted when two agents enter a reject/resubmit conflict. Carries `sourceAgent`, `targetAgent`, `conflictType` (feedback_rejection | authority_collision | retry_spiral | scope_disagreement | quality_gate_dispute), `resolutionPath` (self_resolve | soundingboard | simplification | sme_consult | user_escalation), and a free-text `summary`.
- **`AuthorityHandoffResolvedEvent`** — telemetry event type defined for when a stale or conflicted delegation authority is resolved. Carries `previousAgent`, `newAgent`, and `reason` (task_complete | stale_delegation | conflict_escalation | manual_reset). **Not yet emitted** — type is defined and included in `V619Event` for forward compatibility; emit wiring is deferred to a follow-up PR.
- Both types added to the `V619Event` union.

---

#### New hook (`src/hooks/conflict-resolution.ts`)

- **`resolveAgentConflict(input: ResolveAgentConflictInput): void`** — centralized conflict-resolution policy. Encapsulates advisory injection and telemetry emission for agent-vs-agent conflicts.
  - `rejectionCount < 3` → pushes a `CONFLICT DETECTED` advisory into `pendingAdvisoryMessages` (`self_resolve` path). Instructs the agent to attempt self-resolution using `.swarm/plan.md`, `.swarm/spec.md`, and `.swarm/context.md`.
  - `rejectionCount >= 3` → pushes a `CONFLICT ESCALATION` advisory (`soundingboard` path). Instructs routing to Critic in SOUNDING_BOARD mode before any user escalation.
  - No-ops if no session exists for `sessionID`.
  - Emits `agent_conflict_detected` telemetry (fire-and-forget cast; see Known caveats).

---

#### Live wiring — Wire Target A (`src/hooks/guardrails.ts`)

`resolveAgentConflict()` is now called automatically on every reviewer→coder rejection cycle:

- **Trigger**: `coderRevisions > 1 && qaSkipCount === 0` inside the `toolAfter` handler for coder task completions.
  - `coderRevisions > 1` ensures the coder has been delegated more than once (i.e., a prior reviewer cycle occurred).
  - `qaSkipCount === 0` guards against false positives: if the architect re-delegated to the coder without invoking reviewer/test_engineer (a QA gate skip), `qaSkipCount` is non-zero at that point, so the conflict hook does **not** fire. This correctly distinguishes reviewer rejections from gate-bypass violations, which are handled separately by the QA enforcement logic.
- **Phase**: derived from `loadPlan()` + `extractCurrentPhaseFromPlan()` — same async pattern used by the reviewer-tracking block in the same function. Avoids stale phase numbers that could arise from reading `reviewerCallCount` keys across task boundaries.
- **Call**: `resolveAgentConflict({ sourceAgent: 'reviewer', targetAgent: 'coder', conflictType: 'feedback_rejection', rejectionCount: coderRevisions - 1, ... })`.
- **Side effect**: `session.lastDelegationReason = 'review_rejected'` is set at the same site.

---

#### Live wiring — Wire Target B (`src/index.ts`)

`parseSoundingBoardResponse()` is now called automatically when a `critic_sounding_board` delegation completes:

- **Trigger**: `baseAgentName === 'critic_sounding_board'` inside the task-handoff block of `toolAfter`, after the existing `[PIPELINE]` advisory is injected.
- **Output is not truncated**: `input.tool === 'task'` for agent delegations, which is not in `defaultTruncatableTools`, so the full critic response is available at parse time.
- **On successful parse**: a structured `[SOUNDING_BOARD] Verdict: <V>. <reasoning>` advisory is pushed into `pendingAdvisoryMessages`, with optional `Rephrase to:`, `Answer:`, and `WARNING:` suffixes. `taskSession.lastDelegationReason = 'critic_consultation'` is set.
- **On parse failure** (malformed or empty response): a fallback `[SOUNDING_BOARD] WARNING: Could not parse a structured verdict...` advisory is injected so the architect is not left without guidance. The fallback instructs the architect to treat the response as APPROVED and proceed, but review the raw output.

---

#### New adversarial pattern (`src/hooks/adversarial-detector.ts`)

- **`REJECTION_SPIRAL`** — new pattern group added to `detectAdversarialPatterns()`. Triggers at `severity: HIGH` / `confidence: HIGH` when agent text signals a stuck reject/resubmit cycle.
  - Pattern 1: ordinal repetition — "rejected again for the 3rd time", "failed review for the 4th time".
  - Pattern 2: same-feedback recurrence — "same feedback again", "same issues repeated multiple times".
  - Pattern 3: stuck-cycle language — requires explicit `stuck|trapped|endless|repeating` **and** `loop|cycle`. Chosen to avoid false positives on neutral process language like "our feedback loop for the reviewer".
- `AdversarialPatternMatch.pattern` union extended with `'REJECTION_SPIRAL'`.

---

#### Delegation reason tracking (`src/state.ts`, `src/hooks/delegation-tracker.ts`)

- **`DelegationReason` type** exported from `src/state.ts`: `normal_delegation | review_rejected | critic_consultation | retry_circuit_breaker | conflict_escalation | stale_recovery`.
- **`lastDelegationReason?: DelegationReason`** added to `AgentSessionState`. Populated at three sites:
  - `'normal_delegation'` — set by `createDelegationTrackerHook` for every non-architect agent activation.
  - `'review_rejected'` — set by Wire Target A when `resolveAgentConflict()` fires on a reviewer→coder rejection cycle.
  - `'critic_consultation'` — set by Wire Target B when a parseable sounding-board verdict is received.
- `DelegationReason` re-exported from `src/hooks/delegation-tracker.ts` for consumer convenience.

---

#### Sounding board parser (`src/agents/critic.ts`)

- **`SoundingBoardVerdict`** type: `'UNNECESSARY' | 'REPHRASE' | 'APPROVED' | 'RESOLVE'`.
- **`SoundingBoardResponse`** interface: `verdict`, `reasoning`, optional `improvedQuestion` (REPHRASE path), `answer` (RESOLVE path), `warning` (MANIPULATION DETECTED path).
- **`parseSoundingBoardResponse(raw: string): SoundingBoardResponse | null`** — regex-based parser for raw Critic sounding board output. Case-insensitive, flexible whitespace. Returns `null` for empty input or missing/unrecognized verdict. Optional-field regexes use explicit section-header lookaheads (`Answer|Warning|Verdict`) to correctly capture multiline values without cross-field bleed. Populates `warning` from explicit `Warning:` lines or `[MANIPULATION DETECTED]` markers.

---

#### Tests

- `src/__tests__/conflict-resolution.test.ts` (new, 5 tests) — unit coverage for `resolveAgentConflict`: self-resolve path, escalation path, soundingboard advisory content, no-op on missing session, default `rejectionCount`.
- `src/__tests__/sounding-board-parser.test.ts` (new, 10 tests) — unit coverage for `parseSoundingBoardResponse`: all four verdicts, optional fields, `[MANIPULATION DETECTED]`, `Warning:` line, null on empty/no-verdict, case-insensitivity.
- `src/__tests__/security-adversarial.test.ts` (extended, 4 new tests) — `REJECTION_SPIRAL` detection: ordinal repetition, same-feedback, stuck-loop, no false positive on clean text.

## Why

The Conflict Resolution Mechanisms Audit (Issue #414) identified five gaps in the pipeline's ability to detect, record, and resolve agent-vs-agent conflicts:

1. No typed event for recording when two agents enter a conflict, making post-mortem analysis impossible.
2. No centralized policy for escalating repeated reviewer rejections — logic was scattered across prompts and hooks.
3. No adversarial pattern for detecting when the pipeline stalls in a reject/resubmit spiral.
4. No typed parser for Critic sounding-board output, forcing all consumers to handle raw strings.
5. No semantic reason recorded on delegation transitions, making conflict audit trails incomplete.

This release implements all five additions and wires the central policy and parser into their live runtime call sites.

## Migration steps

No migration required. All additions are purely additive:

- New event types in `V619Event` union — existing type guards on specific event types are unaffected.
- `lastDelegationReason?` is optional — existing `AgentSessionState` consumers do not break.
- `resolveAgentConflict` is not on the plugin's public surface — no accidental API exposure.
- `guardrails.ts` and `index.ts` changes are additive blocks; no existing code paths were modified or removed.

## Breaking changes

None.

## Known caveats

- `agent_conflict_detected` telemetry is emitted via a type cast (`'agent_conflict_detected' as Parameters<typeof emit>[0]`) because the string is not yet in the `TelemetryEvent` union. The event appears correctly in `telemetry.jsonl` and is delivered to listeners at runtime; the type gap is a maintenance concern only. Union wiring is deferred to a follow-up PR.
- `AuthorityHandoffResolvedEvent` is defined in `src/types/events.ts` and included in `V619Event` but is **not yet emitted** anywhere in runtime code. It is present for forward compatibility. An emit site will be added in a follow-up PR when stale-delegation recovery is instrumented.
- Wire Target A fires on every coder re-delegation where `qaSkipCount === 0`. In projects that never invoke test_engineer (so `qaSkipCount` never resets via the AND condition), this guard may have different characteristics — see `delegation-gate.ts` for the exact reset condition (`lastCoderIndex !== -1 && hasReviewer && hasTestEngineer`).
