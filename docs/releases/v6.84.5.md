# v6.84.5

## What changed

### Bug fix: prevent first-initialization error on new workspaces

The repo graph builder had a race condition that caused ENOENT errors on first initialization:
when `repoGraphHook.init()` was called fire-and-forget (not awaited), the async `saveGraph()` 
would dispatch `fsPromises.writeFile()` to a libuv worker thread before the `.swarm/` directory 
was created. The worker's async `open()` call would race ahead of the main thread's directory 
creation, resulting in "file not found" errors that only appeared on first run, not on subsequent runs 
(when `.swarm/` already existed).

**Root cause:** Two overlapping initialization issues:
1. Ordering: `initTelemetry()` (which creates `.swarm/` synchronously) was called AFTER `repoGraphHook.init().catch()` was fired
2. Race: Even with correct ordering, the async write could outpace directory creation

**Fixes applied:**
1. **Defensive mkdir in saveGraph**: Added `await fsPromises.mkdir(path.dirname(tempPath), { recursive: true })` before any file write operations in `saveGraph()`. This ensures the directory exists before the libuv worker attempts to open the temp file.
2. **Reorder initTelemetry**: Moved `initTelemetry(ctx.directory)` to execute BEFORE `createRepoGraphBuilderHook()` and `repoGraphHook.init().catch()`, ensuring `.swarm/` exists as a defensive second layer.
3. **Try-catch protection for malformed files in full rebuild**: Wrapped `extractTSSymbols()`, `extractPythonSymbols()`, and `parseFileImports()` calls in try-catch in `buildWorkspaceGraph()`. Single malformed source files can no longer abort the entire graph build.
4. **Try-catch protection for malformed files in incremental updates**: Applied identical try-catch protection to `scanFile()`, used by incremental graph updates when files change.
5. **Logging downgrade for non-fatal failures**: Changed graph build and incremental update error logging from `console.error` to `console.warn` in both `init()` and `toolAfter()` handlers. Graph failures are non-fatal startup issues; warn-level logging prevents false alarms in OpenCode UI.

## Why

Reported in issue #585: architects opening opencode-swarm for the first time on new workspaces saw a brief error message that disappeared on subsequent runs. The root cause was the race condition in `saveGraph()` combined with initialization order in the plugin startup sequence.
