# v6.82.2 — Issue #577: Prevent .swarm directories in project sub-folders

## What changed

- **save-plan tool**: Added project root anchor check — rejects `working_directory` arguments that point to any location other than the project root
- **resolveWorkingDirectory utility**: Extended anchor validation to all six tools using this shared helper (`completion_verify`, `check-gate-status`, `convene-council`, `declare-council-criteria`, `phase-complete`, `test-runner`)
- **role-filter**: Removed unsafe `process.cwd()` fallback in metrics logging
- **guardrails.ts**: Emits actionable warning when legacy call path falls back to `process.cwd()`
- **evidence-summary-integration**: Renamed `swarmDir` field to `projectDir` for clarity

## Why

Despite prior hardening in v6.71.1 (#528), `.swarm` directories were still being created in project subdirectories (reported in #577). Root cause: the `save_plan` tool and other directory-accepting tools did not validate that `working_directory` arguments resolved to the project root — they only checked for path traversal and existence. LLM agents could supply subdirectory paths (e.g., `src/components`), causing `.swarm` to be created there.

The fix closes two separate vulnerability paths:
1. **save_plan** — now anchors `working_directory` to project root with explicit path comparison
2. **All resolveWorkingDirectory callers** — shared utility now enforces project root as the only acceptable location

## Migration steps

None. The changes are backward-compatible:
- Agents that currently pass the project root as `working_directory` continue to work
- Agents passing subdirectories will receive a clear error message guiding them to use the project root
- Agents omitting `working_directory` continue to default to the project root (unchanged)

## Known caveats

None. The fix is comprehensive and covers all entry points for `.swarm` directory creation.
