# v6.71.0

## What changed

### Interpreter escape-hatch gating + redacted shell audit log

#### 1. Interpreter gating (`interpreter_allowed_agents`)

Adds an opt-in config field that restricts which agent roles may invoke the
`bash`/`shell` interpreter tools. When unset (default), all agents are allowed,
preserving backward compatibility.

```jsonc
{
  "guardrails": {
    "interpreter_allowed_agents": ["architect", "coder"]
  }
}
```

When a blocked agent (e.g. `reviewer`, `explorer`) attempts a `bash`/`shell`
call, the call is rejected before it executes:

```
BLOCKED: Agent "reviewer" is not permitted to use the bash/shell interpreter.
Allowed agents: ["architect", "coder"]
```

Tool-name matching is **case-insensitive** (`'Bash'`, `'BASH'`, `'shell'` are
all covered). Namespace prefixes (`opencode:bash`) are stripped via the existing
`normalizeToolName` utility.

#### 2. Shell audit log (`.swarm/session/shell-audit.jsonl`)

Every `bash`/`shell` tool call — including **denied attempts** — is appended to
`.swarm/session/shell-audit.jsonl` in the working directory before enforcement
runs. Audit log writes are fire-and-forget (errors are swallowed) so a log write
failure can never block execution.

Each entry is a JSON line:
```json
{
  "ts": "2026-04-16T01:23:45.678Z",
  "sessionID": "abc123",
  "agent": "coder",
  "tool": "bash",
  "command": "TOKEN=[REDACTED] node deploy.js"
}
```

The audit log is enabled by default (`shell_audit_log: true`). Disable with:
```jsonc
{ "guardrails": { "shell_audit_log": false } }
```

#### 3. Sensitive value redaction (`redactShellCommand`)

Before writing to the audit log, command strings are redacted by
`redactShellCommand`. Patterns covered:

| Pattern | Example → Redacted |
|---------|-------------------|
| Env-var assignments | `TOKEN=abc123` → `TOKEN=[REDACTED]` |
| CLI `--flag=value` | `--password=abc` → `--password=[REDACTED]` |
| CLI `--flag value` | `--token abc` → `--token [REDACTED]` |
| Bearer/Basic tokens | `Bearer eyJ...` → `Bearer [REDACTED]` |
| curl `-H` headers | `-H "Authorization: Bearer x"` → `-H "Authorization: [REDACTED]"` |

Non-sensitive commands pass through unchanged.

### Security hardening (adversarial review findings)

Three findings from the post-implementation adversarial review were fixed before
shipping:

- **Case-variant tool bypass**: `handleInterpreterGating` and `appendShellAuditLog`
  previously used exact string comparison (`tool !== 'bash'`). An agent invoking
  `tool: 'Bash'` or `tool: 'SHELL'` would bypass both gating and audit logging.
  Fixed by running the tool name through `normalizeToolName(tool).toLowerCase()`.

- **Audit after enforcement**: The original order ran enforcement first (throwing
  on denied calls) and audit second — blocked attempts were never logged. Fixed
  by running audit _before_ enforcement so both allowed and denied shell calls
  appear in the audit trail.

- **Short Bearer token leak**: The Bearer redaction regex required ≥8 characters,
  leaving short tokens (e.g. `Bearer abcd`) unredacted. Lowered to ≥4 characters.

## Why

- Reviewer and explorer agents have no legitimate reason to run arbitrary shell
  commands; gating them closes a privilege escalation vector where an
  LLM-controlled agent could exfiltrate data or modify system state.
- The audit log creates a forensic trail for incident response. Without it,
  unauthorized shell command attempts leave no record.
- Sensitive credential redaction prevents API keys and passwords from accumulating
  in plain-text log files.

## Migration steps

- **No breaking changes**. `interpreter_allowed_agents` defaults to `undefined`
  (all agents allowed). `shell_audit_log` defaults to `true` (auditing enabled).
- To restrict interpreter access, add `interpreter_allowed_agents` to the
  `guardrails` section of `.swarm/config.json`. Example: `["architect", "coder"]`.
- The `.swarm/session/` directory is created automatically on first bash/shell
  call. If you rotate or delete this directory, the file will be recreated.
- Sensitive patterns that appear in audit log entries before this version (if any)
  will not be retroactively redacted.

## Breaking changes

None.

## Known caveats

- **Positional credential arguments are not redacted**: Credentials passed as
  positional args without a recognizable flag (e.g. `mysecrettool mypassword`)
  are not redacted. Only named patterns (env-vars, flags, headers) are covered.
- **Audit log is append-only**: No log rotation is implemented. On long-running
  sessions with many shell calls, the file can grow large. Rotate manually if
  needed.
- **gating covers only `bash` and `shell`**: Custom interpreter tools registered
  with non-standard names are not covered by `interpreter_allowed_agents`.
