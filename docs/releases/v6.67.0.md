# v6.67.0

## What changed

### Council evidence writer hardening and round-history audit

Follow-up hardening for the Work Complete Council feature (v6.66.0, PR #477),
addressing LOW-severity findings from issue #478. All changes preserve the
off-by-default behavior — users who have not opted into `council.enabled: true`
see zero behavioral change.

- **Recursive `safeAssignOwnProps`**: The prototype-pollution filter
  (`__proto__`, `constructor`, `prototype`) now recurses into nested objects and
  arrays, not just the top level. Previously, a malicious
  `gates.reviewer.__proto__` would be copied by reference into the evidence file.
- **Empty-verdicts warning**: `synthesizeCouncilVerdicts` returns
  `emptyVerdictsWarning: true` when called with an empty verdicts array, while
  preserving the existing APPROVE-on-empty behavior for backward compatibility.
- **Round-history audit log**: Each call to `writeCouncilEvidence` appends a
  JSONL line to `.swarm/council/{taskId}.rounds.jsonl` with `{round, verdict,
  timestamp, vetoedBy}`. Failures are logged via `console.warn` and do not affect
  the primary evidence write.
- **Cross-reference comments**: `criteria-store.ts` (`safeId`) and
  `council-evidence-writer.ts` (`VALID_TASK_ID`) now have comments explaining
  their intentional divergence — `safeId` sanitizes for safe filenames while
  `VALID_TASK_ID` validates raw taskIds for evidence path consistency with
  `check_gate_status`.
- **Leading-zero regex documentation**: Task ID regex patterns across council
  files documented as consistent with the canonical `STRICT_TASK_ID_PATTERN` in
  `src/validation/task-id.ts`.
- **`escalateOnMaxRounds` documentation**: Replaced vague "forward compatibility"
  note with clear documentation of current behavior (user-facing via
  `buildUnifiedFeedbackMd`) and future wiring options.
- **Tool list ordering**: `completion_verify` positioned before
  `complexity_hotspots` in `AGENT_TOOL_MAP.architect` (alphabetical: at
  position 6, `t` < `x`).
- **Pre-existing test fixes**: Two config-gate tests (`convene-council.test.ts`,
  `declare-council-criteria.unit.test.ts`) that failed when no config file
  existed now seed explicit `council.enabled: false` config for deterministic
  isolation.

## Why

Round-1 and round-2 council reviews of the v6.66.0 PR identified several
LOW-severity findings: shallow prototype-pollution filtering, missing test
coverage for edge cases, vague documentation on deferred config fields, and
test isolation failures caused by user-level config leaking into test
environments.

## Migration steps

None. All changes are backward-compatible.

## Breaking changes

None.

## Known caveats

- **Adversarial test file uses co-located pattern**: The adversarial tests for
  `safeAssignOwnProps` live in `src/council/__tests__/` rather than
  `tests/unit/council/`. This is intentional — the tests re-implement the
  function to test its invariants in isolation without importing internal
  module state.
- **Council re-convocation not yet wired**: Issue #478 requests that a
  four-member council be re-convened for structurally significant changes.
  The `convene_council` and `declare_council_criteria` tools exist and are
  functional, but the architect prompt does not yet automatically dispatch
  them. This is tracked as a future Phase 4 item.
