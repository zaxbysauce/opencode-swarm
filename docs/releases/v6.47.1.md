# v6.47.1 Release Notes

## What Changed

### PlanSyncWorker — No Longer Reverts plan.json (Issue #384)

`PlanSyncWorker` previously called `loadPlan()` on every file-system change event.
`loadPlan()` is write-capable: when it detects a hash mismatch between `plan.json`
and the ledger, it replays the ledger and overwrites `plan.json`. After a swarm
identity migration (where the `swarm` field changes), the hash was always a mismatch,
so every sync cycle silently reverted the plan to stale state within 300–500 ms.

`PlanSyncWorker` now calls `loadPlanJsonOnly()` + `regeneratePlanMarkdown()` — a
read-only path that keeps `plan.md` in sync without touching `plan.json` or consulting
the ledger.

### Migration-Aware Ledger Guard in `loadPlan()`

The hash-mismatch rebuild guard now checks whether the ledger belongs to the same
swarm identity before triggering a destructive rebuild. If the latest ledger event
carries a different `plan_id` (i.e. the ledger predates a swarm rename / migration),
the rebuild is skipped and a warning is logged. Legitimate same-identity drift still
triggers the rebuild as before.

### Ledger Identity Re-init in `savePlan()`

When `savePlan()` detects a swarm identity change (new `plan_id` ≠ first event's
`plan_id` in the existing ledger), it archives the old ledger to a timestamped
`.archived-*` file and initialises a fresh ledger for the new identity. This prevents
the old ledger from poisoning future hash comparisons after a rename.

### Snapshot Event `plan_id` Fix

`takeSnapshotEvent()` was writing `plan_id: plan.title` instead of the canonical
`${plan.swarm}-${plan.title}` format used everywhere else. All ledger lookups now
use a consistent `plan_id`.

### `applyEventToPlan()` Return-Value Bug Fix

A `break` inside the `task_status_changed` switch case caused the function to
return `undefined` instead of the updated plan when the incoming status failed
schema validation. The loop in `replayFromLedger()` then passed `undefined` into
the next iteration, silently corrupting the replay. Changed `break` → `return plan`
so the function always returns a valid plan object.

### `updateTaskStatus()` No Longer Triggers Ledger Rebuild

`updateTaskStatus()` now calls `loadPlanJsonOnly()` instead of `loadPlan()`, avoiding
the revert-then-commit pattern that could overwrite in-flight user edits.

### Atomic Temp-File Path in `appendLedgerEvent()`

The temporary file used during ledger appends now uses a random suffix
(`.tmp.<timestamp>.<random>`) instead of the static `.tmp`, eliminating write
collisions when multiple processes append events concurrently.

## Why

Issue #384 reported that the swarm plan was being aggressively reverted to an older
state within milliseconds of any save. Root cause was the combination of:

1. `PlanSyncWorker` calling the write-capable `loadPlan()` on every `fs.watch` event
2. A swarm identity migration causing every `computePlanHash()` call to diverge from
   the pre-migration ledger hash
3. No guard to distinguish "legitimate drift" from "ledger belongs to a different identity"

The full fix is layered: the sync worker is now read-only, and `loadPlan()` itself is
now migration-aware so the guard can no longer fire across identity boundaries.

## Migration Steps

No action required. The ledger identity re-init in `savePlan()` is automatic on the
next save after a swarm rename. The old ledger is preserved as a `.archived-*` file
in `.swarm/` for audit purposes.

## Breaking Changes

None.

### Follow-up Fixes (post-review hardening)

Six additional issues identified in PR review were fixed before merge:

**`updateTaskStatus()` restored to `loadPlan()`** — Switching to `loadPlanJsonOnly()`
caused data loss when plan.json lagged behind the ledger (same-identity drift). The
migration guard in `loadPlan()` now makes it safe to call from `updateTaskStatus()`.

**`loadPlan()` validation-failure path gets identity guard** — When plan.json fails
schema validation, the old code called `replayFromLedger()` unconditionally. This
allowed a post-migration ledger to overwrite an invalid-but-migrated plan.json. The
catch path now extracts swarm+title from the raw JSON and applies the same identity
guard as the happy path.

**`loadPlan()` identity guard standardized on first event** — The migration guard in
`loadPlan()` was comparing against the *last* ledger event's plan_id, but `savePlan()`
uses the *first* event. Both now use `events[0]` (the `plan_created` anchor).

**`initLedger()` hash computed from in-memory plan** — `initLedger()` previously read
plan.json from disk to compute its initial hash, but it's called before the new
plan.json is written. An optional `initialPlanHash` parameter allows the caller to
pass the correct hash, preventing spurious hash-mismatch rebuild cycles after
identity-only swarm renames.

**`savePlan()` TOCTOU guard** — Two concurrent `savePlan()` calls during a swarm
rename could both attempt to archive the old ledger and reinitialize. The rename is
now guarded with an existence check; `initLedger()` errors are caught and treated as
"another caller already did it."

**`replayFromLedger()` filters by first-event plan_id** — Events from mixed-identity
ledgers (created before this fix was deployed) are now filtered to the first event's
plan_id before replay, preventing old-identity task states from corrupting the
reconstructed plan.

## Known Caveats

One concurrency edge case remains as known tech debt:

- **Concurrent `appendLedgerEvent()` write loss**: The read-rename-write pattern in
  `appendLedgerEvent()` is not atomic at the OS level. Two concurrent writers can
  both read the file before either renames it; the second writer's rename overwrites
  the first writer's event. File locking or a single-writer queue is the correct fix.
