# v6.61.0

## What changed

**Command documentation discoverability** — AI models and CLI users can now determine what any `/swarm` command does without reading source files.

- Added `details` (2-3 line behavioral summary) and `args` (flag/argument documentation) fields to `CommandEntry` type in the command registry
- Populated `details` and `args` for all 29 non-trivial commands across HIGH and MEDIUM complexity tiers
- Rewrote `buildSlashCommandsList()` in architect agent to emit a formatted markdown reference grouped by 7 categories (Session Lifecycle, Planning, Execution Modes, Observation, Knowledge, State Management, Diagnostics) instead of bare comma-separated command names
- Updated the SLASH COMMANDS section header in the architect prompt to remove TUI-only guidance
- Extended CLI `HELP_TEXT` (shown on bare `/swarm`) to include `args`, `details`, and grouped subcommands under parent commands
- Fixed `simulate` command description to accurately describe hidden coupling analysis
- Added `details` to `turbo` and `full-auto` commands (previously missing)
- Fixed flaky `doc-scan-wiring` integration test — cache validation now compares stored mtime instead of `scanned_at` timestamp, eliminating a race condition on CI runners where filesystem clock and `Date.now()` could disagree within the same millisecond

## Why

The `/swarm close` command — and many others — were documented only by a single-line description in the registry. Any AI model (or developer) trying to understand what commands do, what arguments they accept, or what side effects they have would need to read the source handler files. This created a discoverability gap where the architect agent could not reason about command behavior from its own prompt alone.
