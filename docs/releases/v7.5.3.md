# v7.5.3

## What changed

### Bug fix: 3-segment LM Studio model IDs incorrectly stripped

Model IDs with three path segments — such as `lmstudio/qwen/qwen3.6-35b-a3b` or
`lmstudio/qwen/qwen3-coder-next` — were being silently truncated to two segments
by the swarm agent factory.  The previous code treated **any** third segment as a
reasoning-effort variant and stripped it, so `lmstudio/qwen/qwen3.6-35b-a3b` was
rewritten to `model: "lmstudio/qwen"` with `variant: "qwen3.6-35b-a3b"`.  OpenCode
then could not find the provider+model pair and raised `ProviderModelNotFoundError`.

**Root cause:** `applyOverrides()` in `src/agents/index.ts` unconditionally auto-split
any model string with `>= 3` slash-delimited segments, regardless of whether the last
segment was a real variant token.

**Fix:** Auto-split is now gated on a `KNOWN_VARIANT_VALUES` set
(`'low'`, `'medium'`, `'high'`, `'max'`, `'xhigh'`, `'thinking'`).  For model
strings with 3+ slash-delimited segments, only these short reasoning-effort
identifiers in the final segment trigger the deprecation-split path.  Any other
final segment (model name components, version strings, etc.) is left intact.

Backward compatibility for the `provider/model/variant` shorthand is preserved: users
who still write `grove-openai/gpt-5.3-codex/medium` continue to get the deprecation
warning and the correct `model + variant` split.

**Files changed:**
- `src/agents/index.ts` — added `KNOWN_VARIANT_VALUES` set; updated `applyOverrides`
  condition from `length >= 3` to `length >= 3 && KNOWN_VARIANT_VALUES.has(lastSegment)`
- `src/config/schema.ts` — updated `variant` field comment to reflect the new semantics
- `tests/unit/agents/factory.test.ts` — added three new regression tests:
  - `preserves 3-segment lmstudio model ID intact`
  - `preserves 3-segment lmstudio model ID intact for multiple agents`
  - `still auto-splits known variant tokens: low, medium, high, max, xhigh, thinking`

## Why

LM Studio (and similar local AI providers) expose models under a
`<provider>/<namespace>/<model>` path hierarchy that OpenCode itself supports.
The swarm layer was incorrectly reducing these paths before passing them to OpenCode,
preventing any locally-hosted model with a 3-part ID from being spawned as a swarm
agent.

## Migration steps

No migration required.  Users who were previously forced to drop the `lmstudio/`
prefix as a workaround can now use the full `lmstudio/qwen/qwen3.6-35b-a3b` ID
directly in `opencode-swarm.json`.

## Known caveats

- Pre-existing test failures remain in `full-auto-intercept`, `full-auto-intercept.dispatch`,
  `help-compound-commands`, `commands/index`, `commands/issue-command`, and
  `preflight-phase` tests — these are unrelated to this change.
