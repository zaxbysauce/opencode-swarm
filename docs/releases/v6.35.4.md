# Release Notes — v6.35.4

**Patch Release | 2026-03-28**

---

## Overview

This patch release fixes the "agents output text but don't edit files" regression that was introduced when the snapshot writer's debounce timer was added in v6.33.1. When the plugin process exited before the 2-second debounce window fired, the session state was never persisted to disk. On the next startup, the state machine would start fresh, causing agents to re-run already-completed gate sequences as if no coder delegation had ever happened—producing analysis output with no file writes.

Two additional hardening improvements are included: the snapshot reader now quarantines incompatible state files (instead of silently ignoring them on every restart), and the pre-existing test suite for session round-trips is now aligned with the v6.33.1 transient-field reset behavior.

---

## Bug Fixes

### `src/session/snapshot-writer.ts` — remove debounce, write immediately

**Root cause**: `createSnapshotWriterHook` used `setTimeout(..., 2000)` to coalesce writes. If the process exited (user closes OpenCode, session ends) within 2 seconds of the last tool call, the pending write was silently dropped. On the next launch, no state file existed, so `loadSnapshot` returned early and the entire task workflow state machine (coder delegation tracking, gate progression, revision counters) was reset to defaults.

**Fix**: Removed the debounce. Every call to the hook now triggers an immediate atomic write (`writeSnapshot` → temp file + `renameSync`). Concurrent calls are serialised via a promise chain so the last writer wins without interleaving. The `flushPendingSnapshot` function is preserved for callers that require an explicit barrier (phase-complete, handoff) but now issues its own immediate write.

### `src/session/snapshot-reader.ts` — quarantine incompatible snapshot versions

**Root cause**: When `readSnapshot` encountered a snapshot with an unrecognised version number (e.g. from a downgrade or a hand-edited file), it returned `null` and left the file in place. On every subsequent startup, the file would be read again and rejected again—silently, with no indication that stale state was present.

**Fix**: When the version is not `1` or `2`, `readSnapshot` now renames `state.json` → `state.json.quarantine` before returning `null`. This moves the problematic file out of the read path so subsequent startups begin cleanly. The quarantine rename is best-effort; if the rename fails (e.g. permissions), the function still returns `null` without throwing.

### `tests/unit/session/snapshot-reader-adversarial.test.ts` — align null-session assertion with v6.33.1 fix

The test `should handle agentSessions with null session value (throws)` was written against pre-v6.33.1 code that would propagate a `TypeError` when `deserializeAgentSession` received `null`. The v6.33.1 fix added an explicit null-guard that logs a warning and skips the malformed entry. The test has been updated to assert that `rehydrateState` resolves without error and that the null session is absent from `agentSessions`.

### `tests/unit/session/snapshot-integration*.test.ts` — align version and transient-field assertions

- Updated `expect(parsed.version).toBe(1)` → `toBe(2)` across the integration and adversarial integration test files to match the version the writer has emitted since v6.33.1.
- Updated the round-trip test to expect `architectWriteCount === 0` after a reload. This field is a per-process-lifetime counter that is intentionally zeroed on restart (v6.33.1) to prevent false self-coding warnings.

---

## Migration Notes

None. All changes are backward-compatible. Users who previously worked around the "no edits" issue by deleting `.swarm/` should find that `/swarm reset-session` is now sufficient going forward—it clears session state without touching the plan, evidence, or knowledge files.
