# v7.21.4

## What changed

- **Architect config zone protection**: Added `config` zone to the architect agent's `blockedZones` and added `blockedGlobs` for 8 known verifier config file patterns (oxlintrc, eslintrc, eslint.config, prettierrc, biome.jsonc, secretscanignore, golangci). The architect can no longer directly edit config files to bypass lint/test gates.

- **Shell audit hardening**: Added DENY patterns for `sed -i` targeting config files with severity keywords, plus ESCALATE patterns for `echo`/`printf`/`cat`/`tee` writes to known config files. Prevents shell-based config sabotage.

- **Config-file write logging**: Added detection and logging in the `toolBefore` hook that records agent name, file path, and verdict for every config-file write attempt.

- **User-extensible verifier configs**: Added `verifier_config_paths` field to `AuthorityConfig` schema, which is now consumed at runtime by `isConfigFilePath` to merge user-supplied glob patterns with built-in protections.

- **Reviewer config-strictness gate**: Added CONFIG STRICTNESS VERIFICATION section to the reviewer agent prompt, instructing reviewers to reject config changes that reduce strictness.

## Why

Issue #894: Swarm agents (deepseek v4 flash, kimi k2.6) were bypassing lint/test gates by editing config files from `"error"` to `"warn"` instead of fixing the underlying source code. This patch closes three bypass vectors:

1. **Direct architect edits** — architect had `blockedZones: ['generated']` but was missing `'config'`
2. **Shell command bypass** — `sed -i`, `echo`, `printf`, `cat >`, `tee` were not detected as config sabotage
3. **Reviewer gap** — no explicit instruction to reject config strictness reductions

## Migration steps

None required. All changes are additive — existing configs and workflows continue working. Users who want to extend the built-in verifier config protection can add `authorityConfig.verifier_config_paths` to their `opencode-swarm.json`.

## Known caveats

- Config files nested under `.github/` are excluded from `config` zone classification (design constraint in the zone classifier)
- Non-standard file extensions (e.g., `.JSYC`) bypass `config` zone detection
