# v6.86.7 — Council bypass hardening

## What changed

### Tool renamed: `convene_council` → `submit_council_verdicts`

The tool was renamed to accurately describe its role. `convene_council` implied the tool
dispatches council members; it does not. The correct flow is for the architect to dispatch
each council member as a **separate Agent task first**, then call `submit_council_verdicts`
to synthesise the collected verdicts. The new name enforces that semantic at the schema level
and the updated description includes an explicit PREREQUISITE block calling out the required
prior steps and listing the anti-patterns (calling the tool before dispatching members,
passing synthesised verdicts for absent members, etc.).

### Quorum guard added to `submit_council_verdicts`

A new config field `council.minimumMembers` (integer, 1–5, default 3) sets the minimum
number of **distinct** member verdicts required before synthesis proceeds. When the threshold
is not met the tool returns `success: false, reason: "insufficient_quorum"` instead of
accepting a single-agent self-certification as valid. `requireAllMembers: true` overrides
`minimumMembers` and enforces a strict 5-member quorum (backwards-compatible — stricter
constraint wins).

### `quorumSize` threaded through the evidence chain

`CouncilSynthesis`, task-approval state (`taskCouncilApproved` Map), and on-disk evidence
files now carry a `quorumSize: number` field. This closes three bypass paths that existed
even when the quorum guard was added to the tool:

1. **In-memory fast-path** (`advanceTaskState`): re-checks quorum on every state advance.
2. **Rehydration** (`applyRehydrationCache`): legacy evidence without `quorumSize` defaults
   to `quorumSize: 1`, which fails the default `minimumMembers = 3` gate and forces a fresh
   council run after upgrade.
3. **Disk-evidence reader** (`checkCouncilGate` in `update-task-status.ts`): reads
   `quorumSize` from the evidence file and rejects completion when it is below
   `minimumMembers`, even if the stored verdict is `APPROVE` or `CONCERNS`.

### Architect prompt updated

The COUNCIL WORKFLOW section of the architect system prompt now documents the correct
5-step sequence, lists explicit anti-patterns, and references `submit_council_verdicts`.

## Why

The previous implementation had four compounding bypass paths that allowed an LLM architect
to self-certify a council APPROVE without dispatching any council member agents:

- The tool name suggested it dispatched members (it doesn't).
- No minimum-members guard existed — a single-verdict call synthesised as APPROVE.
- The quorumSize was not persisted in evidence, so rehydration could resurrect a
  single-member APPROVE as a valid gate pass after a session restart.
- The disk-evidence reader in `update-task-status.ts` checked only the `verdict` field, not
  the member count, so even correctly gated sessions were re-evaluated without quorum
  enforcement on restart.

The fix was identified via an independent review of the implementation and addressed through
all four paths simultaneously to avoid partial-fix regressions.

## Migration steps

- **No action required for new sessions.** The quorum guard applies automatically.
- **Existing evidence files** (written before this release) that lack `quorumSize` are
  treated as `quorumSize: 1`, which fails the default `minimumMembers = 3` gate. Tasks
  whose council gate was previously satisfied with fewer than 3 members will be blocked
  until a fresh council run is completed with sufficient quorum.
- **Config override:** Set `council.minimumMembers: 1` to replicate the old (no-quorum)
  behaviour, or raise it to 5 (equivalent to `requireAllMembers: true`).
- **Tool name:** Any agent or tool config that references `convene_council` by name must be
  updated to `submit_council_verdicts`. The old export name is removed.

## Breaking changes

- `convene_council` export removed; use `submit_council_verdicts`.
- `CouncilSynthesis` now requires `quorumSize: number`. Code that constructs this type
  directly (e.g. in tests) must add the field.
- `taskCouncilApproved` Map value type now includes `quorumSize: number`.

## Known caveats

- The quorum gate is enforced on the **distinct-agent** count, not on the verdict polarity.
  A council where 3 members all vote REJECT still satisfies the quorum requirement (but the
  REJECT verdict blocks the task, as before).
- `escalateOnMaxRounds` is declared in the schema for forward compatibility but has no
  runtime behaviour in this release.
