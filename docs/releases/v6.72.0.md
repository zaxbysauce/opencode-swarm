# v6.72.0

## What changed

### Knowledge system decay (N-phase TTL) + TODO stale-entry sweep

Knowledge entries (`SwarmKnowledgeEntry`, `HiveKnowledgeEntry`) now age on a
per-phase counter and are swept on every successful phase wrap. Two
complementary sweeps run in sequence against both the project-local swarm
store and the cross-project hive store:

1. **General age sweep (`sweepAgedEntries`)** — bumps `phases_alive` on every
   non-archived, non-promoted entry and flips `status` to `archived` once the
   entry reaches its TTL (`max_phases` if set, otherwise
   `default_max_phases` from config, default `10`). Archived entries remain
   on disk for audit / eventual FIFO eviction but are filtered out of
   injection.
2. **TODO sweep (`sweepStaleTodos`)** — hard-removes entries with
   `category: 'todo'` whose `phases_alive >= todo_max_phases` (default `3`).
   TODOs are intrinsically time-bound; an archived TODO has no audit value.

Promoted entries are TTL-exempt: neither aged nor archived. Archived entries
are filtered out of `readMergedKnowledge`, so they no longer surface in
context injection.

Wiring point is `phase_complete` (runs exactly once per completed phase).
Sweep failures are caught and surfaced as a `warnings` entry — never block
phase completion.

#### New entry fields (both optional)

```ts
interface KnowledgeEntryBase {
  // ... existing fields ...
  phases_alive?: number;   // monotonic counter, bumped at phase-wrap
  max_phases?: number;     // per-entry TTL override
}
```

#### New config keys (with defaults)

```jsonc
{
  "knowledge": {
    "default_max_phases": 10,   // TTL for entries without an explicit max_phases
    "todo_max_phases": 3,       // TTL for category='todo' entries
    "sweep_enabled": true        // master kill switch for the sweep
  }
}
```

#### New `'todo'` category + auto-tagging

`'todo'` is now a first-class `KnowledgeCategory`, and `inferTags()`
auto-tags lessons matching the pattern
`/(^|\s)(?:todo|remember|don't?(?:\s+)?forget)(?:\s|:|,|$)/i` so the TODO
sweep has real content to act on without manual curation.

## Why

Before this release, knowledge entries accumulated indefinitely, bounded
only by a FIFO size cap. A low-confidence candidate from 50 phases ago
could still be retrieved and injected, polluting context with advice that
no longer applied. TODO-style "remember to do X" notes were
indistinguishable from durable lessons and could not age out on a shorter
cadence. This release closes both gaps.

## Migration steps

- **No action required for most users.** All new config keys have safe
  defaults; both new entry fields are optional. Existing on-disk knowledge
  remains valid and is treated as `phases_alive: 0` on first sweep.
- To opt out of age-based sweep, set `knowledge.sweep_enabled: false` in
  `.swarm/config.json`.
- To tune TTL, set `knowledge.default_max_phases` (general entries) and/or
  `knowledge.todo_max_phases` (TODO entries).
- Override TTL per-entry by populating `max_phases` directly on a knowledge
  entry; the sweep will use the per-entry value when present.

## Breaking changes

None. Schema version was **not** bumped — both new entry fields are
optional, and old entries without them behave as if `phases_alive: 0` on
first sweep.

## Known caveats

- **Same-phase aging.** An entry written mid-phase-N is incremented to
  `phases_alive: 1` by that same phase's wrap sweep. Semantically it has
  lived zero complete phases, not one. A follow-up may tag entries with
  `created_at_phase` to skip the bump; see issue #523.
- **Corrupted JSONL line loss.** `readKnowledge` silently skips corrupted
  lines. When a sweep subsequently calls `rewriteKnowledge`, those
  unreadable lines are lost. Low risk in practice (JSONL corruption is
  rare), but noted; a follow-up recovery tool may be considered.
- **Unit tests for sweep share a global fs-mock context.** When run as part
  of the broader `tests/unit/hooks` suite, sweep tests inherit mocked path
  resolvers from sibling tests and fail with `ENOENT /mock/.swarm`. In
  isolation (`bun test tests/unit/hooks/knowledge-store-sweep.test.ts`) all
  sweep tests pass. Tracked in issue #523; a follow-up will move these to an
  integration tier.
- **Hive aging cadence under concurrency.** Non-promoted hive entries may
  age N× faster under N concurrent projects (each project's phase-wrap
  bumps age). This is acceptable today because hive entries are 100%
  promoted by design (`hive-promoter.ts`), so they are TTL-exempt.

## Additional fix

- **L4 — injector filter**: `readMergedKnowledge` now filters
  `status === 'archived'` entries, so decayed entries genuinely stop
  surfacing in context injection. Without this, the sweep would flip
  status but archived entries would still be injected — the decay story
  would be dead code from the consumer side.

## Post-review fixes (from PR #524 council review + GPT 5.4 audit)

- **C1 — Age bumps now persist.** Non-archiving age increments were writing
  to memory only; `mutated` was set exclusively on archive. Now every age
  bump triggers a rewrite so the TTL counter actually accumulates across
  phases.
- **C2 — TODO auto-category wired.** `knowledge-curator.ts` was missing
  `['todo', 'todo']` in its `categoryByTag` map, so auto-tagged TODO
  lessons were stored as `category: 'process'` and never swept. Added the
  missing mapping.
- **C3 — Promoted TODOs exempt from removal.** `sweepStaleTodos` did not
  check `status === 'promoted'`, so promoted TODO entries could be
  hard-removed in violation of the TTL-exempt design. Filter now honors
  promoted status for both sweeps.
- **C4 — Nested-lock deadlock eliminated.** Both sweep functions acquired
  the directory lock, then called `rewriteKnowledge` which re-acquired the
  same lock. `proper-lockfile` is non-reentrant, so this deadlocked on the
  5s stale timeout. Fixed by writing directly with `writeFile` under the
  already-held lock. This is the root cause of the previously reported
  "flaky sweep unit tests".
- **C5 — TTL boundary semantics.** Changed `phases_alive >= ttl` to
  `phases_alive > ttl` so `max_phases: 3` means an entry can live 3
  complete phases before archiving on the 4th bump.
- **C6 — mkdir before lock.** Sweep functions now ensure the directory
  exists before acquiring the lock, so the first sweep on a fresh hive
  install no longer crashes.
- **C7 — Config parse safety.** `KnowledgeConfigSchema.parse` in
  `phase-complete.ts` is now wrapped in try/catch so malformed user config
  doesn't block phase completion (sweep is non-blocking by design).
- **C8 — Richer sweep error context.** Generic sweep failures are now
  labelled as lock timeout / disk full / permission denied where
  applicable, easing debugging.
