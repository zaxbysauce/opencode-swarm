# v7.3.6 — Fix `mock.module` test contamination in full-auto-intercept tests

## What changed

### Root cause eliminated: `agents/critic` and `state.ts` loaded at import-time in the hook

`src/hooks/full-auto-intercept.ts` previously imported `createCriticAutonomousOversightAgent`
from `agents/critic` and `hasActiveFullAuto` / `ensureAgentSession` / `swarmState` from
`state.ts` at the **top level** of the module. This meant that any test calling
`await import('../src/hooks/full-auto-intercept.js')` would load that entire transitive
dependency tree into the Bun test process module cache.

The `full-auto-intercept.*` test files registered process-wide `mock.module` replacements for
those modules before the import. The stubs only exposed the exact exports the full-auto tests
needed — leaving dozens of exports (e.g. `createCriticAgent`, `swarmState.activeAgent`) absent.
When a subsequent test file in the **same Bun process** (e.g. `self-coding-detection.test.ts`)
tried to load those modules via a different code path (delegation-gate → guardrails →
agents/index → agents/critic), it got the stripped stub instead of the real module, causing:

- `SyntaxError: Export named 'createCriticAgent' not found in module '...agents/critic.ts'`
- `SyntaxError: Export named 'emit' not found in module '...telemetry.ts'`
- `TypeError: undefined is not an object (evaluating 'swarmState.activeAgent.set')`

This caused ~30 pre-existing failures in `self-coding-detection.test.ts` when the test suite
was run in wide scope (`bun test tests/unit/`).

## Fix

### Lazy module loading via the `_internals` DI seam

Two lazy loaders were added to `full-auto-intercept.ts`:

```ts
let _stateCache: Awaited<typeof import('../state.js')> | null = null;
async function _loadState() { … }   // loads state.ts on first production use

let _criticCache: Awaited<typeof import('../agents/critic.js')> | null = null;
async function _loadCritic() { … }  // loads agents/critic.ts on first production use
```

The `_internals` seam (introduced in prior releases for other hooks) was extended to include
`createCriticAutonomousOversightAgent` with a `null` default. All four fields now default to
`null`; production code falls back to the lazy loader when a field is null; tests set the
fields in `beforeEach` and restore them to `null` in `afterEach`.

### `mock.module` calls removed from all three test files

`tests/unit/hooks/full-auto-intercept.test.ts`,
`tests/unit/hooks/full-auto-intercept.adversarial.test.ts`, and
`tests/unit/hooks/full-auto-intercept.dispatch.test.ts` no longer call `mock.module` against
`agents/critic.js`, `delegation-gate.js`, or `plan/manager.js`. Instead:

- `_internals.createCriticAutonomousOversightAgent = mock(...)` is set in each `beforeEach`
- Restored to `_defaultCreateCriticAgent` (null) in each `afterEach`

## Why

Running `bun test tests/unit/hooks/full-auto-intercept.adversarial.test.ts tests/unit/hooks/self-coding-detection.test.ts` together previously yielded 1 error + 14 failures in the second file. After this fix the combined run produces exactly the sum of each file's individual results — zero cross-file contamination.

## Migration steps

None. This is a test-infrastructure-only fix. Production runtime behaviour is identical.

## Known caveats

The 36 pre-existing failures in the three `full-auto-intercept.*` test files are **not** resolved
by this PR — they relate to threshold/escalation logic that requires a separate fix. This PR
only resolves the ~30 failures caused by mock.module contamination spreading to other test files.
