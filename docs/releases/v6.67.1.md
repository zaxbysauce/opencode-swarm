# v6.67.1 — Register council tools with the plugin; harden same-class silent-failure paths

## What changed

### Primary fix
- **`src/index.ts`** now imports and registers four tools that were present in `AGENT_TOOL_MAP` but never exposed to opencode's runtime: `convene_council`, `declare_council_criteria`, `lint_spec`, `req_coverage`. Without this, the architect's system prompt would instruct the model to call tools that opencode never registered, and the model's attempt would error as "unknown tool" — silently breaking the council workflow even when `council.enabled: true` in config.

### Hardenings against the same bug class
- **Custom architect prompt override** (`src/agents/architect.ts`): when a user supplies `~/.config/opencode/prompts/architect.md`, the custom prompt previously replaced `ARCHITECT_PROMPT` wholesale and the later `{{COUNCIL_WORKFLOW}}` placeholder substitution became a silent no-op. The council instructions now append to the custom prompt when the placeholder is absent and council is enabled.
- **Tool filter conflict detection** (`src/agents/index.ts`): when `council.enabled: true` and `tool_filter.overrides.architect` is set but omits `convene_council` / `declare_council_criteria`, `getAgentConfigs` now throws a clear configuration-conflict error. The user must resolve the conflict explicitly (disable council, remove the override, or include the council tools). This refuses to silently override explicit user intent while also refusing to silently lose the feature.
- **Tool doctor severity promotion** (`src/services/tool-doctor.ts`): the `AGENT_TOOL_MAP` alignment finding (the exact bug class that shipped in 6.66.0) is now severity `error` instead of `warn`.
- **Tool doctor `BLOCKING` footer** (`src/commands/doctor.ts`): the rendered tool doctor report now includes a grep-able `**BLOCKING**` footer whenever any error-severity finding is present, giving release tooling a machine-readable signal.

### Tests
- `tests/unit/services/plugin-tool-registration.test.ts` — iterates every `AGENT_TOOL_MAP` tool and asserts it is registered in `src/index.ts`. Fails loudly on the original bug pattern.
- `tests/unit/agents/architect-council-hardening.test.ts` — covers default, custom-without-placeholder, disabled, and append-prompt paths.
- `tests/unit/agents/tool-filter-council-hardening.test.ts` — covers the conflict throw (bare and swarm-prefixed architect), the council-disabled respect-override case, and the no-override baseline.
- `tests/unit/commands/doctor-tools-blocking.test.ts` — covers the `BLOCKING` footer presence on errors and absence on warn-only and clean runs.
- `src/services/tool-doctor.test.ts` — updated to assert `error` severity for the alignment finding.

## Why

GLM 5.1 running 6.66.0 reported that the council feature did not activate even with `council.enabled: true` in the user's config. End-to-end trace identified that the council tools (`convene_council`, `declare_council_criteria`) — along with `lint_spec` and `req_coverage` — were imported-and-used in `AGENT_TOOL_MAP` but never registered in the plugin's `tool: { }` block, so opencode never exposed them to the runtime. The architect prompt pipeline (`loadPluginConfig` → `createAgents` → `createArchitectAgent` → `buildCouncilWorkflow`) was correct; the gap was a missing two-line registration in `src/index.ts`.

While tracing, two additional silent-failure paths were identified (custom prompt override, tool filter override), and the existing self-diagnostic (`tool-doctor.ts`) reported this exact class of bug only as a warning. All three are addressed so the feature cannot silently regress again.

## Migration

No migration required. Existing configurations with `council.enabled: true` begin functioning immediately after upgrade.

**One new strict behavior**: if a user has BOTH `council.enabled: true` AND a `tool_filter.overrides.architect` list that omits the two council tools, the plugin now throws at startup with a clear configuration-conflict error. Users in this state must either:
- Disable council (`council.enabled: false`), or
- Remove the architect override entirely (falling back on `AGENT_TOOL_MAP`), or
- Add `convene_council` and `declare_council_criteria` to the override list.

This replaces a pre-upgrade silent failure where council instructions were rendered into the prompt but the tools were stripped from the tool list.

## Breaking changes

None at the feature-API level. The new throw on `tool_filter.overrides.architect` + `council.enabled` conflict is strictly better than the prior silent-failure behavior and only fires on genuinely contradictory configurations.

## Known caveats

- The `BLOCKING` footer in the tool-doctor report is advisory — this release surfaces it as a grep-able marker but does not yet wire it to a non-zero exit code. Release automation that wants hard enforcement can grep the doctor output for `**BLOCKING**`.
- The hardenings are defensive against silent-failure classes; they do not change the observable behavior of any currently-working configuration.
