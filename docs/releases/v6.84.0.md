# OpenCode Swarm v6.84.0 Release Notes

## Summary

Adds **General Council Mode** ŌĆö a multi-model deliberation system for answering questions and reviewing plans. Independent web search per member + one structured deliberation round + optional moderator pass. Distinct from the existing verdict-based Work Complete Council.

This release also hardens the QA gate selection step itself: a new runtime check in `save_plan` blocks plan persistence when gate selection has not occurred, three new behavioral guidance blocks in the architect prompt, and a 9th QA gate (`council_general_review`) that wires the new council into MODE: SPECIFY.

- **`/swarm council <question>`** ŌĆö convene a configurable multi-model council with independent web search per member
- **`council_general_review` QA gate** ŌĆö opt-in, runs the council on a draft spec during MODE: SPECIFY before the critic-gate
- **Two new agents** ŌĆö `council_member` (web_search-only) and `council_moderator` (synthesis-only, no tools)
- **`save_plan` runtime gate** ŌĆö `QA_GATE_SELECTION_REQUIRED` blocks plan save when context.md lacks `## Pending QA Gate Selection` AND no profile exists
- **Three new behavioral guidance blocks** ŌĆö surface anti-rationalization rules at every gate-selection site

---

## Why

Two motivations:

1. **Better planning decisions.** Single-model planning misses architectural alternatives, security considerations, and current best practices. The General Council convenes a configurable mix of models ŌĆö each with independent web search ŌĆö and surfaces both consensus and persisting disagreements so the user can decide rather than the model deciding for them. Grounded in N-Way Self-Evaluating Deliberation (NSED, arXiv:2601.16863), Mixture-of-Agents (MoA, ICLR 2025), and the ConfMAD MAINTAIN/CONCEDE/NUANCE protocol.

2. **Closing a silent skip-the-gate path.** The QA gate dialogue could previously be skipped under rationalization pressure ("the user didn't mention gates", "defaults are fine"), leaving plans without a configured profile. The new `save_plan` check + behavioral guidance blocks make this impossible to skip silently.

## What changed

### General Council Mode (opt-in, off by default)

**Configuration** ŌĆö add a `council.general` block to `opencode-swarm.json`:

```json
{
  "council": {
    "general": {
      "enabled": true,
      "searchProvider": "tavily",
      "searchApiKey": "tvly-xxxxxxxx",
      "deliberate": true,
      "moderator": true,
      "moderatorModel": "anthropic/claude-sonnet-4-6",
      "members": [
        { "memberId": "m1", "model": "anthropic/claude-opus-4-7", "role": "generalist" },
        { "memberId": "m2", "model": "openai/gpt-5", "role": "skeptic" },
        { "memberId": "m3", "model": "google/gemini-2.5-pro", "role": "domain_expert" }
      ]
    }
  }
}
```

API keys can also come from `TAVILY_API_KEY` / `BRAVE_SEARCH_API_KEY` env vars (inline `searchApiKey` takes precedence).

**Workflow** ŌĆö `/swarm council <question>` triggers:
1. **Pre-flight** ŌĆö verify `council.general.enabled: true` and that a search API key is reachable.
2. **Round 1 (parallel)** ŌĆö every configured member runs independently via `web_search`, returning a structured JSON block (response, sources, self-reported confidence 0.0ŌĆō1.0, areas of uncertainty).
3. **Synthesis** ŌĆö the architect calls `convene_general_council`, which detects disagreements (linguistic markers + claim-divergence heuristic) and computes confidence-weighted consensus (Quadratic Voting from NSED).
4. **Round 2 (only when `deliberate: true`)** ŌĆö the architect re-delegates ONLY to disputing members with the opposing position. Each declares MAINTAIN / CONCEDE / NUANCE. Sycophantic capitulation without new evidence is forbidden.
5. **Moderator pass (only when `moderator: true`)** ŌĆö the dedicated `council_moderator` agent (no tools, synthesis only) writes the final user-facing answer from the council output.

**Flags:**
- `--preset <name>` ŌĆö convene a named member group from `council.general.presets`
- `--spec-review` ŌĆö single-pass advisory mode used by the `council_general_review` QA gate

**Evidence path:** `.swarm/council/general/{ISO-timestamp}-{mode}.json` ŌĆö completely separate from the QA council's `.swarm/council/` and `.swarm/evidence/` paths.

### QA gate hardening

- New `council_general_review` gate (default OFF), making 9 total. When enabled, MODE: SPECIFY runs `convene_general_council` with `--spec-review` on the draft spec before the critic-gate, folding council input into the spec.
- New runtime check in `save_plan`: returns `QA_GATE_SELECTION_REQUIRED` when `.swarm/context.md` has no `## Pending QA Gate Selection` section AND no existing `QaGateProfile` for the plan. Allows the replanning path (existing profile = pass). Bypass for CI: `SWARM_SKIP_GATE_SELECTION=1`.
- Three new `BEHAVIORAL_GUIDANCE` blocks in the architect prompt (SPECIFY step 5b, BRAINSTORM Phase 6, PLAN inline) ŌĆö anti-rationalization rules + `MANDATORY PAUSE` blocking until gate selection is recorded.
- Pre-existing drift fix: `mutation_test` was missing from the gate list shown by `/swarm qa-gates` and from `docs/commands.md` ŌĆö now corrected as part of this release.

## Migration steps

**For existing users on v6.83.x:**

- **No action required if you don't enable `council.general`.** The General Council Mode is fully opt-in. Without `council.general.enabled: true` in `opencode-swarm.json`, behavior is unchanged.
- **`save_plan` now requires gate selection.** If your workflow calls `save_plan` programmatically without going through MODE: SPECIFY / BRAINSTORM / PLAN, you will see `QA_GATE_SELECTION_REQUIRED`. Either:
  - Write `## Pending QA Gate Selection` to `.swarm/context.md` before calling `save_plan` (the architect prompt does this automatically when invoked through normal flows), OR
  - Set `SWARM_SKIP_GATE_SELECTION=1` for CI / scripted save_plan invocations.
- **9 gates instead of 8.** The new `council_general_review` gate defaults to OFF. Existing `QaGateProfile` rows are forward-compatible ŌĆö `getProfile` spread copies all fields generically.

**To enable General Council Mode:**

1. Get an API key from [Tavily](https://tavily.com) (recommended) or [Brave Search](https://brave.com/search/api/).
2. Add a `council.general` block to `opencode-swarm.json` (see Configuration section above) ŌĆö at minimum `enabled: true`, `searchProvider`, and `searchApiKey` (or set the env var).
3. Define your members. Each member is `{ memberId, model, role, persona? }`. Roles: `generalist`, `skeptic`, `domain_expert`, `devil_advocate`, `synthesizer`.
4. (Optional) Set `moderator: true` and `moderatorModel: "<model>"` to enable the moderator pass.
5. Test with `/swarm council What database should we use for a write-heavy multi-tenant SaaS?`

## Breaking changes

None. All new behavior is gated:
- `council_general_review` gate defaults OFF
- `council.general` block is optional in the config schema
- `save_plan` gate-selection check has a `SWARM_SKIP_GATE_SELECTION=1` bypass for CI
- The architect prompt's MODE: COUNCIL section is only emitted as a runnable mode when `council.general.enabled` is true (the prompt text is always present)

## Known caveats

- **`CouncilConfigSchema` is `.strict()`.** A typo in any `council.general.*` key (e.g. `searchProvder`) fails Zod validation, and `src/config/loader.ts` falls back to **guardrail-only defaults** ŌĆö silently losing the entire `opencode-swarm.json`, not just the misspelled field. After editing, validate with `/swarm config` and watch for `[opencode-swarm] ŌÜĀ’ĖÅ SECURITY: Falling back to conservative defaults` in the console.
- **Search-API key required.** Without `council.general.searchApiKey` (or env var), `web_search` returns a structured `missing_api_key` error and the council cannot run.
- **Moderator does not fact-check.** The `council_moderator` agent is synthesis-only ŌĆö no `web_search`, no new sources. If the council's existing sources are wrong, the moderator will repeat them.
- **Disagreement detection is heuristic.** Two-pass: explicit linguistic markers ("I disagree with", "contrary to") + claim-divergence heuristic. Subtle disagreements that members do not flag explicitly may slip through.
- **`save_plan` integration test setup change.** Tests that exercise `save_plan` now must write `## Pending QA Gate Selection` to `.swarm/context.md` (or set `SWARM_SKIP_GATE_SELECTION=1`) ŌĆö the existing test files in `tests/unit/tools/save-plan*.test.ts` and `tests/integration/save-plan-round-trip.test.ts` were updated as part of this release; downstream forks that maintain their own save-plan tests should follow the same pattern.
