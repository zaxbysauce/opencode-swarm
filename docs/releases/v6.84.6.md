# v6.84.6

## Summary

- Redact OpenCode API key variants (`OPENCODE_KEY=` env vars, custom `-H` headers without `X-` prefix)
- Warn users on plugin init if `.swarm/` is not in their project's `.gitignore`

## What changed

**redactShellCommand() regex improvements** — The shell command audit log redaction patterns were expanded to catch more API key formats:
1. Environment variables ending in `_KEY` (e.g., `OPENCODE_KEY=sk-xxx`, `ANTHROPIC_KEY=sk-ant-xxx`) are now redacted. Previously only compound keywords like `API_KEY` and `ACCESS_KEY` were caught.
2. HTTP headers without the `X-` prefix are now caught (e.g., `-H "Opencode-Key: sk-xxx"`, `-H "Api-Secret: value"`). The pattern now matches any dash-separated header ending in `key`, `token`, `secret`, `auth`, or `credential`.

**Gitignore warning on plugin init** — When the swarm plugin initializes in a git repository, it now checks if `.swarm/` or `.swarm/session/` is covered by `.gitignore` or `.git/info/exclude`. If not covered, a one-time warning is emitted:
```
[opencode-swarm] WARNING: .swarm/ is not in your .gitignore. Shell audit logs may contain API keys. Add ".swarm/" to your .gitignore to prevent accidental commits.
```
This protects users whose own project `.gitignore` files have not been configured to exclude the swarm session directory.

## Why

**Issue #618** reported that the OpenCode API key was appearing unredacted in `.swarm/session/shell-audit.jsonl`. This file contains a record of shell commands executed by agents and is created in each user's working directory during plugin operation. Two gaps in the redaction logic allowed keys to leak:

1. Environment variable assignments using a bare `_KEY` suffix (e.g., `OPENCODE_KEY=value`) were not being redacted because the keyword list only included compound patterns like `API_KEY`, not the suffix `_KEY` alone.
2. Custom HTTP headers not following the `X-` prefix convention were not caught by the header redaction pattern.

Additionally, the `.swarm/` directory was being created in user projects but was not automatically added to their `.gitignore`. Users could accidentally commit audit logs containing secrets. The plugin now proactively warns them.

## Migration steps

None required. The changes are backward-compatible and do not modify any APIs, config schemas, or command syntax.

## Known caveats

- The gitignore warning fires at most once per process (module-level deduplication). In interactive sessions, if the user ignores the warning and later commits audit logs, they will not see the warning again in the same session.
- The redaction logic is conservative and may redact header values in non-credential headers that happen to end in `key`, `token`, `secret`, etc. (e.g., a custom header like `Monkey-Key: value` would be redacted). This is intentional to avoid false negatives in security-sensitive audit logs.
