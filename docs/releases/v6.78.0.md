# v6.78.0 — Lockable Execution Profile & Fail-Closed Parallelization Enforcement

## What Changed

**New `execution_profile` schema for plan-scoped concurrency controls**
- Architect can now set `parallelization_enabled`, `max_concurrent_tasks`, `council_parallel`, and `locked` fields on any plan
- Profile is **lockable**: once `locked: true`, no subsequent `save_plan` call can modify it (fail-closed enforcement)
- Defaults: `parallelization_enabled: false`, `max_concurrent_tasks: 1`, `council_parallel: false`, `locked: false`

**Ledger durability for execution profiles**
- New ledger event types: `execution_profile_set` and `execution_profile_locked`
- All profile changes are durable, immutable records in `.swarm/plan-ledger.jsonl`
- Profile is deterministically replayed from ledger during plan rebuild

**Fail-closed delegation gate enforcement**
- Stage B parallel dispatch now reads the locked plan's `execution_profile`
- If profile is locked and `parallelization_enabled: false`, parallel dispatch is blocked (sequential only)
- If plan cannot be loaded when parallel is enabled, dispatch is blocked (fail-closed, not defaulting)

**Complete round-trip coverage**
- `execution_profile` persists in `plan.json`, `plan-ledger.jsonl`, checkpoint exports, snapshot events, handoff data, and export data
- `get_approved_plan` tool exposes the critic-approved plan's profile
- Architect and critic prompts updated with profile guidance

## Why

This release completes the stacked parallelization architecture (PR 1, PR 2, PR 3):
- **PR 1**: Dark foundation for locking and evidence accumulation
- **PR 2**: Stage B barrier and bounded concurrent task dispatch
- **PR 3**: Architect-facing concurrency controls tied to plan durability and ledger

The lockable profile ensures that once an architect has set parallelization parameters, they remain stable across the plan lifecycle — no accidental or silent drift during critic review or task execution.

## Migration Steps

**No action required.** The `execution_profile` field is optional. Existing plans continue to work unchanged. New plans can optionally set `execution_profile` via `save_plan`:

```
save_plan with:
  execution_profile: {
    parallelization_enabled: true,
    max_concurrent_tasks: 4,
    locked: false
  }
```

To lock a profile and prevent further modification:

```
save_plan with:
  execution_profile: {
    locked: true
  }
```

Once locked, any `save_plan` call that includes `execution_profile` will be rejected. To unlock, use `reset_statuses: true` in `save_plan` to start the plan fresh.

## Breaking Changes

None. The field is fully optional.

## Known Caveats

- **Locked profile cannot be modified in-place**: to change a locked profile, use `reset_statuses: true` in `save_plan` to start a fresh plan revision. This clears all task statuses and the lock.
- **Delegation gate enforcement is strict**: if a plan is locked with `parallelization_enabled: false` but the global config enables Stage B parallel, the gate blocks dispatch. Plan-scoped intent wins.
- **Ledger malformed profile events are skipped**: if a corrupted ledger entry contains an invalid profile, replay ignores it and continues (does not corrupt state). The bad entry is quarantined to `.swarm/plan-ledger.quarantine`.
