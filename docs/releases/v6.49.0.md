### Laravel baseline detection

Laravel is detected using 2-of-3 signals (prevents false positives against generic Composer projects):
1. `artisan` file in project root
2. `laravel/framework` in `composer.json` `require` (not `require-dev`)
3. `config/app.php` present

When detected, all commands are automatically overridden:
- **Test**: `php artisan test` (wraps both PHPUnit and Pest; supports `--parallel`)
- **Lint**: Pint when `pint.json` present, PHP-CS-Fixer otherwise
- **Static analysis**: `vendor/bin/phpstan analyse` when `phpstan.neon` or `phpstan.neon.dist` present
- **Audit**: `composer audit --locked --format=json`

`.blade.php` files are now included in placeholder scanning, TODO extraction, and SAST scanning. The `test_engineer` agent receives Laravel-specific guidance via `testConstraints` injection (feature tests, Pest/PHPUnit coexistence, `.env.testing`, `config:clear`, parallel DB isolation).

PHP powers approximately 77% of server-side web applications and is the language behind WordPress, Drupal, and Laravel. Issue #308 tracked community demand for PHP support since v6.0. This release brings PHP to the same quality tier as Go, Java, and Kotlin G�� full build/test/lint/audit pipeline with framework-specific detection for the ecosystem's dominant framework.

# Knowledge System + Plan Sync Overhaul

Three interconnected failure modes — silent knowledge injection skip in long sessions, a curator that could never act on entries it couldn't see, and a plan sync pipeline that wiped completed task statuses on every `save_plan` call — have been fixed at the root cause level.

## What changed

### [HIGH] Knowledge injector: budget-aware injection replaces binary context skip

**Problem:** The injector's static `totalChars > 75_000` threshold was calibrated for char-width models and caused it to silently skip injection at 181k chars (47% of the 387k-char model limit). Long sessions — the ones that benefit most from knowledge injection — received zero knowledge signal.

**Fix:** Replaced the binary skip with a three-regime headroom model:

- **High regime** (>60% headroom remaining): full `inject_char_budget` (default 2,000 chars)
- **Moderate regime** (20–60% headroom): half budget (~1,000 chars)
- **Low regime** (<20% headroom): quarter budget (~500 chars)
- **Skip only** when headroom falls below 300 chars (near model limit)

The headroom calculation uses the same `0.33 tok/char` ratio as `context-budget.ts` and targets 128k tokens (~387,878 chars) as the model limit.

**Config additions** (`src/config/schema.ts`):
- `inject_char_budget` — total char cap for the injected block (default: 2,000, range: 200–10,000)
- `max_lesson_display_chars` — per-lesson display truncation (default: 120, range: 40–280; stored lesson is never modified)

**Compact format:** The injected block now uses `[S]`/`[H]` tier markers and `✓`/`✓✓` confirmation indicators instead of verbose star ratings. Star ratings (`★`) have been removed entirely. The block is assembled in priority order (lessons → run memory → drift preamble → rejected warnings) and shed lowest-priority components when budget is tight.

**Recency position:** Knowledge is now injected immediately before the last user message, avoiding the "lost in the middle" attention dead zone that mid-array system injection creates.

Files changed: `src/config/schema.ts`, `src/hooks/knowledge-injector.ts`

### [HIGH] Curator: store-aware pruning and rewrite actions

**Problem:** The curator LLM was told to always use `"new"` as the token for all recommendations, so `archive` and `flag_contradiction` actions always produced `entry_id: undefined` and were silently skipped. The knowledge store grew indefinitely with no mechanism to prune stale or verbose entries.

**Fix:**
- `runCuratorPhase` and `runCuratorInit` now pass the current knowledge store (capped at 30 entries, sorted by recency) as `KNOWLEDGE_ENTRIES` JSON to the LLM, enabling it to reference real UUIDs.
- `CURATOR_PHASE_PROMPT` and `CURATOR_INIT_PROMPT` updated: "always use `new`" instruction removed; `rewrite <uuid>: <new lesson text>` added to the action list.
- New `rewrite` action in `KnowledgeRecommendation`: replaces a lesson's text in-place while preserving all metadata. Enforces 15–280 char bounds; reduces confidence by 0.05 on rewrite to flag it for re-validation.

Files changed: `src/hooks/curator-types.ts`, `src/hooks/curator.ts`, `src/agents/explorer.ts`

### [HIGH] Plan sync: merge-mode `save_plan` + ledger rebuild protection

**Problem (six root causes):**
1. `executeSavePlan` reset all task statuses to `pending` on every call.
2. `loadPlan`'s ledger hash-mismatch rebuild fired on every `loadPlan()` call, not just startup — a concurrent write could overwrite the architect's current work.
3. `replayFromLedger` without a snapshot only replayed `task_status_changed` events; structural plan changes were lost during rebuild.
4. Two competing hash functions could disagree, causing spurious rebuild triggers.
5. The completion gate's evidence files were only written if `subagent_type` fired correctly, leaving the system stateless across restarts.
6. The sync worker fired on temp file events from atomic renames.

**Fixes:**

- **Merge-mode `save_plan`** (`src/tools/save-plan.ts`): reads the existing `plan.json` before writing and builds a status map; all existing task statuses are preserved across plan revisions. New tasks default to `pending`. A ledger snapshot is taken after every `save_plan` call so `replayFromLedger` always has a complete baseline.

- **Startup-only ledger rebuild** (`src/plan/manager.ts`): the hash-mismatch rebuild is now gated by a per-workspace `startupLedgerCheckedWorkspaces` Set and fires at most once per process lifetime. Subsequent calls with a hash mismatch are ignored (plan.json is authoritative during active sessions). `resetStartupLedgerCheck()` exported for test isolation.

- **Durable gate evidence** (`src/tools/update-task-status.ts`): a seed evidence file is written atomically (`wx` flag) when a task transitions to `in_progress`, ensuring the evidence system has a record even if the `toolAfter` hook's `subagent_type` never fires. This makes `checkReviewerGate`'s evidence-first path viable after session restarts.

- **Sync worker hardening** (`src/background/plan-sync-worker.ts`): temp file events (`.tmp.*`, `.rebuild`) are filtered before triggering a debounced sync; plans with zero phases are not rendered to `plan.md`; default debounce extended from 300ms to 500ms.

Files changed: `src/tools/save-plan.ts`, `src/plan/manager.ts`, `src/tools/update-task-status.ts`, `src/background/plan-sync-worker.ts`

## Why

These three systems compound each other in production: long sessions trigger the injector skip, the curator never tightens the growing knowledge store because it cannot act on entries it cannot see, and a single `save_plan` revision wipes all completed statuses — forcing the architect to re-mark completed tasks or accept a stale plan context. The fixes address root causes, not symptoms.
