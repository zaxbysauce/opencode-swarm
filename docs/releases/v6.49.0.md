# v6.49.0 — PHP First-Class Support + Laravel Baseline

## What changed

### PHP as a first-class language (12th in the registry)

PHP joins the profile registry at Tier 3 with a complete build/test/lint/audit pipeline. Closes GitHub Issue #308.

**Build**
- Composer install (`composer install --no-interaction --prefer-dist`) detected from `composer.json`
- `php-composer` ecosystem registered in build discovery alongside Node, Python, Rust, Go, etc.

**Test framework detection** (priority order)
- Pest (priority 1) — detected via `Pest.php`
- PHP artisan test (priority 2) — when Laravel is detected (see below)
- PHPUnit (priority 3/4) — detected via `phpunit.xml` or `phpunit.xml.dist`

**Lint and static analysis**
- PHPStan (priority 1) — via `phpstan.neon`; Larastan-compatible when configured
- PHPStan (priority 2) — via `phpstan.neon.dist`
- Pint (priority 3) — detected via `pint.json`, runs `vendor/bin/pint --test`
- PHP-CS-Fixer (priority 4) — detected via `.php-cs-fixer.php`, fallback formatter

**Dependency audit**
- `composer audit --locked --format=json` wired into the `pkg_audit` pipeline
- Exit code semantics: 0 = clean, 1 = abandoned packages only (not a security failure), 2 = vulnerabilities
- Structured JSON output with per-package CVE findings

**SAST rules** — 13 native PHP rules total (10 pre-existing + 3 new Laravel-specific):
- `sast/php-laravel-sql-injection` (high): `DB::raw`/`whereRaw` with unparameterized string concatenation
- `sast/php-laravel-mass-assignment` (medium): empty `$guarded = []` disabling all mass-assignment protection
- `sast/php-laravel-destructive-migration` (medium): `Schema::drop`/`dropIfExists` without a `down()` method

### Laravel baseline detection

Laravel is detected using 2-of-3 signals (prevents false positives against generic Composer projects):
1. `artisan` file in project root
2. `laravel/framework` in `composer.json` `require` (not `require-dev`)
3. `config/app.php` present

When detected, all commands are automatically overridden:
- **Test**: `php artisan test` (wraps both PHPUnit and Pest; supports `--parallel`)
- **Lint**: Pint when `pint.json` present, PHP-CS-Fixer otherwise
- **Static analysis**: `vendor/bin/phpstan analyse` when `phpstan.neon` or `phpstan.neon.dist` present
- **Audit**: `composer audit --locked --format=json`

`.blade.php` files are now included in placeholder scanning, TODO extraction, and SAST scanning. The `test_engineer` agent receives Laravel-specific guidance via `testConstraints` injection (feature tests, Pest/PHPUnit coexistence, `.env.testing`, `config:clear`, parallel DB isolation).

### CI coverage

A new `php-validation` CI job validates PHP/Laravel support on every push:
- PHP 8.2 via `shivammathur/setup-php@accd6127cb78bee3e8082180cb391013d204ef9f # v2`
- Runs profile command surface, framework detection, Composer audit, Laravel SAST rule, and fixture-backed tests
- `smoke` job now depends on `php-validation`

### Intentionally deferred (not in this release)

Artisan scaffolding, Livewire/Inertia detection, Symfony/Slim support, Laravel queue/event systems, deep Eloquent ORM modeling, multi-tenant architecture awareness.

## Why

PHP powers approximately 77% of server-side web applications and is the language behind WordPress, Drupal, and Laravel. Issue #308 tracked community demand for PHP support since v6.0. This release brings PHP to the same quality tier as Go, Java, and Kotlin — full build/test/lint/audit pipeline with framework-specific detection for the ecosystem's dominant framework.

The 2-of-3 detection threshold for Laravel and the distinct exit-code handling for `composer audit` (abandoned ≠ vulnerability) are deliberate design choices to minimize false positives and false failures in CI environments.
