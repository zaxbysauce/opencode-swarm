# v6.42.1 — Register Curator as Named Swarm Agent with Session-Aware Resolution

## What changed

### Core fix: curator as named swarm agent pair

- **`src/agents/curator-agent.ts`** (new): `createCuratorAgent()` factory following the
  same pattern as `createCriticAgent`. Two roles: `curator_init` and `curator_phase`, each
  carrying its own baked-in system prompt (`CURATOR_INIT_PROMPT` / `CURATOR_PHASE_PROMPT`
  now exported from `explorer.ts`). Read-only tool config (`write: false, edit: false, patch: false`).

- **`src/agents/explorer.ts`**: Exported `CURATOR_INIT_PROMPT` and `CURATOR_PHASE_PROMPT`
  (previously unexported `const`) so `curator-agent.ts` can import them without duplication.
  `curator.ts` now imports them directly — removes the stale `createExplorerCuratorAgent`
  call that was used solely to extract the prompt string.

- **`src/config/constants.ts`**: Added `'curator_init'` and `'curator_phase'` to
  `ALL_SUBAGENT_NAMES`, `AGENT_TOOL_MAP` (read-only, `knowledgeRecall` only), and
  `DEFAULT_MODELS` (`opencode/trinity-large-preview-free` — same lightweight model as
  explorer/critic).

- **`src/agents/index.ts`**: Registered both curator agents in `createSwarmAgents()` following
  the critic variant pattern (steps 5d and 5e). Both support the `getPrompts()` custom prompt
  override and `isAgentDisabled()` check. Names are prefixed for non-default swarms exactly
  like all other agents (e.g. `swarm1_curator_init`, `swarm1_curator_phase`).

### Session-aware multi-swarm resolution

- **`src/state.ts`**: Added `curatorInitAgentNames: [] as string[]` and
  `curatorPhaseAgentNames: [] as string[]` to `swarmState`. Both arrays are populated at
  plugin init with ALL registered curator names across all swarms (not just the first one).
  Reset in `resetSwarmState()`.

- **`src/index.ts`**: After `getAgentConfigs()` builds the agent map, collects all matching
  curator agent names into the arrays using `filter()`. Both `createPhaseMonitorHook` call
  sites pass `mode: 'init'`.

- **`src/hooks/curator-llm-factory.ts`**: Completely rewritten resolution logic. Factory
  accepts an optional `sessionId` parameter. Resolution happens lazily at delegate call time
  (not factory creation time) via `resolveCuratorAgentName(mode, sessionId)`:
  1. **Direct lookup** (when `sessionId` provided): looks up `swarmState.activeAgent.get(sessionId)`,
     extracts the swarm prefix by stripping the base agent name suffix, finds the matching
     curator — deterministic even when multiple named-swarm sessions run in parallel.
     When the calling agent has no named-swarm prefix (default swarm), explicitly returns
     `prefixMap.get('')` rather than falling through to heuristic scan — prevents a default-swarm
     session from being incorrectly assigned a named-swarm curator.
  2. **Heuristic scan** (fallback when no `sessionId`): iterates `activeAgent.values()` and
     finds the longest registered prefix that matches any active agent name. Longest-match
     avoids prefix collisions (e.g. `alpha_extended_` beats `alpha_` for `alpha_extended_architect`).
  3. **Static fallback**: default-swarm curator (empty prefix), then first registered name,
     then bare suffix string.
  Also: `system:` field removed from `session.prompt` body — the registered agent's baked-in
  prompt (including any user-configured custom prompt overrides) is used instead. The
  `_systemPrompt` parameter is now accepted but intentionally ignored by the factory.

- **`src/tools/phase-complete.ts`**: Passes `sessionID` to the delegate factory.

- **`src/tools/curator-analyze.ts`**: Added `ctx?: ToolContext` to `execute` signature;
  passes `ctx?.sessionID` to the delegate factory.

- **`src/hooks/phase-monitor.ts`**: Changed 4th parameter from `llmDelegate?: CuratorLLMDelegate`
  (pre-built delegate, no session context) to `delegateFactory?: CuratorDelegateFactory`
  (called lazily inside the handler with the `sessionID` extracted from `input`). A new
  `CuratorDelegateFactory` type alias is exported: `(sessionId?: string) => CuratorLLMDelegate | undefined`.
  Existing test injection callers pass `() => myDelegate` to preserve the original behavior.

- **`src/index.ts`**: Phase-monitor call sites updated from passing pre-built delegate
  `createCuratorLLMDelegate(ctx.directory, 'init')` to a factory lambda
  `(sessionId) => createCuratorLLMDelegate(ctx.directory, 'init', sessionId)`,
  ensuring each hook invocation resolves the correct swarm's curator using the live session ID.

## Why

On first run under v6.42.0 the TUI crashed with:

```
TypeError: undefined is not an object (evaluating 'agent.name')
    at createUserMessage (src/session/prompt.ts:832:14)
```

Root cause: `curator-llm-factory.ts` passed `agent: 'explorer'` to `session.prompt`. For
non-default swarms (e.g. `swarmId: 'local'`) the explorer agent is registered as
`local_explorer`, so `agents.get('explorer')` returns `undefined`. The `createUserMessage`
guard persists the unknown agent name and on the next `runLoop` iteration `agent.name`
dereferences `undefined`.

The earlier stopgap (removing the `agent:` field entirely) was rejected because it would
fall through to whatever the server assigns as the default agent — potentially an expensive
general/opus agent — defeating the purpose of using a cheap, lightweight analysis agent.

The correct fix registers `curator_init` and `curator_phase` as proper named swarm agents
(following the `critic` / `critic_sounding_board` / `critic_drift_verifier` pattern). The
plugin collects all registered curator names from the agent map at init time, and the factory
resolves the correct swarm's curator lazily at each call using the calling session's ID.

## Migration steps

None. The change is internal to the plugin. New agents are enabled by default (same as
`critic_sounding_board` and `critic_drift_verifier`). Users can disable them via:

```json
{ "agents": { "curator_init": { "disabled": true }, "curator_phase": { "disabled": true } } }
```

When disabled, no curator names are registered, the arrays are empty, and the factory falls
back to bare `curator_init` / `curator_phase` — `session.prompt` fails → curator degrades
to data-only mode (same graceful fallback path as before).

## Breaking changes

None. The `CuratorLLMDelegate` type signature is unchanged. The optional `sessionId`
parameter added to `createCuratorLLMDelegate` is backward-compatible.

## Multi-swarm behavior

For any number of custom non-default swarms:
- All curator names are registered and stored in `curatorInitAgentNames[]` / `curatorPhaseAgentNames[]`
- Prefix map is built from the registered names: `'swarm1_curator_init'` → prefix `'swarm1_'`
- When `phase_complete` fires from `swarm3_architect`'s session, the factory uses
  `ctx.sessionID` to look up `swarm3_architect`, extracts prefix `'swarm3_'`, and returns
  `'swarm3_curator_init'` — correct regardless of config ordering or parallel sessions
- For `phase-monitor` (no session context at init): heuristic scan resolves correctly since
  only one swarm's architect is active at session initialization
