# v6.42.1 — Fix Curator LLM Delegation Crash

## What changed

- **`src/hooks/curator-llm-factory.ts`**: Removed the `agent: 'explorer'` field from the
  `session.prompt` body in `createCuratorLLMDelegate`. A comment now documents the reason so the
  field is not reintroduced accidentally.
- **`tests/unit/hooks/curator-llm-factory.test.ts`**: Updated the `toHaveBeenCalledWith` assertion
  to match the corrected body shape (no `agent` field).

## Why

On the first run under v6.42.0 the TUI crashed with:

```
TypeError: undefined is not an object (evaluating 'agent.name')
    at createUserMessage (src/session/prompt.ts:832:14)
```

The curator LLM delegate passed `agent: 'explorer'` (a string) to `session.prompt`. The OpenCode
server receives the string, looks up the corresponding `Agent` object by name, then accesses
`agent.name` on the result. In a freshly-created ephemeral session the lookup returns `undefined`,
causing the crash.

The fix removes the `agent` field entirely. The custom `system` prompt passed on every call already
controls curator behavior; no named agent is needed. When `agent` is absent the server uses its
default handling and the undefined-lookup code path is never reached.

The plugin's graceful-fallback logic (`LM delegation failed → data-only mode`) was working
correctly but the server crash still surfaced in the TUI.

## Migration steps

None. The change is internal to the plugin; no config, API surface, or file format is affected.

## Breaking changes

None.

## Known caveats

The ephemeral session will now run under whatever default agent the OpenCode server assigns to a
new session rather than the `explorer` agent specifically. Curator analysis is driven entirely by
the `system` prompt override, so behavior is functionally identical.
