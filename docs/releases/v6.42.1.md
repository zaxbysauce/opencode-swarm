# v6.42.1 — Register Curator as Named Swarm Agent

## What changed

- **`src/agents/curator-agent.ts`** (new): `createCuratorAgent()` factory following the
  same pattern as `createCriticAgent`. Two roles: `curator_init` and `curator_phase`, each
  carrying its own baked-in system prompt (`CURATOR_INIT_PROMPT` / `CURATOR_PHASE_PROMPT`
  from `explorer.ts`). Read-only tool config (`write: false, edit: false, patch: false`).

- **`src/agents/explorer.ts`**: Exported `CURATOR_INIT_PROMPT` and `CURATOR_PHASE_PROMPT`
  (previously unexported `const`) so `curator-agent.ts` can import them without duplication.

- **`src/config/constants.ts`**: Added `'curator_init'` and `'curator_phase'` to
  `ALL_SUBAGENT_NAMES`, `AGENT_TOOL_MAP` (read-only, `knowledgeRecall` only), and
  `DEFAULT_MODELS` (`opencode/trinity-large-preview-free` — same lightweight model as
  explorer/critic).

- **`src/agents/index.ts`**: Registered both curator agents in `createSwarmAgents()` following
  the critic variant pattern (steps 5d and 5e). Both support the `getPrompts()` custom prompt
  override and `isAgentDisabled()` check. Names are prefixed for non-default swarms exactly
  like all other agents (e.g. `local_curator_init`, `local_curator_phase`).

- **`src/state.ts`**: Added `curatorInitAgentName` and `curatorPhaseAgentName` (both `null`
  initially) to `swarmState`. Reset in `resetSwarmState()`.

- **`src/index.ts`**: After `getAgentConfigs()` builds the agent map, resolves the
  registered curator names (including any swarm prefix) into `swarmState`. Both
  `createPhaseMonitorHook` call sites now pass `mode: 'init'`.

- **`src/hooks/curator-llm-factory.ts`**: Added `mode: 'init' | 'phase'` parameter
  (default `'init'`). Uses `swarmState.curatorInitAgentName` or
  `swarmState.curatorPhaseAgentName` as the `agent:` field in `session.prompt`, with a
  bare-name fallback if state was not populated. The `system:` override continues to carry
  the mode-specific assembled context from `runCuratorInit` / `runCuratorPhase`.

- **`src/tools/phase-complete.ts`** and **`src/tools/curator-analyze.ts`**: Updated
  `createCuratorLLMDelegate` calls to pass `'phase'`.

- **`tests/unit/hooks/curator-llm-factory.test.ts`**: Updated and extended — added tests
  for init/phase mode dispatch, prefixed agent names, and bare-name fallback.

## Why

On first run under v6.42.0 the TUI crashed with:

```
TypeError: undefined is not an object (evaluating 'agent.name')
    at createUserMessage (src/session/prompt.ts:832:14)
```

Root cause: `curator-llm-factory.ts` passed `agent: 'explorer'` to `session.prompt`. For
non-default swarms (e.g. `swarmId: 'local'`) the explorer agent is registered as
`local_explorer`, so `agents.get('explorer')` returns `undefined`. The `createUserMessage`
guard persists the unknown agent name and on the next `runLoop` iteration `agent.name`
dereferences `undefined`.

Previous stopgap (removing the `agent:` field entirely) was rejected because it would fall
through to whatever the server assigns as the default agent — potentially the expensive
general/opus agent — defeating the purpose of using a cheap, lightweight analysis agent.

The correct fix is to register `curator_init` and `curator_phase` as proper named swarm
agents (following the `critic` / `critic_sounding_board` / `critic_drift_verifier` pattern).
The plugin then looks up the registered prefixed names from the agent map at init time, stores
them in `swarmState`, and the factory uses those names at call time — guaranteed to resolve
correctly in any swarm configuration.

## Migration steps

None. The change is internal to the plugin. New agents are enabled by default (same as
`critic_sounding_board` and `critic_drift_verifier`). Users can disable them via:

```json
{ "agents": { "curator_init": { "disabled": true }, "curator_phase": { "disabled": true } } }
```

When disabled, the factory receives `null` names and falls back to bare `curator_init` /
`curator_phase` strings; since those names won't be registered, `session.prompt` will fail
and the curator will degrade to data-only mode (same graceful fallback path as before).

## Breaking changes

None. The `CuratorLLMDelegate` type signature is unchanged. The `mode` parameter added to
`createCuratorLLMDelegate` defaults to `'init'` for backward compatibility.

## Known caveats

None. `curator_init` and `curator_phase` are proper subagents using the same lightweight
model as `explorer` and `critic` variants. Their tool permissions are minimal (read-only,
`knowledgeRecall` only) consistent with the original design intent.
