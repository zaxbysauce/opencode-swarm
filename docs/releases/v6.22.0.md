# v6.22.0 Release Notes

**Release Date:** 2026-03-09

## New Features

### Curator Background Analysis System

A new optional background analysis system that provides phase-level intelligence across the project lifecycle. Disabled by default — set `curator.enabled = true` in `.opencode/opencode-swarm.json` to activate.

#### CuratorConfig Schema (`src/config/schema.ts`)

New `CuratorConfigSchema` with the following fields and defaults:

| Field | Default | Effect |
|-------|---------|--------|
| `enabled` | `false` | Master switch — must be `true` for any Curator activity |
| `init_enabled` | `true` | Run curator init on first phase |
| `phase_enabled` | `true` | Run phase analysis + drift check after each phase |
| `max_summary_tokens` | `2000` | Cap on curator summary size |
| `min_knowledge_confidence` | `0.7` | Minimum confidence for knowledge entry inclusion |
| `compliance_report` | `true` | Include compliance report in phase result |
| `suppress_warnings` | `true` | Suppress non-critical curator warnings |
| `drift_inject_max_chars` | `500` | Max chars of drift text injected into architect context |

**Impact:** LOW — disabled by default, zero effect on existing behavior

#### Curator Agent Factories (`src/agents/explorer.ts`, `src/agents/critic.ts`)

Two new internal agent factories:
- `createCuratorExplorerAgent` — builds an explorer agent configured for curator summarization
- `createDriftCriticAgent` — builds a critic agent configured for drift detection

**Impact:** LOW — internal factories, no public API change

#### Curator Core (`src/hooks/curator.ts`)

Seven new exported functions:
- `runCuratorInit` — initializes `.swarm/curator-summary.json` on first phase
- `runCuratorPhase` — collects phase events and compliance results (data preparer, no LLM calls)
- `applyCuratorKnowledgeUpdates` — merges knowledge recommendations into curator summary, capped at `max_summary_tokens`
- `readCuratorSummary` / `writeCuratorSummary` — read/write `.swarm/curator-summary.json` via `Bun.file`
- `filterPhaseEvents` — filters event-bus events by phase
- `checkPhaseCompliance` — validates phase gate requirements against events

**Impact:** MEDIUM — new capability, no changes to existing code paths

#### Curator Drift Detection (`src/hooks/curator-drift.ts`)

Four new exported functions:
- `runCriticDriftCheck` — compares planned vs. actual decisions, writes a drift report to `.swarm/drift-report-phase-N.json`
- `buildDriftInjectionText` — formats prior drift reports into markdown for knowledge-injector consumers
- `readPriorDriftReports` — reads all drift reports from `.swarm/`, returns array sorted by phase descending
- `writeDriftReport` — writes a `DriftReport` to `.swarm/drift-report-phase-N.json`

Drift report format (full `DriftReport` interface):
```typescript
interface DriftReport {
  schema_version: 1;
  phase: number;
  timestamp: string;          // ISO 8601
  alignment: 'ALIGNED' | 'MINOR_DRIFT' | 'MAJOR_DRIFT' | 'OFF_SPEC';
  drift_score: number;        // 0.0 (aligned) to 1.0 (completely off-spec)
  first_deviation: { phase: number; task: string; description: string } | null;
  compounding_effects: string[];
  corrections: string[];
  requirements_checked: number;
  requirements_satisfied: number;
  scope_additions: string[];
  injection_summary: string;
}
```

**Impact:** MEDIUM — new capability, drift reports written to `.swarm/` directory

#### Curator Integration Points

Three integration points wired in this release:

**1. Phase Monitor Init (`src/hooks/phase-monitor.ts`)**
On the first phase of a project, `runCuratorInit` is called (gated on `enabled && init_enabled`, wrapped in try/catch).

**2. Phase Complete Pipeline (`src/tools/phase-complete.ts`)**
After each `curateAndStoreSwarm()` call, the Curator pipeline runs:
`runCuratorPhase` → `applyCuratorKnowledgeUpdates` → `runCriticDriftCheck`
Entire pipeline gated on `enabled && phase_enabled`, wrapped in a single try/catch. `phase_complete` is never blocked.

**3. Knowledge Injector Drift Injection (`src/hooks/knowledge-injector.ts`)**
At the start of each phase, the latest drift report summary is **prepended** to the architect's knowledge context ahead of standard knowledge entries. Wrapped in try/catch — injection never blocks.

**Impact:** HIGH (when enabled) — provides live plan-vs-reality drift feedback to the architect. Zero impact when disabled (default).

## Bug Fixes

### Issue #81 v2: taskWorkflowStates Snapshot Persistence

Session snapshots now correctly persist and restore the task workflow state machine.

#### Changes

- **`src/session/snapshot-writer.ts`**: Added `taskWorkflowStates?: Record<string, string>` to `SerializedAgentSession` interface. Serializes the in-memory `Map<string, string>` to a plain object on write.

- **`src/session/snapshot-reader.ts`**: Replaced `taskWorkflowStates: new Map()` (which always lost state on deserialization) with validated deserialization — iterates entries, validates each value against `VALID_TASK_WORKFLOW_STATES`, skips invalid entries (defaults to `idle`), reconstructs the `Map`. Added `reconcileTaskStatesFromPlan` — reads `plan.json`, seeds completed tasks to `tests_run` and in-progress tasks to `coder_delegated` if not already advanced. Called from `loadSnapshot` after `rehydrateState`.

- **`src/tools/update-task-status.ts`**: Added diagnostic `console.warn` in `checkReviewerGate` when all sessions show `idle` state for the queried `taskId` — regression detection logging for Issue #81.

**Impact:** HIGH — fixes silent task state loss on session restore; prevents false "gate not passed" errors caused by state reset

## Documentation

- **`README.md`**: Added complete `## Curator` section with configuration table (8 fields), pipeline overview, drift report path (`.swarm/drift-report-phase-N.json`), and Issue #81 hotfix notes.

## Migration Notes

No breaking changes. All additions are opt-in:
- Curator is disabled by default (`curator.enabled = false`)
- Issue #81 fix is fully backward-compatible (legacy snapshots without `taskWorkflowStates` deserialize to empty `Map` without error)
- All curator calls are wrapped in try/catch — no new failure modes introduced
