# OpenCode Swarm v6.15.0 Release Notes

## Summary

Version 6.15.0 introduces a comprehensive **Requirements-Driven Planning** system that transforms how the Architect captures, clarifies, and validates requirements before creating implementation plans. This release adds requirement traceability, automated gap analysis, and continuous drift detection throughout the project lifecycle.

- **SPECIFY mode**: Generate structured requirement specs (FR-###, SC-###) from feature descriptions
- **CLARIFY-SPEC mode**: Resolve ambiguities one question at a time with multiple-choice options
- **Soft Spec Gate**: Warns when planning without a spec, offers to create one or skip
- **ANALYZE mode**: Audit plans against specs for gaps and gold-plating
- **DRIFT-CHECK mode**: Automatic requirement drift detection at phase boundaries
- **Project Governance**: Auto-detect MUST/SHOULD rules from project-instructions.md
- **Research Caching**: SME caches external research to avoid redundant fetches

---

## Detailed Features

### 1. SPECIFY Mode (Architect)

**Problem**: Architects often start planning without captured requirements, leading to plans that miss features or over-engineer. Users had to plan in external tools then paste into OpenCode.

**How it works**: `/swarm specify <description>` triggers MODE: SPECIFY — the Architect delegates to the Explorer for codebase context and the SME for domain research, then generates `.swarm/spec.md` with:

- **FR-###** functional requirements — WHAT the system must do
- **SC-###** success criteria — measurable conditions for success
- **Given/When/Then** user scenarios — concrete usage examples
- **[NEEDS CLARIFICATION]** markers — max 3, flagging ambiguous requirements

**External plan import**: If you already have a plan, `/swarm specify` can reverse-engineer a spec from it, validate the plan's task format, and surface improvement suggestions.

**Spec constraints**: A spec MUST NOT contain technology choices, file paths, or implementation details — only WHAT and WHY.

---

### 2. CLARIFY-SPEC Mode (Architect)

**Problem**: Specs often have vague requirements ("fast", "secure") or explicit [NEEDS CLARIFICATION] markers that block confident planning.

**How it works**: `/swarm clarify [topic]` triggers MODE: CLARIFY-SPEC — reads spec.md, scans for ambiguities, delegates to SME for research, then presents questions **ONE AT A TIME** (max 8), each with 2–4 multiple-choice options. After each accepted answer, spec.md is updated with the clarification.

**Constraint**: CLARIFY-SPEC never creates a spec — if no spec.md exists, it tells you to run `/swarm specify` first.

---

### 3. Soft Spec Gate in PLAN Mode

**Problem**: Starting planning without requirements is risky but should not be blocked.

**How it works**: When entering PLAN mode with no spec.md present, the Architect warns you and offers two choices:

1. **Create a spec first** → switches to SPECIFY mode
2. **Skip and plan directly** → planning proceeds exactly as before

**Backward compatible**: If you choose "Skip", zero behavior change to existing PLAN mode. The warning appears only once per session.

---

### 4. ANALYZE Mode (Critic)

**Problem**: No automated way to verify that a plan covers all specified requirements.

**How it works**: `/swarm analyze` delegates the Critic with ANALYZE context — the critic:

- Maps every FR-### requirement to plan tasks
- Flags **gaps**: requirements with no corresponding task
- Flags **gold-plating**: tasks with no corresponding requirement
- Validates task format (FILE/TASK/CONSTRAINT fields, no compound verbs, explicit deps)
- Produces a coverage table with CRITICAL/HIGH/MEDIUM/LOW severity findings

**Read-only**: The Critic never modifies files in ANALYZE mode.

---

### 5. DRIFT-CHECK Mode (Critic, PHASE-WRAP)

**Problem**: During long projects, implemented features drift from original requirements. This drift was previously invisible.

**How it works**: At the end of every phase (MODE: PHASE-WRAP), if spec.md exists, the Architect automatically delegates the Critic with DRIFT-CHECK context — the critic compares completed tasks against FR-### requirements and returns:

- **CLEAN**: All requirements addressed
- **MINOR DRIFT**: Some changes but not significant
- **SIGNIFICANT DRIFT**: Substantial deviation from original requirements

**SIGNIFICANT DRIFT** surfaces as a warning before proceeding to the next phase.

**Advisory only**: Drift detection does not block phase transitions — it's informational.

**Automatic**: No manual trigger needed — runs at every phase boundary when spec.md is present.

---

### 6. Project Governance (MODE: DISCOVER)

**Problem**: Projects with strict conventions (MUST/SHOULD rules) had no way to automatically inject those rules into the planning context.

**How it works**: During MODE: DISCOVER, the Architect checks for:

- `project-instructions.md` (project root)
- `docs/project-instructions.md`
- `INSTRUCTIONS.md`

If found, MUST rules (mandatory) and SHOULD rules (recommended) are extracted and written to `.swarm/context.md` under `## Project Governance`. The Critic then validates plans against:

- **MUST rules**: CRITICAL severity — plan must comply
- **SHOULD rules**: Recommendation level — advisory

**Silent skip**: If no governance file is found, nothing changes.

---

### 7. Research Caching (SME)

**Problem**: SME calls to external URLs were repeated on every consultation, wasting tokens and time.

**How it works**: Before any fetch, the SME checks `.swarm/context.md` for a `## Research Sources` section:

- **Cache hit**: Reuses the cached summary
- **Cache miss**: Fetches URL, produces response, appends a `CACHE-UPDATE:` line
- **Cache bypass**: Include "re-fetch", "ignore cache", or "latest" in your request

**SME is read-only**: The SME never writes to files. Cache persistence is the Architect's responsibility.

## New Commands

| Command | Description |
|---------|-------------|
| `/swarm specify [description]` | Generate spec.md from a description, or import/validate an existing plan |
| `/swarm clarify [topic]` | Resolve [NEEDS CLARIFICATION] markers and vague language in spec.md one question at a time |
| `/swarm analyze` | Audit spec.md vs plan.md for gaps and gold-plating |

---

## Upgrade Notes

- **Backward compatible**: All existing behavior preserved. No breaking changes.
- **New file**: `.swarm/spec.md` — optional, only created when you use `/swarm specify`
- **New commands**: `/swarm specify`, `/swarm clarify`, `/swarm analyze` — opt-in features
- **Soft spec gate**: When entering PLAN mode without spec.md, you'll see a one-time warning with an offer to create a spec. Choose "Skip and plan directly" to continue exactly as before.
- **PHASE-WRAP drift-check**: Added step 5.5 to PHASE-WRAP. Only runs when spec.md exists. No spec.md = no change.
- **Governance detection**: Only runs in DISCOVER mode when a governance file exists in the project root or docs/. Most projects will not have this file.
- **SME cache**: Transparent — only activates when context.md has a `## Research Sources` section.

## Before/After: PLAN Mode

### Before (v6.14 and earlier)

```
User: "Plan this feature"
→ Architect asks 1-3 clarifying questions
→ Writes plan.md directly
```

### After (v6.15, with spec.md)

```
User: "/swarm specify <feature>" → spec.md with FR-### requirements
User: "/swarm clarify" (optional) → resolve [NEEDS CLARIFICATION] markers  
User: "Plan this feature"
→ Architect reads spec.md, cross-references FR-### when decomposing tasks
→ Every task maps to a requirement
→ Writes plan.md with full requirement traceability
```

### After (v6.15, without spec.md — unchanged behavior)

```
User: "Plan this feature"
→ Architect warns "No spec found — create one or skip?"
→ User: "Skip"
→ Architect proceeds exactly as before
```

## Release Type

This is a **prompt-only release**. All new functionality is implemented through updated agent prompts (Architect, Critic, SME). No changes were made to hooks, core logic, or infrastructure code.
