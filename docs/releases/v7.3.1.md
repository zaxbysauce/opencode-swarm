# v7.3.1

## What changed

### Council mode additive model fix
- **Stage B gates always run**: Per-task reviewer and test_engineer delegations now unconditionally advance task workflow state regardless of council mode being active.
- **Council mode is phase-level**: Separated council review from per-task gates. Council now runs holistically at phase end via `submit_phase_council_verdicts` before `phase_complete`, never as a per-task review.
- **New tool `submit_phase_council_verdicts`**: Phase-level council submission tool with quorum enforcement and evidence writing to `.swarm/evidence/{phase}/phase-council.json`.
- **Fixed delegation gate guards**: Removed `if (!councilActive)` guards that were incorrectly suppressing Stage B advancement when council mode was enabled.

### Test integrity improvements
- Updated COUNCIL_SENTINELS test fixtures to match actual template strings in the new buildCouncilWorkflow().
- Fixed misleading file header comments describing the old broken behavior.
- Corrected test assertion for WHEN TO RUN COUNCIL section presence.

## Why

Council mode was meant to be **additive** at the phase level (providing holistic phase-end review) but was incorrectly implemented as **replacing** per-task Stage B gates (reviewer + test_engineer). This broke the guarantee that Stage B always runs as the mandatory intermediate QA gate before any council review.

The fix separates concerns: Stage B (per-task gates) runs unconditionally. Council (phase-level review) runs holistically at phase end and is gated by `submit_phase_council_verdicts` before `phase_complete`. Council can now never suppress Stage B, and the workflow model matches the original design intent.

## Migration steps

No API changes. Council mode configuration remains the same (`config.council.enabled = true`). Per-task workflow behavior now correctly advances through reviewer → tests_run → complete states unconditionally.

If you have existing code that assumes council mode blocks Stage B, that assumption is now incorrect — Stage B will always run.

## Breaking changes

None at the API level. Behavioral: Stage B gates (reviewer + test_engineer) are no longer optional when council mode is active.

## Known caveats

- `roundNumber` omission in `submit_phase_council_verdicts` calls defaults to 1 (not currently validated in tests — follow-up PR candidate).
- `synthesizePhaseCouncilAdvisory` exceptions are caught by outer tool wrapper, resulting in structured error response rather than tool-level error propagation.
