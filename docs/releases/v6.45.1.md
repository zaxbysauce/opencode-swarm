# v6.45.1 — Curator knowledge update pipeline fix

## What changed

### `parseKnowledgeRecommendations` — UUID v4 validation (`src/hooks/curator.ts`)

Added a UUID v4 regex guard to `parseKnowledgeRecommendations`. Previously any non-whitespace token in LLM output was accepted as an `entry_id` verbatim. Because `runCuratorPhase` never passes existing knowledge entry IDs to the LLM delegate, the LLM routinely hallucinated human-readable slugs (e.g. `tool-name-normalization`) that could never match a real entry.

Any token that is not the literal string `"new"` and does not match the UUID v4 pattern `^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$` (case-insensitive) is now normalised to `undefined` before the recommendation reaches the store.

### `applyCuratorKnowledgeUpdates` — new entry creation path (`src/hooks/curator.ts`)

Implemented the previously dead `entry_id === undefined` code path. A recommendation with `entry_id: undefined` (the `"new"` token) now creates a real `SwarmKnowledgeEntry` (status: `candidate`, confidence: `0.5`, `auto_generated: true`) and appends it via `appendKnowledge`. Entries whose lesson text is shorter than the 15-character minimum are skipped with `skipped++`.

Previously, `"new"` recommendations were silently discarded — no entry was created, no counter was incremented, and no warning was emitted.

### `CURATOR_PHASE_PROMPT` note corrected (`src/agents/explorer.ts`)

The `KNOWLEDGE_UPDATES:` format hint now instructs the LLM to always use `"new"` and explains that any non-UUID token will be treated as `"new"` by the parser. The prior note falsely claimed that CURATOR_INIT handles existing entry management; that claim is removed.

### Tests (`tests/unit/hooks/curator.test.ts`)

Three new tests cover the fixed paths:
- New `SwarmKnowledgeEntry` created when `entry_id` is `undefined`
- Short-lesson guard (`< 15` chars) skips creation and increments `skipped`
- End-to-end: hallucinated slug through `parseKnowledgeRecommendations` → `applyCuratorKnowledgeUpdates` creates a real entry

## Why

`[curator] applyCuratorKnowledgeUpdates: entry_id 'tool-name-normalization' not found — skipping` appeared in production when MiniMax-M2.7-highspeed ran as architect. Root cause: LLMs in `CURATOR_PHASE` context have no access to existing knowledge entry IDs (UUIDs), so they fabricate plausible slugs. The slug fails lookup, the warning fires, and the lesson is lost.

The original minimal fix (prompt-only) was incorrect — it suppressed the warning by routing recommendations to a silent-discard path (worse outcome). This release applies the correct three-part structural fix.

## Migration steps

No configuration changes required. Knowledge entries produced by prior runs are unaffected. New entries created by the `"new"` path will have `auto_generated: true` and `status: "candidate"`.

## Breaking changes

None.

## Known caveats

- `runCuratorPhase` still does not pass existing knowledge entry IDs to the LLM delegate. Real UUID references in `KNOWLEDGE_UPDATES` output (promote/archive of a specific existing entry) require a future change to include entry IDs in the phase context. The current fix ensures that hallucinated slugs produce useful new entries rather than silent failures.
- The pre-existing test isolation failure (57 pass / 25 fail when `curator-llm-delegation.test.ts` and `curator.test.ts` are run in the same Bun process) is caused by `vi.mock()` leaking module cache across files. It is unrelated to this fix and predates this branch.
