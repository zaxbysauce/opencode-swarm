# OpenCode Swarm v6.17.0 Release Notes

## Summary

Version 6.17.0 introduces a **Two-Tier Cross-Project Knowledge Base** that captures, validates, and reuses lessons learned across sessions and projects. This release adds persistent knowledge storage with swarm-tier (per-project) and hive-tier (cross-project) levels, automatic validation gates, lifecycle-based promotion, and intelligent injection into future Architect context.

- **Two-tier knowledge storage**: Swarm tier (`.swarm/knowledge.jsonl`) for per-project lessons, Hive tier (`~/.local/share/opencode-swarm/shared-learnings.jsonl`) for cross-project lessons
- **Three-layer validation gate**: Structural checks, content safety blocklist, and semantic quality analysis before any lesson is stored
- **Auto-promotion lifecycle**: Lessons progress from `candidate` → `established` → `promoted` based on confirmation counts and age
- **Knowledge Injector**: At phase start, up to 5 ranked lessons are prepended to Architect context with star ratings and tier labels
- **Bad-rule mitigation**: RETRACT/BAD RULE commands in retrospectives and slash commands for quarantine/restore
- **Entry health audit**: Automatic detection of low-utility, near-zero confidence, and unconfirmed auto-generated entries
- **Dark matter NPMI analysis**: `/swarm dark-matter` detects hidden file coupling via Normalized Pointwise Mutual Information
- **Diagnose health check expansion**: 7 new checks for swarm identity, phase boundaries, orphaned evidence, config backup, git detection, and spec staleness
- **QA gate hardening**: Delegation tracking prevents batch coder tasks from bypassing reviewer/test_engineer gates
- **Legacy migration**: Automatic import of lessons from `.swarm/context.md` into the knowledge base

## Detailed Features

### 1. Two-Tier Knowledge Storage

**Problem**: Lessons and patterns learned during a project were only captured in `.swarm/context.md` retrospectives. They were invisible to future projects and lost when a context compaction occurred.

**How it works**: The knowledge system uses two storage tiers:

- **Swarm tier** (`.swarm/knowledge.jsonl`) — per-project lessons. Automatically written when the Architect writes a retrospective (Lessons Learned section).
- **Hive tier** (`~/.local/share/opencode-swarm/shared-learnings.jsonl` on Linux, platform-equivalent on macOS/Windows) — cross-project lessons that have been promoted from multiple projects.
- All entries are JSONL with fields: id, tier, lesson, category, tags, scope, confidence, status, confirmed_by, retrieval_outcomes.

---

### 2. Three-Layer Validation Gate

**Problem**: Not all lessons are good. Some are vague, contradictory, or even harmful (e.g., "skip tests when blocked"). Without a validation gate, bad rules could accumulate in the knowledge base.

**How it works**: Every lesson passes through three layers before storage:

- **Layer 1 — Structural**: Checks length (15–280 chars), valid category, scope format, confidence range (0.0–1.0).
- **Layer 2 — Content Safety**: Blocklist of dangerous command patterns (`rm -rf`, `kill`, `eval(`, etc.), security-degrading patterns, and injection patterns (control chars, `system:` prefix).
- **Layer 3 — Semantic Quality**: Detects contradictions (lessons with shared tags that use negation pairs) and vague lessons that lack a tech reference or action verb.

Lessons that fail Layer 2 go to `knowledge-rejected.jsonl`. Lessons that fail Layer 3 with `warning` severity are still accepted.

---

### 3. Auto-Promotion Lifecycle

**Problem**: A single lesson seen once has low reliability. Lessons need to be confirmed across multiple sessions and projects before they are trusted enough to inject into future context.

**How it works**: Lessons advance through a lifecycle:

- `candidate` → `established`: confirmed by 3+ distinct phase IDs in the swarm tier
- `established` → `promoted`: confirmed by 3+ distinct phase IDs OR entry age ≥ 90 days
- `promoted` + `hive_eligible` flag → written to hive tier as `candidate`
- Hive `candidate` → `established`: confirmed by 3+ distinct project names
- Fast-track: entries tagged `hive_fast_track` bypass phase count requirements

---

### 4. Knowledge Injector — Tier Labels and Star Ratings

**Problem**: The Architect context window had no awareness of previously learned lessons from other projects or sessions.

**How it works**: At phase start (when a new Architect session begins), the Knowledge Injector reads both tiers, deduplicates (hive wins over swarm for same normalized lesson), ranks by `confidence × tier_weight × phase_relevance`, and prepends up to 5 entries (configurable) to the Architect's message as:

```
[HIVE] ★★★★★★★★ Always use bounded quantifiers in regex — avoids ReDoS (source: project-x, phase 3)
[SWARM] ★★★★★★ vi.mocked() not available in Bun/Vitest — use local mock variable pattern
⚠️ REJECTED PATTERN: "skip tests when blocked" — blocked at Layer 2 (dangerous pattern)
```

The injector is Architect-only: coder/reviewer/test_engineer/designer agents receive no injection.

---

### 5. Bad-Rule Mitigation — RETRACT / BAD RULE Commands

**Problem**: Incorrect or misleading lessons could accumulate and be promoted to hive tier before being caught. Users needed a way to explicitly mark a lesson as wrong.

**How it works**: Two mechanisms:

- **In retrospectives**: Writing `RETRACT: <lesson text>` or `BAD RULE: <lesson text>` in the Lessons Learned section triggers automatic quarantine of any matching entry in knowledge.jsonl. The RETRACT/BAD RULE lines are NOT stored as new lessons.
- **Slash commands**: `/swarm knowledge quarantine <id> [reason]` and `/swarm knowledge restore <id>` allow manual quarantine and restoration.
- Quarantined entries go to `knowledge-quarantined.jsonl` with `quarantine_reason`, `quarantined_at`, and `reported_by` metadata.
- A fingerprint is added to `knowledge-rejected.jsonl` to prevent re-learning the same lesson.

---

### 6. Entry Health Audit

**Problem**: Over time, low-utility lessons (repeatedly shown but never applied) waste context window space.

**How it works**: `auditEntryHealth(entry)` evaluates:

- **Low-utility**: shown ≥ 5 times with utility_score ≤ 0 → `concern: low-utility`
- **Near-zero confidence**: confidence < 0.1 → `concern: near-zero confidence`
- **Unconfirmed auto-generated**: auto_generated true and never confirmed → `concern: info` (not blocking)

---

### 7. Dark Matter Co-Change NPMI Analysis

**Problem**: Files with hidden coupling (they always change together but don't have explicit import relationships) cause mysterious failures and coordination overhead that never gets documented.

**How it works**: `/swarm dark-matter` runs `git log --name-only` over recent history, computes Normalized Pointwise Mutual Information (NPMI) for every file pair, filters out pairs with known static import edges, and reports pairs with NPMI ≥ 0.5 as "dark matter". Results can be converted to knowledge entries with category `architecture`. Requires ≥ 20 commits to produce meaningful results; gracefully degrades on git timeout or sparse history.

---

### 8. Diagnose Health Check Expansion

**Problem**: The `/swarm diagnose` command did not check for swarm identity drift, orphaned evidence, or spec-plan misalignment.

**How it works**: 7 new health checks added:

1. **Swarm identity**: plan.json Swarm field matches active swarm ID
2. **Phase boundary correctness**: every task ID prefix matches its parent phase number
3. **Orphaned evidence**: evidence file task IDs cross-referenced against plan task IDs
4. **Plan sync**: plan.json task count vs plan.md task count
5. **Config backup accumulation**: warns at 5+ `.opencode-swarm.yaml.bak*` files, fails at 20+
6. **Git repository detection**: verifies `git rev-parse --git-dir` succeeds
7. **Spec staleness**: compares spec.md first heading to plan title

---

### 9. QA Gate Hardening (Architect Prompt + Runtime)

**Problem**: The Architect could bypass QA gates by batching multiple coder tasks without triggering reviewer/test_engineer delegations.

**How it works**:

- **Delegation gate**: Tracks `qaSkipCount` per session. First consecutive coder delegation without reviewer/test_engineer issues a warning; second throws a hard-block error naming the skipped task IDs.
- **Per-task gate tracking**: `partialGateWarningsIssuedForTask` is a `Set<string>` keyed by actual task ID (via `getCurrentTaskId()`). Warning fires once per task; resets when a new coder delegation for that task is detected.
- **Prompt updates**: Added `## SLASH COMMANDS` section listing all 21 `/swarm` commands; added 3 new anti-exemption rules (POC/prototype justification, batch-QA-at-end, past-violation consistency argument).

---

### 10. Legacy Migration

**Problem**: Existing projects with lessons in `.swarm/context.md` had no path to import those lessons into the knowledge base.

**How it works**: `migrateContextToKnowledge(directory, config)` is called once per project. It reads `.swarm/context.md`, extracts bullet points from Lessons Learned, Patterns, Decisions, and SME Cache sections, pipes each through the validation gate, deduplicates, and appends to `knowledge.jsonl`. A `.swarm/.knowledge-migrated` sentinel file is written LAST (after all entries) to ensure interrupted migration is safe to re-run.

## Configuration

All knowledge system settings are configurable in `.opencode-swarm.yaml` under the `knowledge:` key:

```yaml
knowledge:
  enabled: true
  swarm_max_entries: 100
  hive_max_entries: 200
  auto_promote_days: 90
  max_inject_count: 5
  dedup_threshold: 0.6
  hive_enabled: true
  rejected_max_entries: 20
  validation_enabled: true
  evergreen_confidence: 0.9
  evergreen_utility: 0.8
  low_utility_threshold: 0.3
  min_retrievals_for_utility: 3
  schema_version: 1
```

| Command | Description |
|---------|-------------|
| `/swarm dark-matter` | Analyze git history for file co-change patterns using NPMI, filter static imports, report hidden coupling |

| `/swarm knowledge restore <id>` | Restore a quarantined knowledge entry |
| `/swarm knowledge migrate` | Trigger legacy migration from context.md to knowledge.jsonl |

## Upgrade Notes

- **Backward compatible**: All existing behavior preserved. No breaking changes.
- **New files**: `.swarm/knowledge.jsonl`, `.swarm/knowledge-rejected.jsonl`, `.swarm/knowledge-quarantined.jsonl`, `.swarm/.knowledge-migrated` — only created when knowledge features are used.
- **Hive tier**: Created on first promotion to cross-project level; location varies by platform.
- **Validation gate**: Lessons that fail Layer 2 (dangerous patterns) are rejected and logged. Layer 3 failures with warning severity are accepted.
- **Slash commands**: 4 new commands added to the 17 existing commands (21 total).
- **Diagnose expansion**: 7 new health checks run automatically; no manual action required.
- **QA gate hardening**: Architects will see warnings/errors when bypassing reviewer/test_engineer gates. This is intentional — do not bypass QA gates.
- **Legacy migration**: Run `/swarm knowledge migrate` once per existing project to import lessons from context.md.

---

## Release Type

This is a **hybrid release**. Core knowledge infrastructure (validation, storage, injection, migration) is implemented in TypeScript (hooks, tools, types). QA gate hardening, diagnose expansion, and dark-matter analysis are implemented through updated agent prompts (Architect, Critic). The knowledge system is enabled by default but respects the `knowledge.enabled` config flag.
