# v6.68.0

## What changed

### SQLite constraint store

Two new SQLite databases managed via the synchronous `bun:sqlite` API:

- **Global DB** (`global-rules.db` under the platform config dir) with tables
  `global_rules`, `agent_prompt_sections`, and `schema_migrations`.
- **Project DB** (`.swarm/swarm.db` under the swarm directory) with tables
  `qa_gate_profile`, `project_constraints`, and `schema_migrations`.

Both DBs use WAL journaling, cached singletons per directory, and idempotent
migration runners. Project DB installs a `BEFORE UPDATE` SQLite trigger that
refuses mutations to a locked `qa_gate_profile` row, providing trigger-level
immutability in addition to the app-layer guard.

### QA gate profile system

Per-plan QA gate profiles keyed by `plan_id` (derived from
`${plan.swarm}-${plan.title}` with non-alphanumeric characters normalized).
Profiles are ratchet-tighter only — loosening a gate is rejected — and are
automatically locked when the critic approves the plan, via
`write-drift-evidence → lockProfile`. Locking writes `locked_at` and
`locked_by_snapshot_seq` keyed to the append-only plan ledger snapshot seq.

- `get_approved_plan` now returns `qa_profile_hash` for Architect-side drift
  comparison.
- New tool `get_qa_gate_profile` (architect-only, read-only) returns the
  current profile including `profile_hash`.
- New tool `set_qa_gates` (architect-only) creates or tightens the profile.
  Rejects with `profile_locked` / `ratchet_violation` / `set_gates_failed`
  reasons.
- `AgentSessionState` gains an optional `qaGateSessionOverrides?: Partial<QaGates>`
  field for per-session overlays, initialized to `{}` and migration-safe for
  existing in-flight sessions.

### MODE: BRAINSTORM

New architect planning mode invoked by `/swarm brainstorm [topic]`. Runs a
strictly-ordered 7-phase workflow: CONTEXT SCAN → DIALOGUE (one question per
message) → APPROACHES → DESIGN SECTIONS → SPEC WRITE + SELF-REVIEW → QA GATE
SELECTION → TRANSITION.

- Command handler at `src/commands/brainstorm.ts`; registered in
  `src/commands/registry.ts` and exposed via `src/index.ts` slash routing.
- Architect prompt gains priority-0 override recognition for `/swarm brainstorm`
  and natural-language triggers ("brainstorm", "think through", "workshop"),
  plus a full `### MODE: BRAINSTORM` section with reviewer integration at
  Phase 5 and `get_qa_gate_profile`/`set_qa_gates` integration at Phase 6.

### `/swarm qa-gates` command

New slash subcommand with `show` / `enable` / `override` actions for inspecting
and modifying QA gate state. Displays spec gates, session overrides, effective
gates, and the profile hash. Handles locked-profile rejections with a clear
user-facing reason.

## Why

The swarm previously held QA policy only in volatile spec text and session
state, with no durable binding between a critic-approved plan and the gates
that were in effect at approval time. A later run with a weakened gate profile
could unknowingly ship under looser quality rules than the approved baseline
required. The SQLite profile store binds gates to the approved plan's ledger
snapshot and locks them immutably at approval, closing that drift window.

MODE: BRAINSTORM addresses a related user-experience gap: the architect could
enter implementation mode before the user had finished shaping intent, leading
to spec churn and wasted delegation. The explicit 7-phase workflow enforces a
dialogue-first, approaches-before-design discipline.

## Migration steps

None required. All additions are opt-in:

- Existing sessions continue to run without a QA gate profile; profile lookups
  return `null` and the get-approved-plan drift check tolerates a null
  `qa_profile_hash`.
- `qaGateSessionOverrides` is optional on `AgentSessionState`; legacy session
  fixtures without this field load unchanged.
- MODE: BRAINSTORM activates only on explicit command or natural-language
  trigger.

## Breaking changes

None.

## Known caveats

- Locking is one-way. Once a profile is locked via a critic-approved snapshot,
  it cannot be unlocked through the tooling — by design. Starting a new plan
  (distinct `plan_id`) creates a fresh profile.
- The SQLite DBs are per-directory singletons. Multiple processes in the same
  project directory share the WAL-mode DB, which bun:sqlite handles safely;
  however, explicit `closeProjectDb` / `closeGlobalDb` calls are required
  before deleting the files (e.g., in tests).
- `set_qa_gates` can loosen only session overrides, never spec gates. Attempts
  to relax a spec gate return `ratchet_violation`.
