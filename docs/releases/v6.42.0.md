# v6.42.0 — Curator LLM Delegation Wired

## What changed

### Core feature: Curator now performs real LLM analysis
The Curator's LLM delegation callback was fully scaffolded (timeout, parsing, fallback) but never wired — every call site fell through to data-only mode, producing empty knowledge recommendations. This release connects all three call sites to the Explorer agent running with curator-specific system prompts.

**Files modified:**
- `src/hooks/curator-llm-factory.ts` *(new)* — `createCuratorLLMDelegate(directory)` creates an ephemeral opencode session, prompts the Explorer agent with `CURATOR_INIT_PROMPT` or `CURATOR_PHASE_PROMPT`, extracts text parts, and deletes the session (best-effort cleanup)
- `src/state.ts` — `swarmState.opencodeClient` field stores the SDK client set at plugin init; cleared in `resetSwarmState()`
- `src/index.ts` — `swarmState.opencodeClient = ctx.client` wired early in plugin init; both `createPhaseMonitorHook` calls now pass the delegate as 4th arg
- `src/hooks/phase-monitor.ts` — `curatorRunner` changed from default-value param to optional with `??` fallback; `llmDelegate?` added as 4th param and threaded through to `runner()`
- `src/hooks/index.ts` — barrel export added for `createCuratorLLMDelegate`
- `src/tools/phase-complete.ts` — delegate created and passed as 6th arg to `runCuratorPhase`
- `src/tools/curator-analyze.ts` — same pattern

### Config default change: `curator.enabled` now defaults to `true`
Previously `curator.enabled` defaulted to `false` (opt-in). Since the delegation is now wired and the curator falls back to data-only mode when no SDK client is available (e.g. in unit tests), enabling by default is safe.

## Why

The Curator was designed to be the Explorer agent running with specialized knowledge-curation prompts. The infrastructure was built but the SDK client was never stored anywhere accessible to hooks or tools — `ctx.client` from `PluginInput` was only available in the plugin init closure. This release stores it in `swarmState` and creates a factory function that all call sites can use.

## Migration steps

No action required for most users. If you explicitly set `curator.enabled: false` in `.swarm/config.json`, that setting continues to be respected.

If you have tests that assert `CuratorConfigSchema.parse({}).enabled === false`, update them to `=== true`.

## Breaking changes

None for production use. The `curator.enabled` schema default change from `false` to `true` is a behavioral change: curator analysis now runs by default when no config file is present. The curator falls back gracefully to data-only mode if the SDK client is unavailable.

## Known caveats

- Ephemeral session cleanup on delete failure is best-effort (`.catch(() => {})`). Failed deletions are silently swallowed; sessions may accumulate in extreme failure scenarios.
- Abort signal is not threaded into the curator delegate for v1. The existing 30-second `Promise.race` timeout in `curator.ts` provides protection against hangs.
- The v1 SDK API (`{ path: {}, body: {} }` style) is used, not the v2 SDK. This matches the `PluginInput.client` type from `@opencode-ai/plugin`.
