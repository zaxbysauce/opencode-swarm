# Repository memory graph contract

> Status: contract for the ordered graph-memory PR series. This document defines
> the required behavior before graph extraction, persistence, or agent behavior
> is expanded further. Current implemented repo-graph details remain documented
> in `docs/repo-graph-call-graph.md` and `docs/repo-graph-symbol-graph.md`.

## Purpose

Repository memory is the graph-backed context-reduction layer that lets agents
ask targeted structural questions before opening broad file sets. It builds on
the native `repo_map` tool, `.swarm/repo-graph.json`, graph schema `1.2.0`,
`exportRanges`, `symbolEdges`, and `context_pack`, then adds a durable contract
for provenance, freshness, confidence, truncation, warnings, and graph-first
retrieval.

The contract exists so future implementation PRs can be judged against one
definition of "fully implemented" across Windows, macOS, Linux, and every
language profile in `LANGUAGE_REGISTRY`.

## Non-goals

- Do not introduce SQLite, embeddings, or an external analyzer in this contract
  slice.
- Do not auto-install or bundle third-party code-memory binaries.
- Do not change Swarm agent behavior until the graph output contract is
  implemented and validated.
- Do not move runtime state outside the project root `.swarm/` directory.
- Do not rebuild or repair repository graphs on the plugin initialization path.

## Current foundation

The current repository graph is stored at `.swarm/repo-graph.json`. It is
generated by `repo_map action="build"` and queried through `repo_map` actions
including `importers`, `dependencies`, `blast_radius`, `localization`,
`key_files`, `ontology`, `package_boundaries`, `preflight_packet`, `callers`,
`dead_exports`, `context_pack`, and `graph_health`.

Current schema `1.2.0` is additive:

| Schema | Added fields | Compatibility rule |
| --- | --- | --- |
| `1.0.0` | file nodes, file edges, graph metadata | Must continue to load when structurally valid. |
| `1.1.0` | `GraphEdge.usedSymbols`, `GraphNode.exportLines` | Optional fields; symbol-usage queries self-gate when absent. |
| `1.2.0` | `GraphNode.exportRanges`, `RepoGraph.symbolEdges` | Optional fields; `context_pack` returns `schemaSupported: false` and a rebuild note on older graphs. |
| additive diagnostics | `RepoGraph.diagnostics` | Optional on every schema version; `graph_health` reports empty diagnostics with a rebuild note for old graphs. |

The loader validates that a graph is structurally safe enough to read. Query
functions are responsible for exact schema gates. Future graph-memory fields must
follow the same pattern: old graphs load safely, new queries degrade with an
explicit warning or rebuild instruction, and no query silently treats missing
optional data as authoritative absence.

## Production MVP vs fully implemented

Production MVP means graph memory can be trusted as a bounded, source-grounded
first pass:

- The graph builds and loads on Windows, macOS, and Linux.
- Every output includes provenance, freshness, truncation, confidence, and
  warnings.
- Agents can request a source-grounded context pack for a symbol or file and
  know when search fallback was used.
- Existing `repo-graph.json` files remain readable and receive explicit rebuild
  guidance when newer fields are required.
- The implementation never blocks plugin startup and never writes graph state
  outside `.swarm/`.

Fully implemented graph memory means the system is release-ready for broad agent
use:

- All 12 supported language profiles are covered according to their tier
  guarantees.
- Graph-first retrieval is the default for broad codebase orientation, with
  search used as a recorded fallback rather than an untracked substitute.
- Context packs are source-grounded, budgeted, deduped, confidence-labelled, and
  warning-bearing.
- Schema upgrades have migration tests for old graphs, malformed graphs, stale
  graphs, and missing optional fields.
- Cross-platform CI proves build, query, storage, and path normalization behavior.

The MVP milestone is satisfied once KG-01 through KG-08 are merged (this contract,
cross-platform runtime, provenance and freshness, confidence and warnings, schema
compatibility tests, language-tier guarantees, source-grounded context packs, and
retrieval helpers). KG-09 through KG-18 then harden, broaden, and promote graph
memory from MVP to the default agent path.

## OS support matrix

| OS | Required support | Release proof |
| --- | --- | --- |
| Windows | Backslash and drive-letter paths normalize into graph keys and output spans without leaking absolute paths. Atomic save and rename paths tolerate Windows filesystem behavior. | Windows unit and smoke coverage for graph build/load/query paths. |
| macOS | Symlink and realpath behavior stays inside the workspace and `.swarm/`; case-sensitive and case-insensitive filesystems do not corrupt graph keys. | macOS unit and smoke coverage for graph build/load/query paths. |
| Linux | Baseline CI behavior for graph build/load/query, including large-repo budgets and stale graph handling. | Linux unit, integration, and smoke coverage for graph build/load/query paths. |

Passing on one OS is not release proof for the other two. Any PR that changes
graph storage, path normalization, tree-sitter loading, graph traversal, or
context-pack output must explain its Windows, macOS, and Linux validation.

## Language support matrix

The supported language list comes from `LANGUAGE_REGISTRY` and currently has 12
profiles.

| Tier | Profiles | Contract guarantee |
| --- | --- | --- |
| Tier 1 | TypeScript/JavaScript, Python, Rust, Go | Best graph-memory coverage. Definition spans, imports, references, symbol edges, and context packs should be precise enough to guide first-pass edits, subject to confidence labels. |
| Tier 2 | Java, Kotlin, C#, C/C++, Swift | Supported graph-memory coverage with conservative confidence by default where syntax supports multiple resolution meanings. Dynamic dispatch, overloads, generated code, and framework entry points must surface warnings when unresolved. |
| Tier 3 | Dart, Ruby, PHP | Best-effort graph-memory coverage. Dynamic language behavior and framework conventions require explicit conservative or heuristic confidence unless backed by exact source spans. |

All tiers must fail open: if grammar loading, parsing, symbol extraction, or edge
resolution fails, the graph may still contain file-level facts, but it must not
claim symbol-level precision. Missing symbol data is a warning-bearing
degradation, not a successful exact result.

## Confidence levels

Every graph fact used for repository memory must carry one confidence value:

| Confidence | Meaning | Examples |
| --- | --- | --- |
| `exact` | Directly derived from a source span or schema field with no unresolved branch in the reasoning path. | A file node path, an exported symbol's `exportRanges` span, a graph metadata timestamp. |
| `conservative` | Derived from syntax or static references in a way that intentionally avoids high-risk false positives but can miss dynamic behavior. | `dead_exports` candidates, alias-aware static symbol usage, direct import edges. |
| `heuristic` | Derived from naming, package boundaries, file roles, framework conventions, or token estimates. It may rank or prioritize but must not be treated as proof. | Context-pack token estimates, ontology roles, package-boundary grouping. |
| `unresolved` | The graph could not prove or safely infer the fact. Consumers must search, open source, rebuild, or ask for a narrower target. | Namespace imports, dynamic imports, reflection, stale or pre-`1.2.0` graphs, missing `exportRanges`. |

Agents must not convert `heuristic` or `unresolved` facts into edit directives.
They may use them to choose the next source read.

## Required output fields

Every repository-memory output, including future context packs, graph query
responses, and agent-injected capsules, must expose:

| Field | Requirement |
| --- | --- |
| `provenance` | Source of each fact: graph schema version, graph build timestamp, originating file path, source span when available, query/action name, and whether fallback search contributed. |
| `freshness` | Whether the graph is fresh, stale, missing, or built from an older schema; include the timestamp or rebuild instruction. |
| `truncation` | Whether files, symbols, spans, edges, warnings, or search results were omitted because of token, count, size, depth, or time budgets. |
| `confidence` | One of `exact`, `conservative`, `heuristic`, or `unresolved` for each fact or grouped result. |
| `warnings` | Human-readable caveats such as stale graph, missing optional schema fields, unresolved imports, parse failure, fallback search used, path normalization loss, or dynamic dispatch blind spots. |

The current `ContextPackResult` has `schemaSupported`, `target`, `spans`,
`truncated`, `estimatedTokens`, and `note`. Future graph-memory PRs must extend
or wrap that shape to satisfy this required output contract before agent behavior
depends on it.

## Source-grounded context pack

A source-grounded context pack is a bounded context capsule whose primary
contents are traceable to concrete source spans. For a symbol target, it should
include:

- target identity: workspace-relative file and exported symbol;
- source spans: definition span, direct caller/callee spans, and relevant
  periphery as signature spans;
- provenance for every span: graph schema version, graph build timestamp, file,
  line range, and how the span was selected;
- confidence and warnings for each span or group;
- truncation metadata for token, span-count, depth, and file-size budgets;
- fallback records when graph data was missing and search or manual reads were
  used.

The pack is source-grounded only when a reader can trace each included claim to a
file and line range or to a named fallback search. A capsule that only lists
related files or inferred concepts is not source-grounded.

## Graph-first, search-fallback retrieval

Graph-first retrieval means agents query the graph before broad grep/search/file
reads when the task is structural: impact analysis, refactors, API changes,
callers, dependencies, ownership boundaries, route/data/security preflight, or
context reduction.

Required retrieval order:

1. Load or build the graph explicitly through `repo_map action="build"` when a
   graph is missing and the task warrants graph memory.
2. Query the narrowest graph action that matches the task.
3. Use `repo_map action="graph_health"` when graph freshness or extraction
   quality affects trust in the result.
4. Inspect confidence, freshness, truncation, and warnings before using results.
5. Use search fallback only for `unresolved`, stale, old-schema, truncated, or
   unsupported-language gaps.
6. Record fallback provenance in the final output so later agents know which
   facts came from graph memory and which came from grep/search/file reads.

Search fallback is not a failure when it is explicit. Silent fallback is a
contract violation because downstream agents cannot distinguish proven graph
facts from broad textual guesses.

## Backward compatibility policy

Future schema upgrades must preserve these rules:

- Existing `1.0.0`, `1.1.0`, and `1.2.0` graphs remain readable when structurally
  valid.
- New fields are additive unless a documented migration PR provides a safe
  rebuild path and compatibility tests.
- Missing optional fields must produce explicit `unresolved` confidence or
  schema/rebuild warnings.
- Missing diagnostics must produce an explicit graph-health rebuild note rather
  than silently implying a clean extraction.
- Query code self-gates on the minimum schema it needs.
- The loader must not silently rewrite old graphs on read.
- Rebuild instructions must name `repo_map action="build"` and the missing
  capability.
- Runtime state remains under `.swarm/repo-graph.json`; committed docs are the
  only version-controlled graph-memory deliverable in this contract slice.

## Glossary

| Term | Definition |
| --- | --- |
| Symbol node | A named exported or enclosing declaration that can own source spans or symbol edges. |
| File node | A graph node representing one source file, with module name, exports, imports, language, metadata, and optional ontology facts. |
| Edge | A relationship between graph facts, such as file import edges or symbol-to-symbol reference edges. |
| Source span | A 1-based inclusive line range inside a source file. |
| Snippet | The source text extracted from one source span for a context capsule. |
| Provenance | The evidence trail for a graph fact: schema version, build timestamp, query/action, file, line range, and fallback source if any. |
| Confidence | The precision label for a graph fact: `exact`, `conservative`, `heuristic`, or `unresolved`. |
| Stale graph | A graph whose build timestamp or schema is too old for the requested query or whose source files may have changed since build. |
| Unresolved edge | A relationship the graph could not safely attribute, such as namespace imports, dynamic imports, reflection, generated code, or missing symbol ranges. |
| Context capsule | A bounded, provenance-bearing packet of graph facts, source spans, snippets, confidence labels, warnings, and fallback records. |
